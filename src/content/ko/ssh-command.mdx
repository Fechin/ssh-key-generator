# SSH 명령어: Linux, Mac, Windows 예제가 포함된 완전 참조 가이드

`ssh` 명령어는 커맨드라인에서 SSH 연결을 시작하는 주요 도구입니다. 원격 Linux 서버에 로그인하거나, 데이터베이스 접근을 위해 포트를 포워딩하거나, 배스천 호스트를 통해 내부 네트워크에 접근하는 경우—`ssh` 명령어가 모든 것의 시작점입니다. 이 가이드는 기본 문법부터 고급 플래그, 터널링, ProxyJump 설정까지—Linux, macOS, Windows에서의 실제 예제와 함께 모든 것을 다룹니다.

SSH 명령어를 실행하기 전에 SSH 키가 필요합니다. <a href="/ko/what-is-an-ssh-key/">SSH 키란 무엇인가</a> 가이드를 참조하거나 <a href="/">무료 온라인 생성기</a>로 키 쌍을 만드세요.

## SSH 명령어 문법

`ssh` 명령어의 기본 문법은 다음과 같습니다:

```
ssh [options] [user@]hostname [command]
```

분석:

- **`[options]`** — 동작을 수정하는 0개 이상의 플래그(이 가이드 전반에 걸쳐 설명)
- **`[user@]`** — 선택적 사용자 이름. 생략하면 SSH는 현재 로컬 사용자 이름을 사용
- **`hostname`** — 연결할 원격 호스트. IP 주소, 도메인명, 또는 `~/.ssh/config`에 정의된 별칭 가능
- **`[command]`** — 원격 호스트에서 실행할 선택적 명령어. 생략하면 SSH가 인터랙티브 셸 세션 시작

**기본 예제:**

```bash
# 현재 사용자 이름으로 연결
ssh server.example.com

# 특정 사용자로 연결
ssh alice@server.example.com

# IP 주소로 연결
ssh ubuntu@192.168.1.100

# 단일 명령어를 실행하고 즉시 연결 해제
ssh user@server.example.com "df -h"

# 명령어를 실행하고 연결 유지
ssh user@server.example.com
```

## Linux와 macOS에서의 SSH

리눅스 배포판과 macOS에는 **OpenSSH**가 기본으로 포함되어 있습니다. 리눅스에서는 `openssh-client` 패키지(Debian/Ubuntu) 또는 `openssh-clients`(RHEL/CentOS/Fedora)의 일부입니다. macOS에서는 시스템에 내장되어 있어 터미널이나 iTerm2에서 바로 사용할 수 있습니다.

### SSH 사용 가능 여부 확인

```bash
ssh -V
# 출력: OpenSSH_9.6p1, LibreSSL 3.3.6
```

### 기본 연결

```bash
# 서버에 연결
ssh user@myserver.example.com

# 첫 연결 시 SSH가 ~/.ssh/known_hosts를 확인하고 프롬프트 표시:
# The authenticity of host 'myserver.example.com' can't be established.
# ED25519 key fingerprint is SHA256:abc123...
# Are you sure you want to continue connecting (yes/no)? yes

# 호스트 핑거프린트가 ~/.ssh/known_hosts에 저장됨
```

### 편의를 위한 SSH 설정 파일 사용

매번 전체 연결 정보를 입력하는 대신 `~/.ssh/config`를 만듭니다:

```
Host myserver
    HostName myserver.example.com
    User alice
    Port 22
    IdentityFile ~/.ssh/id_ed25519

Host dev
    HostName dev.example.com
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_work
```

이제 `ssh myserver`가 전체 연결로 확장됩니다—플래그 없이.

## Windows에서의 SSH(PowerShell과 OpenSSH)

Windows 10(버전 1809)과 Windows 11에는 **OpenSSH 클라이언트**가 선택적 기능으로 포함되어 있어, PowerShell, 명령 프롬프트, Windows 터미널에서 `ssh` 명령어를 사용할 수 있습니다.

### OpenSSH 설치 여부 확인

```powershell
# PowerShell에서
ssh -V
```

명령어를 찾을 수 없으면 설치합니다:

**설정을 통해:**
1. **설정 → 앱 → 선택적 기능** 열기
2. **기능 추가** 클릭
3. **OpenSSH 클라이언트** 찾아서 설치

**PowerShell(관리자)을 통해:**

```powershell
Add-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0
```

### PowerShell에서 SSH 사용

설치 후 문법은 Linux와 macOS와 동일합니다:

```powershell
# 서버에 연결
ssh user@server.example.com

# 커스텀 포트로 연결
ssh -p 2222 user@server.example.com

# 특정 키 파일 사용
ssh -i C:\Users\YourName\.ssh\id_ed25519 user@server.example.com
```

### Windows SSH 키 위치

Windows의 SSH 키는 `C:\Users\YourUsername\.ssh\`에 저장됩니다. PowerShell은 `~`를 단축키로 인식합니다:

```powershell
# 이 둘은 동일
ssh -i ~/.ssh/id_ed25519 user@server
ssh -i C:\Users\YourUsername\.ssh\id_ed25519 user@server
```

### Windows에서 PuTTY 대 OpenSSH

Windows에서 OpenSSH를 사용할 수 있기 전에는 **PuTTY**가 표준 SSH 클라이언트였습니다. PuTTY는 자체 키 형식(`.ppk` 파일)과 자체 키 생성기(PuTTYgen)를 사용합니다. `.ppk` 키가 있어서 표준 OpenSSH 형식으로 변환해야 한다면:

```bash
# PuTTY 도구 사용
puttygen mykey.ppk -O private-openssh -o id_ed25519
```

새 설정에는 Windows 내장 OpenSSH 클라이언트를 사용하는 것을 권장합니다—OpenSSH 프로젝트에서 관리하며 Linux 및 macOS와 동일한 키 형식과 설정 파일을 사용합니다.

## 키 인증을 사용한 SSH: -i 플래그

기본적으로 SSH는 `~/.ssh/`에서 표준 이름의 키 파일(`id_ed25519`, `id_rsa`, `id_ecdsa`, `id_dsa`)을 찾습니다. **`-i` 플래그**(아이덴티티 파일)로 다른 키 파일을 명시적으로 지정할 수 있습니다.

```bash
# 특정 키 파일 사용
ssh -i ~/.ssh/id_ed25519_production user@prod.example.com

# 비표준 위치에 저장된 키 사용
ssh -i /path/to/my/custom_key deploy@server.example.com

# 다른 옵션과 결합
ssh -i ~/.ssh/deploy_key -p 2222 -l deploy server.example.com
```

`-i` 플래그가 필수인 상황:
- 다른 서버나 목적을 위한 여러 키 쌍이 있을 때
- 키가 기본 `~/.ssh/` 디렉토리에 없을 때
- CI/CD 환경에서 경로를 명시적으로 지정할 때
- AWS나 다른 클라우드 제공업체로부터 `.pem` 키를 다운로드했을 때

### AWS EC2 예제

AWS는 EC2 인스턴스를 시작할 때 `.pem` 키 파일을 제공합니다:

```bash
# .pem 키로 AWS EC2 인스턴스에 연결
ssh -i ~/Downloads/my-aws-key.pem ec2-user@54.123.456.789

# 필요한 경우 권한 수정(필수——SSH는 모든 사람이 읽을 수 있는 키를 거부)
chmod 400 ~/Downloads/my-aws-key.pem
```

## 필수 SSH 명령어 옵션과 플래그

`ssh` 명령어에는 수십 개의 옵션이 있습니다. 실제로 사용하게 될 것들을 소개합니다.

| 플래그 | 설명 | 예제 |
|---|---|---|
| `-p PORT` | 비표준 포트로 연결 | `ssh -p 2222 user@host` |
| `-i FILE` | 특정 아이덴티티(키) 파일 사용 | `ssh -i ~/.ssh/mykey user@host` |
| `-l USER` | 로그인 사용자 이름 지정(`user@`의 대안) | `ssh -l alice host` |
| `-L local:host:remote` | 로컬 포트 포워딩 | `ssh -L 8080:localhost:80 host` |
| `-R remote:host:local` | 원격 포트 포워딩 | `ssh -R 9090:localhost:3000 host` |
| `-D port` | 동적(SOCKS) 프록시 | `ssh -D 1080 host` |
| `-N` | 원격 명령 실행 안 함(터널링 전용) | `ssh -N -L 3307:db:3306 host` |
| `-f` | 인증 후 백그라운드로 이동 | `ssh -f -N -D 1080 host` |
| `-v` | 상세(디버그) 출력 | `ssh -v user@host` |
| `-vv` 또는 `-vvv` | 더 상세한 출력 | `ssh -vvv user@host` |
| `-A` | SSH 에이전트 포워딩 활성화 | `ssh -A user@jump-host` |
| `-X` | X11 포워딩 활성화 | `ssh -X user@host xclock` |
| `-C` | 압축 활성화 | `ssh -C user@host` |
| `-T` | 가상 터미널 할당 비활성화 | `ssh -T git@github.com` |
| `-q` | 조용한 모드(경고 억제) | `ssh -q user@host` |
| `-o OPTION=VALUE` | 설정 옵션을 인라인으로 설정 | `ssh -o StrictHostKeyChecking=no host` |
| `-J JUMP` | 배스천 호스트를 통한 ProxyJump | `ssh -J jump.host user@internal` |
| `-W host:port` | stdio를 host:port로 포워드(ProxyJump용) | `-J`가 내부적으로 사용 |

### 디버깅을 위한 상세 모드

SSH가 실패할 때 `-v`(또는 `-vv`나 `-vvv`)로 정확히 무슨 일이 일어나는지 알 수 있습니다:

```bash
ssh -v user@server.example.com
# 표시: 어떤 키가 시도되는지, 서버가 허용하는 인증 방식,
# 에이전트가 관여하는지, 연결이 어디서 실패하는지

ssh -vvv user@server.example.com
# 최대 상세도——세부 암호화 협상 표시
```

### 호스트 키 확인 우회(주의하여 사용)

자동화 및 테스트 환경에서 호스트 키 검증이 불편한 경우:

```bash
ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null user@host
```

**프로덕션에서는 이것을 사용하지 마세요.** 중간자 공격에 대한 보호가 비활성화됩니다. 대신 프로비저닝 스크립트에서 `ssh-keyscan`을 사용하여 `known_hosts`를 미리 채우세요.

### 에이전트 포워딩

```bash
# 로컬 SSH 에이전트를 원격 서버로 포워딩
ssh -A user@jump-server

# jump-server에서 로컬 키를 사용하여 내부 호스트에 SSH 가능
# 점프 서버에 개인키를 복사할 필요 없음
ssh internal-server.private
```

에이전트 포워딩은 유용하지만 보안 의미가 있습니다. 중간 서버에 루트 접근 권한이 있는 누구든 포워딩된 에이전트를 사용할 수 있습니다. 신뢰할 수 있는 서버에서만 사용하고 가능하면 **ProxyJump**를 선호하세요(중간 호스트에 에이전트를 노출하지 않음).

## SSH 터널링 명령어

SSH 터널링은 암호화된 SSH 채널을 통해 TCP 연결을 라우팅합니다. 터널링이 무엇인지에 대한 배경은 <a href="/ko/what-is-ssh/">SSH 가이드의 터널링 섹션</a>을 참조하세요.

### 로컬 포트 포워딩

원격 서비스를 로컬 머신에서 접근 가능하게 만듭니다.

```bash
# 로컬 포트 3307로 원격 MySQL 데이터베이스에 접속
ssh -L 3307:localhost:3306 user@db-server.example.com -N

# SSH 호스트에서만 접근 가능한 원격 웹 서버에 연결
ssh -L 8080:intranet.company.com:80 user@vpn.company.com -N

# 프라이빗 네트워크의 Redis 인스턴스에 접속
ssh -L 6380:10.0.1.50:6379 user@bastion.example.com -N
```

위의 명령어 중 하나를 실행한 후, 로컬 애플리케이션을 `127.0.0.1:로컬포트`에 연결합니다.

### 원격 포트 포워딩

원격 서버에 로컬 서비스를 노출합니다.

```bash
# 로컬 개발 서버(포트 3000)를 원격 서버의 포트 9090에 노출
ssh -R 9090:localhost:3000 user@remote.example.com -N

# 원격 서버의 모든 인터페이스가 연결하도록 허용(sshd_config에서 GatewayPorts yes 필요)
ssh -R 0.0.0.0:9090:localhost:3000 user@remote.example.com -N
```

### 동적 포트 포워딩(SOCKS 프록시)

```bash
# 로컬 포트 1080에 SOCKS5 프록시 생성, 백그라운드로 실행
ssh -D 1080 -N -f user@remote.example.com

# 이제 브라우저나 시스템을 SOCKS5 프록시: 127.0.0.1:1080으로 설정
# 모든 트래픽이 원격 서버를 통해 라우팅됨
```

curl과 함께 사용하려면:

```bash
curl --socks5 127.0.0.1:1080 https://api.example.com/
```

### 백그라운드에서 터널 실행

```bash
# -f는 SSH를 백그라운드로 보내고, -N은 원격 명령 없음을 의미
ssh -f -N -L 3307:localhost:3306 user@db-server

# 백그라운드 SSH 프로세스 찾기
ps aux | grep ssh

# 종료하기
kill PID
```

## SSH ProxyJump와 배스천 호스트

많은 조직에서 프로덕션 서버는 인터넷에서 직접 접근할 수 없는 프라이빗 네트워크에 있습니다. **배스천 호스트**(또는 점프 호스트)는 네트워크 경계에 있는 강화된 서버로, 먼저 여기에 SSH로 접속한 다음 내부 서버로 이동합니다. `ssh -J` 플래그(ProxyJump, OpenSSH 7.3에 추가)로 이것을 원활하게 할 수 있습니다.

### 기본 ProxyJump

```bash
# 하나의 배스천을 통해 내부 서버에 도달
ssh -J ec2-user@bastion.example.com ubuntu@10.0.1.50

# 단축 형식
ssh -J user@jump-host user@internal-server
```

이렇게 하면 두 개의 SSH 연결이 생성됩니다. 배스천에 먼저 연결하고, 그 다음 배스천에서 내부 서버로—모두 단일 명령어로, 종단 간 암호화로. 로컬 SSH 에이전트가 두 호스트 모두에 대한 인증을 처리합니다.

### 여러 점프 연결

```bash
# 두 중간 호스트를 통해 점프
ssh -J user@jump1.example.com,user@jump2.example.com user@final-destination
```

### SSH 설정 파일에서 ProxyJump

자주 접근하는 서버의 경우 `~/.ssh/config`에 ProxyJump 설정을 넣습니다:

```
Host bastion
    HostName bastion.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host prod-web
    HostName 10.0.1.100
    User ubuntu
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519_aws

Host prod-db
    HostName 10.0.2.50
    User postgres
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519_aws
```

이제 `ssh prod-web`이 자동으로 배스천을 통해 연결됩니다. 수동 `-J` 플래그가 필요 없습니다.

### ProxyJump 대 에이전트 포워딩

ProxyJump가 에이전트 포워딩(`-A`)보다 선호됩니다:
- 개인키가 중간 호스트에 포워딩되지 않음
- 배스천 호스트가 자격 증명을 사용하여 다른 곳에서 인증할 수 없음
- 연결이 종단 간으로 인증됨—목적지 서버가 직접 사용자를 검증

더 많은 설정 세부 사항은 <a href="/ko/how-to-set-up-ssh/">SSH 설정 방법</a> 완전 가이드를 참조하세요.

## 자주 묻는 질문

### 비밀번호 없이 SSH 명령어를 실행하려면?

SSH 키 인증을 설정합니다. 키 쌍을 생성하고(`ssh-keygen -t ed25519`), `ssh-copy-id user@host`를 사용하여 서버의 `~/.ssh/authorized_keys` 파일에 공개키를 복사하면 이후 연결은 비밀번호 없이 키를 사용합니다. 키에 패스프레이즈가 있다면 `ssh-agent`와 `ssh-add`를 사용하여 캐시하면 세션당 한 번만 입력하면 됩니다.

### SSH 세션을 종료하려면?

`exit`를 입력하거나 **Ctrl+D**를 눌러 원격 셸을 닫고 연결을 해제합니다. 세션이 멈춰 있는 경우(예: 네트워크가 끊어진 경우) SSH 이스케이프 시퀀스를 사용합니다. **Enter**를 누르고 `~.`(물결표 다음에 점)를 입력합니다. SSH가 즉시 연결을 종료합니다. 다른 이스케이프 시퀀스: `~^Z`는 SSH를 백그라운드로 일시 중지, `~?`는 모든 이스케이프 시퀀스 목록을 표시합니다.

### SSH로 원격 서버에서 명령어를 실행하려면?

호스트명 뒤에 명령어를 추가합니다:

```bash
ssh user@server "command to run"
ssh user@server "ls -la /var/log/"
ssh user@server "sudo systemctl status nginx"
```

여러 줄 명령어나 스크립트에는 히어독을 사용합니다:

```bash
ssh user@server << 'EOF'
cd /var/www/html
git pull origin main
systemctl reload nginx
EOF
```

인터랙티브 명령어(터미널이 필요한 명령어 등)에는 가상 터미널을 할당하기 위해 `-t` 플래그를 추가합니다: `ssh -t user@server sudo apt upgrade`.

### ssh와 scp의 차이점은?

`ssh`는 원격 머신에서 터미널 세션을 열거나(또는 단일 명령어를 실행) 합니다. `scp`(시큐어 카피 프로토콜)는 암호화와 인증을 위해 SSH를 사용하여 로컬 머신과 원격 머신 사이, 또는 두 원격 머신 사이에서 파일을 복사합니다. 두 명령어 모두 동일한 SSH 프로토콜과 키 인프라를 사용합니다. 파일 전송에는 `scp`가 전용 도구입니다:

```bash
# 로컬에서 원격으로 scp
scp file.txt user@server:/destination/path/

# 원격에서 로컬로 scp
scp user@server:/remote/file.txt ./local-copy.txt
```

더 고급 파일 전송(재귀적, 재개 가능, 디렉토리 탐색)에는 `sftp`를 사용합니다. 대규모 전송에는 `rsync -avz -e ssh`가 더 효율적입니다.

### 설정 파일로 SSH를 사용하려면?

`~/.ssh/config`를 만들고 호스트 항목을 추가합니다. 각 `Host` 블록은 전체 설정이 있는 연결 별칭을 정의합니다:

```
Host alias-name
    HostName actual.server.com
    User username
    Port 22
    IdentityFile ~/.ssh/specific_key
```

그러면 `ssh alias-name`이 정의된 모든 설정을 자동으로 사용합니다. 설정 파일은 와일드카드를 지원합니다. `Host *.example.com`은 모든 일치하는 호스트에 설정을 적용합니다. 끝에 있는 `Host *`는 모든 연결에 기본값을 적용합니다. 특정 `Host` 블록의 옵션은 전역 기본값을 재정의합니다.

## SSH 키가 필요하신가요? 무료로 생성하세요

비밀번호의 필요성을 없애는 SSH 키 인증을 사용하려면 키 쌍이 필요합니다. 서버에 아무것도 전송하지 않고 브라우저에서 즉시 Ed25519 또는 RSA 키를 생성합니다.
