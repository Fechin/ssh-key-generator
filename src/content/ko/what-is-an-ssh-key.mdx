# SSH 키란? 공개키, 개인키, 키 쌍 완전 해설

**SSH 키**는 비밀번호 없이 SSH 서버에 인증하기 위한 암호화 자격 증명입니다. SSH 키는 쌍으로 존재합니다—**공개키**와 **개인키**—이 조합은 전통적인 비밀번호 인증보다 더 안전하고 편리한 원격 접속을 가능하게 합니다. 이 가이드에서는 SSH 키가 무엇인지, 키 쌍 시스템이 어떻게 동작하는지, 어떤 키 유형을 선택해야 하는지, 그리고 GitHub나 GitLab 같은 서비스에서 SSH 키를 사용하는 방법을 설명합니다.

## SSH 키의 동작 방식

SSH 키는 **비대칭 암호화**(공개키 암호화라고도 함)를 기반으로 합니다. 대칭 암호화(단일 비밀키로 데이터를 암호화하고 복호화하는 방식)와 달리, 비대칭 시스템에서는 수학적으로 연결된 키 쌍을 사용하여 한쪽 키로 암호화한 데이터는 반드시 다른 쪽 키로만 복호화할 수 있습니다.

SSH 키로 서버에 로그인할 때의 인증 흐름은 다음과 같습니다:

1. `ssh user@server.example.com`을 실행합니다.
2. 서버가 `~/.ssh/authorized_keys`에서 공개키를 조회합니다.
3. 서버가 무작위 챌린지를 생성하고 공개키로 암호화합니다.
4. SSH 클라이언트가 개인키를 사용하여 챌린지를 복호화합니다.
5. 클라이언트가 암호화 증명(복호화된 챌린지와 세션 ID에서 도출)을 서버에 전송합니다.
6. 서버가 증명을 검증합니다. 개인키 보유자만이 이것을 생성할 수 있습니다.
7. 인증 성공. 비밀번호는 머신 밖으로 나가지 않습니다.

핵심은 이것입니다: **개인키는 절대 네트워크를 통해 전송되지 않습니다**. 서버는 공개키와 사용자가 생성한 암호화 증명만을 볼 뿐입니다. SSH 세션의 모든 패킷이 가로채여도 사용자를 사칭하는 데 도움이 되는 정보는 얻을 수 없습니다.

SSH 프로토콜의 배경에 대해서는 <a href="/ko/what-is-ssh">SSH란 무엇인가</a> 가이드를 참조하세요.

## 공개키 vs 개인키: 차이점은?

SSH를 올바르게 사용하려면 두 키의 차이를 이해하는 것이 필수입니다.

### 공개키—자물쇠

**공개키**는 자유롭게 공유할 수 있도록 설계되었습니다. 자물쇠라고 생각하세요. 접속하고 싶은 시스템을 가진 모든 사람에게 복사본을 나누어 줍니다. 서버의 `authorized_keys` 파일에 공개키가 있으면 개인키를 몰라도 신원을 확인할 수 있습니다.

공개키의 특성:
- 안전하게 공유 가능—사용자를 사칭하는 데 사용할 수 없음
- 서버의 `~/.ssh/authorized_keys`에 배치
- GitHub, GitLab 등의 서비스에 업로드
- 보통 `.pub` 확장자를 가짐(예: `id_ed25519.pub`)
- 형식: `ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... user@hostname`

### 개인키—열쇠

**개인키**는 쌍의 비밀 절반입니다. 절대로 머신을 벗어나서는 안 되며 누구와도 공유해서는 안 됩니다—관리자도, 클라우드 제공업체도, 지원 스태프도 포함하여. 개인키가 유출되면, 이를 획득한 누구든 공개키가 설치된 모든 서버에서 사용자를 사칭할 수 있습니다.

개인키의 특성:
- 비밀을 유지하고 안전하게 보관해야 함
- 제한된 권한(600)으로 로컬 `~/.ssh/`에 저장
- 선택적으로 패스프레이즈로 암호화 가능
- 확장자 없음 또는 언어별 확장자(예: `id_ed25519`)

### `authorized_keys` 파일

서버 측에서는 각 사용자의 공개키가 `~/.ssh/authorized_keys`에 저장됩니다. 이것은 각 줄에 하나의 공개키가 포함된 일반 텍스트 파일입니다. SSH 클라이언트가 인증을 시도하면, 서버는 클라이언트가 이 파일의 항목 중 하나에 해당하는 개인키 보유를 증명할 수 있는지 확인합니다.

```
# ~/.ssh/authorized_keys 형식
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGq... alice@laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHr... alice@workstation
ssh-rsa AAAAB3NzaC1yc2EAAAA... backup-deploy-key
```

이 파일에 필요한 만큼 공개키를 보유할 수 있어, 여러 장치나 자동화 시스템이 동일한 서버에 인증할 수 있습니다.

## SSH 키 알고리즘: Ed25519 대 RSA

SSH 키 쌍을 생성할 때 알고리즘을 선택합니다. 현재 가장 일반적인 두 가지 옵션은 **Ed25519**와 **RSA**입니다. ECDSA와 DSA도 있지만, DSA는 더 이상 사용되지 않고 ECDSA는 구현상의 함정이 있어 추천되는 두 가지에 집중합니다.

### Ed25519—현대적인 선택

Ed25519는 2014년 OpenSSH 6.5에서 도입되었으며 현재 권장되는 기본값입니다. Daniel J. Bernstein이 설계한 Twisted Edwards 곡선(Curve25519) 위의 타원 곡선 암호화를 사용합니다.

**Ed25519의 장점:**
- **간결함** — 공개키는 68자에 불과하고 파일 크기가 작음
- **고속** — 서명 생성과 검증이 매우 빠름
- **설계상 안전** — 일부 RSA 구현에 영향을 미치는 부채널 공격에 취약하지 않음
- **고정 키 크기** — 256비트 보안 수준, 키 크기를 선택할 필요 없음
- **결정론적** — 서명이 결정론적이어서 난수 생성기 실패 유형 제거

```bash
# Ed25519 키 쌍 생성
ssh-keygen -t ed25519 -C "your@email.com"
```

### RSA—범용 옵션

RSA는 더 오래된 알고리즘(1977년 발명)으로, 4096비트에서는 여전히 안전하지만 Ed25519보다 느리고 더 큰 키를 생성합니다. 그러나 RSA는 매우 오래된 SSH 서버, 레거시 시스템, 일부 하드웨어 보안 키와의 호환성이 훨씬 넓습니다.

**RSA를 사용할 경우:**
- 이전 버전의 OpenSSH(6.5 이전)를 실행하는 레거시 시스템에 연결해야 할 때
- Ed25519를 지원하지 않는 스마트카드나 HSM을 사용할 때
- 이전 도구와의 호환성이 필요할 때

```bash
# RSA 4096비트 키 쌍 생성
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

**4096비트 미만의 RSA는 절대 사용하지 마세요.** 컴퓨팅 파워가 증가함에 따라 RSA-1024와 RSA-2048은 점점 더 부적절하다고 여겨지고 있습니다. Ed25519의 256비트는 키 크기의 일부만으로 동등하거나 더 나은 보안을 제공합니다.

### 요약 비교

| 기능 | Ed25519 | RSA 4096 |
|---|---|---|
| 키 크기(공개) | 약 68자 | 약 725자 |
| 보안 수준 | 128비트(포스트 양자 내성 구조) | 약 140비트 고전적 보안 |
| 속도 | 매우 빠름 | 느림 |
| 호환성 | 최신 SSH(OpenSSH 6.5 이상) | 범용 |
| 권장 용도 | 모든 새 키 | 레거시 시스템만 |

**결론: 모든 현대적 용도에는 Ed25519 키를 생성하세요.** 호환성 요구사항에 강제될 때만 RSA를 사용하세요.

## GitHub에 SSH 키 추가하는 방법

SSH 공개키를 GitHub에 추가하면 모든 작업에서 사용자 이름과 비밀번호 없이 SSH를 통해 푸시하고 풀할 수 있습니다. 단계별 과정을 안내합니다.

### 1단계: 키 쌍 생성

SSH 키 쌍이 아직 없다면 생성합니다. <a href="/">무료 온라인 SSH 키 생성기</a>를 사용하거나 다음을 실행합니다:

```bash
ssh-keygen -t ed25519 -C "your-github-email@example.com"
```

기본 파일 위치(`~/.ssh/id_ed25519`)를 그대로 받아들이고 선택적으로 패스프레이즈를 설정합니다.

### 2단계: 공개키 복사

공개키를 표시하고 클립보드에 복사합니다:

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows(PowerShell)
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

또는 단순히 표시하고 수동으로 복사합니다:

```bash
cat ~/.ssh/id_ed25519.pub
```

### 3단계: GitHub에 키 추가

1. **GitHub.com**에 로그인
2. 프로필 사진 클릭 → **Settings(설정)**
3. 왼쪽 사이드바에서 **SSH and GPG keys** 클릭
4. **New SSH key** 클릭
5. 설명적인 제목 입력(예: "MacBook Pro 2024" 또는 "Work Laptop")
6. **Key** 필드에 공개키 붙여넣기
7. **Add SSH key** 클릭

### 4단계: 연결 테스트

```bash
ssh -T git@github.com
```

다음 메시지가 표시되어야 합니다: `Hi username! You've successfully authenticated, but GitHub does not provide shell access.`

### 기존 저장소를 SSH로 전환

HTTPS로 클론한 저장소를 SSH로 전환하려면:

```bash
# 현재 원격 URL 확인
git remote -v

# SSH로 업데이트
git remote set-url origin git@github.com:username/repository.git
```

GitLab도 과정이 동일합니다—**Preferences → SSH Keys**로 이동하여 같은 단계를 따르세요.

## SSH 키는 어디에 저장되나?

SSH 키는 사용자 홈 폴더의 `.ssh` 디렉토리에 저장됩니다. 운영체제에 따라 위치가 다릅니다.

### Linux와 macOS

```
~/.ssh/
├── id_ed25519          # 개인키(권한: 600)
├── id_ed25519.pub      # 공개키(권한: 644)
├── id_rsa              # RSA 개인키(있는 경우)
├── id_rsa.pub          # RSA 공개키
├── authorized_keys     # 이 머신에 로그인을 허용하는 공개키
├── known_hosts         # 연결한 서버의 핑거프린트
└── config              # 클라이언트 설정 파일
```

`~/.ssh/` 디렉토리 자체는 **700** 권한(소유자만 읽기/쓰기/실행 가능)이어야 합니다. 개인키 파일은 **600** 권한(소유자만 읽기/쓰기 가능)이어야 합니다. 권한이 너무 허용적인 키는 SSH가 사용을 거부합니다.

```bash
# 필요한 경우 권한 수정
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

### Windows

OpenSSH가 설치된 Windows 10과 Windows 11에서 SSH 키는 다음에 저장됩니다:

```
C:\Users\YourUsername\.ssh\
├── id_ed25519
├── id_ed25519.pub
├── authorized_keys
├── known_hosts
└── config
```

Windows에서는 파일 탐색기(숨김 파일 표시 활성화 필요할 수 있음)나 PowerShell에서 `~\.ssh\`를 사용하여 `.ssh` 디렉토리에 접근할 수 있습니다.

### 여러 키 파일

다른 목적(개인 GitHub, 회사 GitLab, 프로덕션 서버)의 키가 있는 경우 `~/.ssh/config` 파일로 각 호스트를 해당 키에 매핑합니다:

```
Host github.com
    IdentityFile ~/.ssh/id_ed25519_github

Host gitlab.work.com
    IdentityFile ~/.ssh/id_ed25519_work

Host prod-server
    HostName 203.0.113.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_production
```

## SSH 키 패스프레이즈

SSH 키 쌍을 생성할 때 `ssh-keygen`이 **패스프레이즈** 설정 여부를 묻습니다. 이것은 디스크의 개인키 파일을 암호화하는 선택적 비밀번호입니다.

### 패스프레이즈를 사용하는 이유?

강력한 패스프레이즈로 개인키 파일이 보호되어 있으면, 파일을 훔치는 것만으로는 사용자를 사칭하기 부족합니다—공격자는 패스프레이즈도 해독해야 합니다. 이것은 특히 물리적으로 도난당할 수 있는 노트북에 대한 중요한 두 번째 보안 레이어를 제공합니다.

패스프레이즈 없이는 개인키가 디스크에 평문으로 저장됩니다. 악성코드, 물리적 접근, 적절히 보호되지 않은 백업을 통해 파일을 읽을 수 있는 누구나 즉시 사용할 수 있습니다.

### 패스프레이즈와 ssh-agent의 상호작용

패스프레이즈의 단점은 키를 사용할 때마다 입력해야 한다는 것입니다. **ssh-agent**는 세션 동안 복호화된 키를 메모리에 보관하여 이 문제를 해결합니다.

```bash
# ssh-agent 시작 및 키 추가
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
# 패스프레이즈를 한 번 입력——세션 동안 캐시됨
```

macOS에서는 시스템 키체인이 ssh-agent와 통합됩니다. 재부팅 후에도 유지하려면 `-K`를 추가합니다:

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

### 패스프레이즈 변경

키 쌍을 재생성하지 않고도 기존 키의 패스프레이즈를 변경할 수 있습니다:

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

이렇게 하면 이전 패스프레이즈를 요청한 다음 새 것을 설정할 수 있습니다. 공개키는 변경되지 않으므로 서버에서 업데이트할 필요가 없습니다.

## SSH 키 핑거프린트

모든 SSH 키 쌍에는 **핑거프린트**—공개키의 짧은 해시로 고유하게 식별합니다—가 있습니다. 핑거프린트는 여러 목적으로 사용됩니다:

- **호스트 신원 확인** — 처음 서버에 연결할 때 SSH가 호스트 키 핑거프린트를 표시합니다. 기대하는 것(호스팅 제공업체가 제공한 것 등)과 일치하는지 확인하면 올바른 서버에 연결되어 있다는 것을 알 수 있습니다.
- **키 식별** — 많은 키를 관리할 때 핑거프린트로 `authorized_keys`나 GitHub 같은 서비스의 어떤 항목이 어떤 물리적 키에 해당하는지 식별합니다.
- **감사 및 폐기** — 감사 로그는 종종 전체 키 대신 핑거프린트를 기록합니다. 키의 핑거프린트를 알면 필요할 때 찾아서 제거할 수 있습니다.

### 키 핑거프린트 확인

```bash
# 공개키의 핑거프린트 확인
ssh-keygen -lf ~/.ssh/id_ed25519.pub
# 출력: 256 SHA256:abcdef1234567890... your@email.com (ED25519)

# 서버 호스트 키의 핑거프린트 확인
ssh-keyscan server.example.com | ssh-keygen -lf -
```

GitHub는 **Settings → SSH and GPG keys**에서 추가된 모든 SSH 키의 핑거프린트를 표시하여, 거기에 표시된 키가 머신의 키와 일치하는지 확인할 수 있습니다.

### `known_hosts` 파일

새 SSH 서버에 처음 연결할 때 다음과 같은 메시지가 표시됩니다:

```
The authenticity of host 'example.com (203.0.113.10)' can't be established.
ED25519 key fingerprint is SHA256:abcdef1234567890...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

`yes`를 입력하면 서버의 핑거프린트가 `~/.ssh/known_hosts`에 추가됩니다. 이후 연결에서 SSH는 서버 키가 저장된 핑거프린트와 일치하는지 자동으로 확인합니다. 일치하지 않으면—설정 오류나 중간자 공격을 나타낼 수 있습니다—SSH는 큰 경고를 표시하고 연결을 거부합니다.

## 자주 묻는 질문

### SSH 키는 어디에 사용되나요?

SSH 키는 비밀번호 없이 원격 서버에 인증합니다. 가장 일반적인 용도는: 관리를 위한 Linux/Unix 서버 로그인, 푸시/풀 접근을 위한 Git 호스팅 서비스(GitHub, GitLab, Bitbucket) 인증, 비밀번호 없는 서버 접근이 필요한 자동화 배포 스크립트, 프로덕션에 코드를 배포하는 CI/CD 파이프라인. SSH 키는 무차별 대입 공격에 면역이 있고, 침해 사이에서 재사용되지 않으며, 사람의 개입 없이 사용할 수 있어 비밀번호보다 선호됩니다.

### SSH 키는 몇 개 가져야 하나요?

하나의 정답은 없지만 일반적인 모범 사례는 **장치당 하나의 키**(서비스당이 아닌)입니다. 노트북에는 하나의 키 쌍, 데스크톱에는 다른 하나, CI 서버에는 또 다른 하나. 이렇게 하면 장치가 도난당하거나 침해되어도 해당 장치의 키만 폐기하면 됩니다. 일부 사람들은 컨텍스트(개인 vs 업무)로 더 세분화하여 장치당 두 개의 키를 사용합니다. 여러 머신 간에 단일 키를 공유하는 것은 피하세요—해당 키가 침해되면 모든 접근이 동시에 위험에 처합니다.

### SSH 공개키를 공유할 수 있나요?

예. 공개키는 공유하도록 설계되었습니다. 이메일로 보내거나, GitHub 문서에 게시하거나, 서버 접근을 요청하는 티켓에 포함하는 것이 안전합니다. 공개키만으로는 사용자로 인증할 수 없습니다—그것은 대응하는 개인키가 필요하며, 그것은 오직 사용자만 가지고 있습니다. 이것이 비대칭 암호화의 근본적인 속성입니다. **개인**키는 절대 누구와도 공유하지 마세요.

### SSH 개인키를 잃어버리면 어떻게 되나요?

개인키를 잃어버리고 백업도 없다면, 그것으로 인증하는 능력을 잃게 됩니다. 서버와 서비스에 여전히 설치된 공개키는 고아가 됩니다. 이전 공개키를 `authorized_keys`에서 제거하고 새것을 추가하기 위해 다른 방법으로 서버에 접근해야 합니다(비밀번호 인증이 활성화된 경우, 콘솔 접근, 또는 호스팅 제공업체의 긴급 접근 메커니즘). 이것이 개인키 백업(비밀번호 관리자나 암호화된 백업에)이 중요한 이유입니다. 개인키는 공개키로부터 재구성할 수 없습니다.

### Ed25519와 RSA 중 어느 것이 더 낫나요?

**모든 새 키 생성에는 Ed25519가 더 낫습니다.** 더 빠르고, 더 작은 키를 생성하며, RSA에 영향을 미치는 특정 구현 취약점에 취약하지 않고, 256비트 키로 RSA-4096에 동등하거나 더 나은 보안을 제공합니다. 오늘날 RSA를 사용하는 유일한 이유는 매우 오래된 SSH 서버 소프트웨어(2014년 출시된 OpenSSH 6.5 이전)나 레거시 하드웨어 보안 모듈과의 호환성입니다. 현대적인 서버나 GitHub 같은 서비스에 접속한다면 Ed25519를 사용하세요.

## SSH 키 쌍 생성
