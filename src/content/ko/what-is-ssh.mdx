# SSH란 무엇인가? 시큐어 셸 완전 가이드

SSH는 **Secure Shell(시큐어 셸)**의 약자로, 보안이 취약한 네트워크 환경에서도 원격 컴퓨터에 안전하게 접속하고 관리할 수 있게 해주는 암호화 네트워크 프로토콜입니다. 리눅스 서버에 로그인하거나, GitHub에 코드를 푸시하거나, 머신 간에 파일을 전송할 때—거의 모든 경우에 SSH가 뒤에서 작동하고 있습니다. 이 가이드에서는 SSH가 무엇인지, 어떻게 동작하는지, 그리고 왜 안전한 원격 접속의 표준이 되었는지를 상세히 설명합니다.

## SSH는 무슨 약자인가?

**SSH는 Secure Shell의 약자**입니다. 이름이 프로토콜의 역할을 그대로 설명합니다—원격 머신에 안전하게 암호화된 셸 세션을 제공합니다. SSH가 등장하기 전에는 관리자들이 Telnet이나 rsh(원격 셸) 같은 도구로 원격 서버를 관리했습니다. 이러한 프로토콜들은 사용자 이름과 비밀번호를 포함한 모든 데이터를 평문으로 전송하여, 공유 네트워크에서 쉽게 가로챌 수 있었습니다.

SSH는 1995년 핀란드의 컴퓨터 과학자 **Tatu Ylönen**이 개발했습니다. 그가 소속된 대학 네트워크에서 발생한 패스워드 스니핑 공격을 직접 목격한 것이 계기였습니다. 그는 SSH-1을 자유 소프트웨어로 공개했고, 빠르게 인터넷 전반에 보급되었습니다. 이후 IETF에서 SSH-2(RFC 4251~4254)로 표준화되었으며, 더 강력한 암호화와 기존 설계의 보안 취약점들이 개선되었습니다.

오늘날 "SSH"라고 하면 거의 대부분 SSH-2를 의미합니다. 가장 널리 사용되는 구현체는 **OpenSSH**로, OpenBSD 팀이 관리하는 오픈소스 프로젝트이며 거의 모든 리눅스 배포판과 macOS에 기본 포함되어 있습니다.

**SSH의 실질적인 의미**는 단순한 로그인 도구를 넘어섭니다. 다음을 지원하는 완전한 프로토콜 스위트입니다:
- 안전한 원격 터미널 접속
- 암호화 파일 전송(SFTP 및 SCP)
- Git 인증(GitHub, GitLab, Bitbucket)
- 안전한 포트 포워딩 및 터널링
- 자동화된 배포 파이프라인

## SSH의 동작 방식

**시큐어 셸 프로토콜**은 암호화, 인증, 채널 다중화를 각각 별도로 처리하는 계층형 아키텍처로 동작합니다. 이 계층들을 이해하면 SSH가 왜 민감한 작업에도 신뢰받는지 파악할 수 있습니다.

### 3가지 프로토콜 계층

**1. 트랜스포트 계층(RFC 4253)**

SSH 연결을 시작하면 클라이언트와 서버는 먼저 핸드셰이크를 수행하여 암호화 알고리즘을 협상하고 공유 비밀을 확립합니다. 최신 SSH는 기본적으로 Curve25519 기반 키 교환을 사용합니다. 핸드셰이크 이후의 모든 통신은 암호화됩니다. 트랜스포트 계층은 메시지 인증 코드(MAC)를 통한 데이터 무결성도 제공하여, 전송 중 데이터 변조를 방지합니다.

**2. 사용자 인증 계층(RFC 4252)**

암호화 채널이 확립되면 서버는 클라이언트에게 인증을 요청합니다. SSH는 여러 인증 방식을 지원합니다:

- **공개키 인증** — 클라이언트가 네트워크를 통해 개인키를 전송하지 않고도 그 보유를 증명합니다. 가장 안전하고 일반적인 방식입니다.
- **비밀번호 인증** — 암호화된 채널을 통해 사용자 비밀번호를 전송합니다. 편리하지만 무차별 대입 공격에 취약합니다.
- **키보드 인터랙티브** — 챌린지/응답 메커니즘으로 다단계 인증에 자주 사용됩니다.
- **GSSAPI/Kerberos** — 중앙 집중식 ID 관리를 갖춘 기업 환경에서 사용됩니다.

**3. 연결 계층(RFC 4254)**

인증 후, 연결 계층은 단일 암호화 연결을 여러 논리 채널로 다중화합니다. 덕분에 단일 TCP 연결을 통해 인터랙티브 셸, 원격 명령 실행, 포트 포워딩, 파일 전송을 동시에 수행할 수 있습니다.

### 키 교환 상세 과정

`ssh user@example.com`을 입력하면 처음 몇 밀리초 안에 다음 일이 발생합니다:

1. 클라이언트가 서버의 TCP 포트 22에 연결합니다.
2. 양쪽이 프로토콜 버전 문자열을 교환합니다.
3. 양쪽이 지원하는 알고리즘 목록(키 교환, 암호화, MAC, 압축)을 전송합니다.
4. Diffie-Hellman 또는 ECDH 키 교환으로 공유 세션 키를 도출합니다. 이 때문에 누군가가 암호화된 트래픽을 녹화해도 나중에 복호화할 수 없습니다.
5. 서버가 **호스트 키**(공개키)를 전송하고 클라이언트는 이를 `~/.ssh/known_hosts` 파일과 대조하여 검증합니다.
6. 위에서 설명한 방식으로 인증을 진행합니다.
7. 세션이 시작됩니다.

## SSH 대 Telnet: SSH가 승리한 이유

SSH 이전에는 **Telnet**이 원격 터미널 접속의 표준 프로토콜이었습니다. Telnet은 단순하고 지금도 동작하지만 치명적인 결함이 있습니다—모든 것이 평문으로 전송됩니다. 사용자 이름, 비밀번호, 입력하는 모든 명령, 모든 출력—네트워크 트래픽을 가로챌 수 있는 누구에게나 노출됩니다.

1990년대, 인터넷 백본이 신뢰할 수 있는 학술·연구 네트워크로 구성되었을 때는 이것이 심각한 문제로 여겨지지 않았습니다. 인터넷이 상업화되고 적대적 환경이 되면서 Dsniff나 Ettercap 같은 도구로 LAN에서 Telnet 자격 증명을 가로채는 것이 쉬워졌습니다.

SSH는 모든 것을 암호화하여 이 문제를 해결했습니다. 직접 비교는 다음과 같습니다:

| 기능 | Telnet | SSH |
|---|---|---|
| 암호화 | 없음 | 강력(AES, ChaCha20) |
| 인증 | 평문 비밀번호 | 공개키 + 암호화 비밀번호 |
| 데이터 무결성 | 없음 | HMAC 검증 |
| 포트 | 23 | 22 |
| 현재 권장 여부 | 아니오 | 예 |

rsh(원격 셸)와 rlogin도 유사한 문제를 가지고 있었습니다. 이들은 IP 기반 신뢰에 의존했는데, 이는 쉽게 위조할 수 있습니다. SSH는 이 모든 프로토콜을 대체했으며 25년 이상 도전받지 않는 표준으로 남아 있습니다.

## SSH 키란 무엇인가?

SSH는 두 가지 주요 인증 방식을 지원합니다: 비밀번호와 **SSH 키**. SSH 키는 비대칭 암호화 기반의 암호화 키 쌍—**공개키**와 **개인키**—입니다.

**개인키**는 로컬 머신에 보관하며, 선택적으로 패스프레이즈로 보호할 수 있습니다. **공개키**는 접속하려는 서버의 `~/.ssh/authorized_keys` 파일에 배치합니다. 연결 시 서버는 해당 개인키 보유자만이 응답할 수 있는 암호화 챌린지를 발행합니다. 네트워크를 통해 비밀번호는 전송되지 않습니다.

이 방식에는 여러 장점이 있습니다:
- **무차별 대입 공격 불가** — 추측할 것이 없습니다. 256비트 Ed25519 키는 2²⁵⁶개의 가능한 값을 가집니다.
- **탈취할 비밀번호 없음** — 핸드셰이크를 가로채도 유용한 정보를 얻을 수 없습니다.
- **자동화 가능** — 스크립트와 CI/CD 파이프라인이 사람의 개입 없이 인증할 수 있습니다.
- **폐기 가능** — `authorized_keys`에서 공개키를 제거하면 즉시 접근 권한이 폐기됩니다.

가장 현대적이고 권장되는 키 유형은 **Ed25519**로, Edwards 곡선 위의 타원 곡선 암호화를 사용합니다. RSA 키도 널리 지원되지만, 최소 4096비트를 사용해야 합니다.

키 쌍에 대한 심층 분석은 <a href="/ko/what-is-an-ssh-key/">SSH 키란 무엇인가</a> 가이드를 참조하세요.

## SSH의 주요 활용 사례

SSH는 놀랍도록 다양한 시나리오에서 사용됩니다. 가장 중요한 사례들을 소개합니다.

### 원격 서버 관리

가장 기본적인 활용 사례입니다. 시스템 관리자들은 SSH 터미널 세션을 통해 Linux 및 Unix 서버에 로그인하여 소프트웨어 설치, 설정 파일 편집, 서비스 관리, 문제 진단을 수행합니다.

```bash
ssh user@192.168.1.100
ssh admin@myserver.example.com
ssh -p 2222 deploy@staging.example.com
```

### SFTP와 SCP를 이용한 파일 전송

SSH는 동일한 암호화 채널 위에 구축된 두 가지 파일 전송 프로토콜을 제공합니다:

**SCP(시큐어 카피 프로토콜)**는 파일 복사를 위한 간단한 커맨드라인 도구입니다:

```bash
# 로컬 파일을 원격 서버에 복사
scp report.pdf user@server:/home/user/documents/

# 원격 파일을 로컬 머신에 복사
scp user@server:/var/log/app.log ./local-copy.log

# 전체 디렉토리를 재귀적으로 복사
scp -r ./project/ user@server:/var/www/
```

**SFTP(SSH 파일 전송 프로토콜)**는 FTP와 유사한 인터랙티브 인터페이스를 제공하며, SCP보다 풍부한 기능(전송 재개, 디렉토리 목록 등)을 지원합니다.

### Git 인증

GitHub, GitLab, Bitbucket에 SSH 키를 등록하면 SSH를 통한 모든 `git clone`, `git push`, `git pull` 작업이 개인키로 인증됩니다. 이를 통해 Git 작업마다 사용자 이름과 비밀번호를 입력할 필요가 없어지며, 개발자에게 권장되는 인증 방식입니다.

```bash
# SSH로 저장소 클론
git clone git@github.com:username/repository.git

# GitHub SSH 연결 확인
ssh -T git@github.com
```

### 자동화 배포와 CI/CD

GitHub Actions, GitLab CI, Jenkins 같은 배포 스크립트와 CI/CD 시스템은 SSH 키를 사용하여 사람의 개입 없이 프로덕션 서버에 코드를 배포합니다. 개인키는 CI 시스템의 시크릿으로 저장되고, 대응하는 공개키는 서버의 `authorized_keys` 파일에 추가됩니다.

### 데이터베이스 및 애플리케이션 포트 포워딩

SSH 터널링을 사용하면 원격 서비스를 로컬 머신에 안전하게 노출할 수 있습니다. 예를 들어, 인터넷에 노출되지 않은 원격 MySQL 데이터베이스에 접속하려면:

```bash
ssh -L 3307:localhost:3306 user@dbserver.example.com -N
```

이렇게 하면 로컬 터널이 생성됩니다. 로컬 포트 3307에 연결하면 트래픽이 투명하게 원격 서버의 포트 3306으로 전달되며, SSH 터널 안에서 암호화됩니다.

## SSH 포트와 설정 옵션

### 기본 포트 22

SSH는 기본적으로 **TCP 포트 22**를 사용합니다. 이는 프로토콜 생성 이후 SSH를 위해 예약된 IANA 지정 잘 알려진 포트입니다. 포트를 지정하지 않고 `ssh user@host`를 실행하면 SSH 클라이언트는 포트 22로 연결합니다.

일부 관리자는 로그의 자동 스캔 노이즈를 줄이기 위한 소극적 보안 조치로 SSH 포트를 변경(예: 2222 또는 8022)하기도 합니다. 이것은 "보안을 통한 모호성"이라고 하며 올바른 키 기반 인증의 대체제가 될 수 없지만, 로그 노이즈를 줄이는 효과는 있습니다.

```bash
# 비표준 SSH 포트에 연결
ssh -p 2222 user@example.com
```

### SSH 클라이언트 설정 파일

매번 긴 커맨드라인 옵션을 입력하는 대신 `~/.ssh/config`에서 SSH 연결을 설정할 수 있습니다. 이 파일을 통해 별칭을 정의하고 호스트별 옵션을 설정하여 워크플로를 크게 단순화할 수 있습니다.

```
Host myserver
    HostName 203.0.113.10
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_prod

Host bastion
    HostName jump.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host internal
    HostName 10.0.1.50
    User ubuntu
    ProxyJump bastion
```

이 설정을 사용하면 `ssh myserver`만으로 지정된 키를 사용해 커스텀 포트의 전체 호스트명에 연결됩니다—플래그 없이.

SSH 명령 옵션과 플래그에 대한 완전한 참조는 <a href="/ko/ssh-command/">SSH 명령 가이드</a>를 참조하세요.

### 서버 측 설정(sshd_config)

SSH 서버 데몬(sshd)은 `/etc/ssh/sshd_config`를 통해 설정됩니다. 주요 설정 항목:

```
# 비밀번호 인증 비활성화——키 전용
 PasswordAuthentication no

# root 로그인 비활성화
 PermitRootLogin no

# 리스닝 포트 변경
Port 2222

# 특정 사용자만 허용
AllowUsers deploy admin

# 공개키 인증 활성화
PubkeyAuthentication yes
```

편집 후 데몬 재로드: `sudo systemctl reload sshd`

## SSH 터널링

SSH 터널링(SSH 포트 포워딩이라고도 함)을 사용하면 암호화된 SSH 연결을 통해 임의의 TCP 트래픽을 라우팅할 수 있습니다. SSH에서 가장 강력하지만 잘 활용되지 않는 기능 중 하나입니다.

### 로컬 포트 포워딩

로컬 포워딩은 원격 서비스를 로컬 머신에서 접근 가능하게 만듭니다. 구문은 `-L local_port:remote_host:remote_port`입니다.

```bash
# 로컬에서 원격 PostgreSQL 데이터베이스에 접속
ssh -L 5433:localhost:5432 user@dbserver -N

# 점프 서버를 통해 내부 웹 앱에 접속
ssh -L 8080:internal-app.company.com:80 user@jump.company.com -N
```

`-N` 플래그는 SSH에게 원격 명령을 실행하지 말라고 지시합니다—터널만 필요할 때 유용합니다.

### 원격 포트 포워딩

원격 포워딩은 원격 서버에 로컬 서비스를 노출합니다. 구문은 `-R remote_port:local_host:local_port`입니다.

```bash
# 로컬 웹 서버를 원격 머신에 노출
ssh -R 9090:localhost:3000 user@server -N
```

원격 호스트에 접근 가능한 동료와 로컬 개발 서버를 일시적으로 공유하거나, 클라이언트에게 진행 중인 작업에 대한 임시 접근 권한을 제공할 때 유용합니다.

### 동적 포트 포워딩(SOCKS 프록시)

`-D` 플래그는 로컬 머신에 SOCKS 프록시를 생성하여 SOCKS 호환 트래픽을 SSH 연결을 통해 라우팅합니다. 이것은 실질적으로 원격 서버를 VPN 출구 노드로 사용하는 것입니다.

```bash
# 로컬 포트 1080에 SOCKS5 프록시 생성
ssh -D 1080 user@remote-server -N -f
```

브라우저나 시스템을 `SOCKS5 프록시: 127.0.0.1:1080`으로 설정하면 모든 웹 트래픽이 원격 서버를 통해 라우팅됩니다. 지역 제한 콘텐츠에 접근하거나 신뢰할 수 없는 네트워크에서 안전하게 브라우징할 때 자주 사용됩니다.

## 자주 묻는 질문

### SSH는 무슨 약자인가요?

SSH는 **Secure Shell**의 약자입니다. 안전하지 않은 네트워크에서 네트워크 서비스를 안전하게 운영하기 위한 암호화 네트워크 프로토콜입니다. 이름은 주요 사용 목적인 원격 컴퓨터에서의 안전한 인터랙티브 셸 세션 제공을 나타냅니다.

### SSH와 SSL은 같은 것인가요?

아닙니다. SSH와 SSL/TLS는 모두 암호화 프로토콜이지만 다른 목적으로 사용됩니다. **SSL/TLS**(Secure Sockets Layer/Transport Layer Security)는 웹 트래픽(HTTPS), 이메일, 기타 애플리케이션 프로토콜 보안에 사용됩니다. **SSH**는 안전한 원격 셸 접속, 파일 전송, 터널링을 위해 특별히 설계되었습니다. 두 프로토콜 모두 유사한 암호화 기본 요소(공개키 암호화, 대칭 암호화)를 사용하지만 핸드셰이크 메커니즘과 사용 사례가 다릅니다. SSH는 내부적으로 SSL/TLS를 사용하지 않습니다.

### SSH는 어떤 포트를 사용하나요?

SSH는 기본적으로 **TCP 포트 22**를 사용합니다. 이는 IANA가 할당한 잘 알려진 포트입니다. `/etc/ssh/sshd_config`의 `Port` 지시어를 사용하여 SSH 서버가 다른 포트를 리스닝하도록 설정할 수 있으며, `ssh -p PORT user@host`로 비표준 포트에 연결할 수 있습니다.

### SSH는 해킹될 수 있나요?

키 기반 인증이 올바르게 설정된 SSH 서버는 침해하기 극히 어렵습니다. 암호화 기술—Ed25519 키, AES-256 암호화, SHA-2 MAC—은 키 인증에 대해 양자 내성이 있다고 여겨지며 무차별 대입 공격은 계산상 불가능합니다. 실제 SSH 침해는 일반적으로 개인키 파일 도난, 약한 패스프레이즈, 약한 비밀번호로 비밀번호 인증을 허용하는 잘못 설정된 서버, 패치되지 않은 CVE를 가진 취약한 소프트웨어 버전, 또는 암호화 공격이 아닌 사회공학적 방법에 의한 것입니다. 서버를 최신 상태로 유지하고, 비밀번호 인증을 비활성화하고, 개인키 파일을 보호하세요.

### SSH와 HTTPS의 차이점은 무엇인가요?

SSH와 HTTPS 모두 네트워크 트래픽을 암호화하지만 다른 목적을 위해 설계되었습니다. **HTTPS**는 웹 브라우징을 보호합니다—TLS를 사용하여 브라우저와 웹 서버 간의 HTTP 요청과 응답을 암호화합니다. **SSH**는 원격 관리 접속, 파일 전송, 터널링을 보호합니다. GitHub에 코드를 푸시할 때 SSH(포트 22, 키 기반 인증) 또는 HTTPS(포트 443, 토큰 기반 인증) 중 하나를 사용할 수 있습니다. SSH는 자격 증명을 다시 입력할 필요가 없어 일반적으로 개발자가 선호합니다. HTTPS는 포트 22를 차단할 수 있는 방화벽에서도 보편적으로 지원됩니다.

## 지금 SSH 키를 생성하세요

SSH 키 기반 인증은 비밀번호보다 더 안전하고 더 편리합니다. 아직 SSH 키를 설정하지 않았다면, 저희 무료 온라인 생성기가 브라우저에서 즉시 Ed25519 또는 RSA 키 쌍을 생성합니다—설치 불필요, 서버로 전송되는 데이터 없음.
