# SSH คืออะไร? คู่มือฉบับสมบูรณ์เกี่ยวกับ Secure Shell

SSH ซึ่งย่อมาจาก **Secure Shell** เป็นโปรโตคอลเครือข่ายการเข้ารหัสที่ช่วยให้คุณเข้าถึงและจัดการคอมพิวเตอร์ระยะไกลผ่านเครือข่ายที่ไม่ปลอดภัยได้อย่างปลอดภัย ไม่ว่าคุณจะเข้าสู่ระบบเซิร์ฟเวอร์ Linux พุชโค้ดไปที่ GitHub หรือถ่ายโอนไฟล์ระหว่างเครื่อง SSH เกือบจะเป็นโปรโตคอลที่ทำงานเบื้องหลังอย่างแน่นอน คู่มือนี้จะอธิบายทุกสิ่งที่คุณจำเป็นต้องรู้เกี่ยวกับ SSH: มันคืออะไร ทำงานอย่างไร และเหตุใดจึงกลายเป็นมาตรฐานสากลสำหรับการเข้าถึงระยะไกลอย่างปลอดภัย

## SSH ย่อมาจากอะไร?

**SSH ย่อมาจาก Secure Shell** ชื่อนี้อธิบายได้อย่างชัดเจนว่าโปรโตคอลทำอะไร — โดยให้เซสชันเชลล์ที่ปลอดภัยและเข้ารหัสแก่เครื่องระยะไกล ก่อนที่จะมี SSH ผู้ดูแลระบบใช้เครื่องมือเช่น Telnet และ rsh (เชลล์ระยะไกล) เพื่อจัดการเซิร์ฟเวอร์ระยะไกล โปรโตคอลเหล่านั้นส่งข้อมูลทุกอย่างในรูปแบบข้อความธรรมดา รวมถึงชื่อผู้ใช้และรหัสผ่าน ทำให้ง่ายต่อการดักจับบนเครือข่ายที่ใช้ร่วมกัน

SSH สร้างขึ้นโดยนักวิทยาศาสตร์คอมพิวเตอร์ชาวฟินแลนด์ **Tatu Ylönen** ในปี 1995 โดยได้รับแรงบันดาลใจโดยตรงจากการโจมตีด้วยการดมรหัสผ่านที่เขาพบเห็นในเครือข่ายมหาวิทยาลัยของเขา เขาเปิดตัว SSH-1 ในรูปแบบซอฟต์แวร์ฟรี และมีการนำไปใช้อย่างรวดเร็วบนอินเทอร์เน็ต โปรโตคอลดังกล่าวได้รับมาตรฐานในภายหลังโดย IETF เป็น SSH-2 (RFC 4251–4254) ซึ่งนำเสนอการเข้ารหัสที่แข็งแกร่งยิ่งขึ้น และแก้ไขจุดอ่อนด้านความปลอดภัยหลายประการในการออกแบบดั้งเดิม

ทุกวันนี้ เมื่อมีคนพูดว่า "SSH" พวกเขามักจะหมายถึง SSH-2 เสมอ การใช้งานที่ใช้กันอย่างแพร่หลายที่สุดคือ **OpenSSH** ซึ่งเป็นโปรเจ็กต์โอเพ่นซอร์สที่ดูแลโดยทีมงาน OpenBSD และรวมเข้ากับแทบทุกการแจกจ่าย Linux และการติดตั้ง macOS

**ความหมายของ SSH** ในทางปฏิบัติมีมากกว่าแค่เครื่องมือเข้าสู่ระบบ เป็นชุดโปรโตคอลเต็มรูปแบบที่สนับสนุน:
- รักษาความปลอดภัยการเข้าถึงเทอร์มินัลระยะไกล
- การถ่ายโอนไฟล์ที่เข้ารหัส (SFTP และ SCP)
- การรับรองความถูกต้อง Git (GitHub, GitLab, Bitbucket)
- การส่งต่อพอร์ตและการทันเนลที่ปลอดภัย
- ไปป์ไลน์การปรับใช้อัตโนมัติ

## SSH ทำงานอย่างไร

**โปรโตคอล Secure Shell** ทำงานบนสถาปัตยกรรมแบบเลเยอร์ที่จัดการการเข้ารหัส การรับรองความถูกต้อง และมัลติเพล็กซ์ของช่องสัญญาณแยกกัน การทำความเข้าใจเลเยอร์เหล่านี้ช่วยให้เข้าใจว่าทำไม SSH ถึงได้รับความไว้วางใจสำหรับการดำเนินการที่มีความละเอียดอ่อน

### สามเลเยอร์โปรโตคอล

**1. ชั้นขนส่ง (RFC 4253)**

เมื่อคุณเริ่มต้นการเชื่อมต่อ SSH ไคลเอนต์และเซิร์ฟเวอร์จะทำการจับมือกันก่อนเพื่อเจรจาอัลกอริธึมการเข้ารหัสและสร้างความลับที่ใช้ร่วมกัน สิ่งนี้ใช้อัลกอริธึมการแลกเปลี่ยนคีย์ - SSH สมัยใหม่มีค่าเริ่มต้นเป็นการแลกเปลี่ยนคีย์ที่ใช้ Curve25519 หลังจากการจับมือกัน การสื่อสารที่ตามมาทั้งหมดจะถูกเข้ารหัส เลเยอร์การขนส่งยังให้ความสมบูรณ์ของข้อมูลผ่านรหัสตรวจสอบข้อความ (MAC) ทำให้มั่นใจได้ว่าข้อมูลจะไม่ถูกแก้ไขระหว่างการส่ง

**2. เลเยอร์การตรวจสอบผู้ใช้ (RFC 4252)**

เมื่อสร้างช่องทางที่เข้ารหัสแล้ว เซิร์ฟเวอร์จะขอให้ไคลเอนต์ตรวจสอบสิทธิ์ SSH รองรับวิธีการรับรองความถูกต้องหลายวิธี:

- **การตรวจสอบสิทธิ์คีย์สาธารณะ** — ลูกค้าพิสูจน์ความเป็นเจ้าของคีย์ส่วนตัวโดยไม่ต้องส่งผ่านสาย นี่เป็นวิธีที่ปลอดภัยที่สุดและพบได้บ่อยที่สุด
- **การตรวจสอบรหัสผ่าน** — รหัสผ่านของผู้ใช้จะถูกส่งผ่านช่องทางที่เข้ารหัส สะดวกกว่า แต่เสี่ยงต่อการถูกโจมตีด้วยกำลังดุร้าย
- **คีย์บอร์ดแบบโต้ตอบ** — กลไกการท้าทาย/การตอบสนอง มักใช้สำหรับการตรวจสอบสิทธิ์แบบหลายปัจจัย
- **GSSAPI/Kerberos** — ใช้ในสภาพแวดล้อมองค์กรที่มีการจัดการข้อมูลประจำตัวแบบรวมศูนย์

**3. ชั้นการเชื่อมต่อ (RFC 4254)**

หลังจากการรับรองความถูกต้อง เลเยอร์การเชื่อมต่อจะมัลติเพล็กซ์การเชื่อมต่อที่เข้ารหัสเพียงครั้งเดียวลงในช่องสัญญาณลอจิคัลหลายช่อง นี่คือสิ่งที่ช่วยให้คุณมีเชลล์แบบโต้ตอบ รันคำสั่งระยะไกล ส่งต่อพอร์ต และถ่ายโอนไฟล์ทั้งหมดผ่านการเชื่อมต่อ TCP เดียว

### รายละเอียดการแลกเปลี่ยนคีย์

เมื่อคุณพิมพ์ `ssh user@example.com` นี่คือสิ่งที่เกิดขึ้นในสองสามมิลลิวินาทีแรก:

1. ไคลเอนต์เชื่อมต่อกับพอร์ต TCP 22 บนเซิร์ฟเวอร์
2. ทั้งสองฝ่ายแลกเปลี่ยนสตริงเวอร์ชันโปรโตคอล
3. ทั้งสองฝ่ายส่งรายการอัลกอริธึมที่รองรับ (การแลกเปลี่ยนคีย์, การเข้ารหัส, MAC, การบีบอัด)
4. พวกเขาทำการแลกเปลี่ยนคีย์ Diffie-Hellman หรือ ECDH เพื่อรับคีย์เซสชันที่ใช้ร่วมกัน - นี่คือเหตุผลว่าทำไมถึงแม้จะมีคนบันทึกการรับส่งข้อมูลที่เข้ารหัส พวกเขาไม่สามารถถอดรหัสได้ในภายหลัง
5. เซิร์ฟเวอร์ส่ง **คีย์โฮสต์** (คีย์สาธารณะ) ซึ่งไคลเอนต์ตรวจสอบกับไฟล์ `~/.ssh/known_hosts`
6. การรับรองความถูกต้องดำเนินการตามที่อธิบายไว้ข้างต้น
7. เซสชั่นเริ่มต้นขึ้น

## SSH กับ Telnet: ทำไม SSH ถึงชนะ

ก่อน SSH **Telnet** เคยเป็นโปรโตคอลมาตรฐานสำหรับการเข้าถึงเทอร์มินัลระยะไกล Telnet นั้นเรียบง่ายและยังคงใช้งานได้ แต่มีข้อบกพร่องร้ายแรงประการหนึ่ง: ทุกอย่างถูกส่งในรูปแบบข้อความธรรมดา ชื่อผู้ใช้ของคุณ รหัสผ่านของคุณ ทุกคำสั่งที่คุณพิมพ์ และทุกส่วนของเอาท์พุต — ทั้งหมดนี้สามารถมองเห็นได้โดยใครก็ตามที่สามารถขัดขวางการรับส่งข้อมูลเครือข่ายได้

ในช่วงทศวรรษ 1990 เมื่อแกนหลักอินเทอร์เน็ตได้รับความไว้วางใจจากเครือข่ายวิชาการและการวิจัย สิ่งนี้ไม่ถือว่าเป็นปัญหาร้ายแรง เมื่ออินเทอร์เน็ตกลายเป็นเชิงพาณิชย์และเป็นศัตรูกัน เครื่องมืออย่าง Dsniff และ Ettercap ทำให้การบันทึกข้อมูลรับรอง Telnet บนเครือข่ายท้องถิ่นเป็นเรื่องเล็กน้อย

SSH แก้ไขปัญหานี้ด้วยการเข้ารหัสทุกอย่าง นี่คือการเปรียบเทียบโดยตรง:

| คุณสมบัติ | เทลเน็ต | เอสเอสเอช |
|---|---|---|
| การเข้ารหัส | ไม่มี | แข็งแกร่ง (AES, ChaCha20) |
| การรับรองความถูกต้อง | รหัสผ่านข้อความธรรมดา | กุญแจสาธารณะ + รหัสผ่านที่เข้ารหัส |
| ความสมบูรณ์ของข้อมูล | ไม่มี | การตรวจสอบ HMAC |
| พอร์ต | 23 | 22 |
| ยังคงแนะนำ | ไม่ | ใช่ |

rsh (เชลล์ระยะไกล) และ rlogin มีปัญหาที่คล้ายกัน พวกเขายังอาศัยความไว้วางใจบน IP ซึ่งมีการปลอมแปลงเล็กน้อย SSH เข้ามาแทนที่โปรโตคอลเหล่านี้ทั้งหมด และยังคงเป็นมาตรฐานที่ไม่มีใครทักท้วงมานานกว่า 25 ปี

## คีย์ SSH คืออะไร?

SSH รองรับวิธีการรับรองความถูกต้องหลักสองวิธี: รหัสผ่านและ **คีย์ SSH** คีย์ SSH คือคู่คีย์การเข้ารหัส — **คีย์สาธารณะ** และ **คีย์ส่วนตัว** — ขึ้นอยู่กับการเข้ารหัสแบบไม่สมมาตร

**รหัสส่วนตัว** จะยังคงอยู่ในเครื่องของคุณ โดยมีการป้องกันด้วยข้อความรหัสผ่านที่เป็นตัวเลือก **คีย์สาธารณะ** ถูกวางไว้บนเซิร์ฟเวอร์ใดๆ ที่คุณต้องการเข้าถึง ในไฟล์ชื่อ `~/.ssh/authorized_keys` เมื่อคุณเชื่อมต่อ เซิร์ฟเวอร์จะออกคำถามเกี่ยวกับการเข้ารหัสซึ่งมีเพียงผู้ถือคีย์ส่วนตัวที่เกี่ยวข้องเท่านั้นที่สามารถตอบได้ ไม่มีการส่งรหัสผ่านผ่านเครือข่าย

วิธีนี้มีข้อดีหลายประการ:
- **ภูมิคุ้มกันต่อความรุนแรง** — ไม่มีอะไรให้เดา คีย์ Ed25519 256 บิตมีค่าที่เป็นไปได้ 2²⁵⁶
- **ไม่มีรหัสผ่านให้ขโมย** — แม้ว่าจะมีใครขัดขวางการจับมือกัน แต่ก็ไม่มีประโยชน์อะไรเลย
- **อัตโนมัติ** — สคริปต์และไปป์ไลน์ CI/CD สามารถตรวจสอบสิทธิ์ได้โดยไม่ต้องมีปฏิสัมพันธ์กับมนุษย์
- **เพิกถอนได้** — การลบคีย์สาธารณะออกจาก `authorized_keys` จะเพิกถอนการเข้าถึงทันที

ประเภทคีย์ที่ทันสมัยและแนะนำที่สุดคือ **Ed25519** ซึ่งใช้การเข้ารหัสแบบเส้นโค้งวงรีเหนือเส้นโค้ง Edwards คีย์ RSA ยังได้รับการรองรับอย่างกว้างขวาง แม้ว่าคุณจะควรใช้อย่างน้อย 4,096 บิตก็ตาม

หากต้องการเจาะลึกคู่คีย์ โปรดดูคำแนะนำของเราเกี่ยวกับ<a href="/th/what-is-an-ssh-key">คีย์ SSH คืออะไร</a>

## กรณีการใช้งาน SSH ทั่วไป

SSH ถูกใช้ในสถานการณ์ที่หลากหลายอย่างน่าทึ่ง นี่คือสิ่งที่สำคัญที่สุด

### การดูแลเซิร์ฟเวอร์ระยะไกล

กรณีการใช้งานขั้นพื้นฐานที่สุด ผู้ดูแลระบบเข้าสู่ระบบเซิร์ฟเวอร์ Linux และ Unix เพื่อติดตั้งซอฟต์แวร์ แก้ไขไฟล์การกำหนดค่า จัดการบริการ และวินิจฉัยปัญหา — ทั้งหมดนี้ผ่านเซสชั่นเทอร์มินัล SSH

**SCP (Secure Copy Protocol)** เป็นเครื่องมือบรรทัดคำสั่งง่ายๆ สำหรับการคัดลอกไฟล์:

**SFTP (SSH File Transfer Protocol)** มอบอินเทอร์เฟซที่คล้ายกับ FTP แบบโต้ตอบพร้อมชุดฟีเจอร์ที่สมบูรณ์กว่า SCP รวมถึงการถ่ายโอนต่อและรายการไดเร็กทอรี

### การรับรองความถูกต้องของ Git

เมื่อคุณเพิ่มคีย์ SSH ให้กับ GitHub, GitLab หรือ Bitbucket ทุก `git clone`, `git push` และ `git pull` บน SSH จะได้รับการตรวจสอบสิทธิ์โดยใช้คีย์ส่วนตัวของคุณ ซึ่งช่วยลดความจำเป็นในการป้อนชื่อผู้ใช้และรหัสผ่านในทุกการทำงานของ Git และเป็นวิธีการตรวจสอบสิทธิ์ที่แนะนำสำหรับนักพัฒนา

### การปรับใช้อัตโนมัติและ CI/CD

สคริปต์การปรับใช้และระบบ CI/CD เช่น GitHub Actions, GitLab CI และ Jenkins ใช้คีย์ SSH เพื่อปรับใช้โค้ดกับเซิร์ฟเวอร์ที่ใช้งานจริงโดยไม่ต้องมีการแทรกแซงจากมนุษย์ รหัสส่วนตัวจะถูกเก็บไว้เป็นความลับในระบบ CI และรหัสสาธารณะที่เกี่ยวข้องจะถูกเพิ่มลงในไฟล์ `authorized_keys` ของเซิร์ฟเวอร์

## พอร์ต SSH และตัวเลือกการกำหนดค่า

### พอร์ตเริ่มต้น 22

SSH ฟังบน **TCP พอร์ต 22** โดยค่าเริ่มต้น นี่คือพอร์ตที่รู้จักซึ่งกำหนดโดย IANA ซึ่งสงวนไว้สำหรับ SSH นับตั้งแต่สร้างโปรโตคอล เมื่อคุณเรียกใช้ `ssh user@host` โดยไม่ระบุพอร์ต ไคลเอ็นต์ SSH จะเชื่อมต่อกับพอร์ต 22

ผู้ดูแลระบบบางรายเปลี่ยนพอร์ต SSH (เช่น เป็น 2222 หรือ 8022) เป็นมาตรการรักษาความปลอดภัยเล็กน้อย เพื่อลดสัญญาณรบกวนการสแกนอัตโนมัติในบันทึก สิ่งนี้เรียกว่า "การรักษาความปลอดภัยผ่านความสับสน" และไม่สามารถทดแทนการตรวจสอบสิทธิ์ตามคีย์ที่เหมาะสมได้ แต่สามารถลดความยุ่งเหยิงของบันทึกได้

@@รหัส4@@

### ไฟล์การกำหนดค่าไคลเอ็นต์ SSH

แทนที่จะพิมพ์ตัวเลือกบรรทัดคำสั่งแบบยาวทุกครั้ง คุณสามารถกำหนดค่าการเชื่อมต่อ SSH ได้ใน `~/.ssh/config` ไฟล์นี้ช่วยให้คุณกำหนดนามแฝง ตั้งค่าตัวเลือกต่อโฮสต์ และทำให้ขั้นตอนการทำงานของคุณง่ายขึ้นอย่างมาก

```
Host myserver
    HostName 203.0.113.10
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_prod

Host bastion
    HostName jump.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host internal
    HostName 10.0.1.50
    User ubuntu
    ProxyJump bastion
```

ด้วยการกำหนดค่านี้ `ssh myserver` จะเชื่อมต่อกับชื่อโฮสต์แบบเต็มบนพอร์ตที่กำหนดเองโดยใช้คีย์ที่ระบุ โดยไม่จำเป็นต้องใช้แฟล็ก

สำหรับการอ้างอิงโดยสมบูรณ์เกี่ยวกับตัวเลือกคำสั่ง SSH และแฟล็ก โปรดดู<a href="/th/ssh-command">คู่มือคำสั่ง SSH</a> ของเรา

## การขุดอุโมงค์ SSH

SSH tunneling (หรือที่เรียกว่าการส่งต่อพอร์ต SSH) ช่วยให้คุณสามารถกำหนดเส้นทางการรับส่งข้อมูล TCP โดยพลการผ่านการเชื่อมต่อ SSH ที่เข้ารหัส มันเป็นหนึ่งในคุณสมบัติที่ทรงพลังและใช้งานน้อยที่สุดของ SSH

### การส่งต่อพอร์ตท้องถิ่น

การส่งต่อในพื้นที่ทำให้สามารถเข้าถึงบริการระยะไกลบนเครื่องท้องถิ่นของคุณได้ ไวยากรณ์คือ `-L local_port:remote_host:remote_port`

@@รหัส7@@

การตั้งค่าสถานะ `-N` จะบอก SSH ไม่ให้ดำเนินการคำสั่งระยะไกล ซึ่งมีประโยชน์เมื่อคุณต้องการเฉพาะช่องสัญญาณเท่านั้น

### การส่งต่อพอร์ตระยะไกล

การส่งต่อระยะไกลจะแสดงบริการภายในเครื่องบนเซิร์ฟเวอร์ระยะไกล ไวยากรณ์คือ `-R remote_port:local_host:local_port`

@@รหัส8@@

สิ่งนี้มีประโยชน์สำหรับการแชร์เซิร์ฟเวอร์การพัฒนาในเครื่องเป็นการชั่วคราวกับเพื่อนร่วมงานที่มีสิทธิ์เข้าถึงโฮสต์ระยะไกล หรือเพื่อให้ไคลเอนต์เข้าถึงงานของคุณชั่วคราวได้

### การส่งต่อพอร์ตแบบไดนามิก (SOCKS Proxy)

การตั้งค่าสถานะ `-D` จะสร้างพร็อกซี SOCKS บนเครื่องของคุณ โดยกำหนดเส้นทางการรับส่งข้อมูลที่เข้ากันได้กับ SOCKS ทั้งหมดผ่านการเชื่อมต่อ SSH สิ่งนี้ใช้เซิร์ฟเวอร์ระยะไกลเป็นโหนดทางออก VPN อย่างมีประสิทธิภาพ

@@รหัส9@@

กำหนดค่าเบราว์เซอร์หรือระบบของคุณให้ใช้ `SOCKS5 proxy: 127.0.0.1:1080` และการรับส่งข้อมูลเว็บทั้งหมดจะถูกส่งผ่านเซิร์ฟเวอร์ระยะไกล มักใช้เพื่อเข้าถึงเนื้อหาที่ถูกจำกัดทางภูมิศาสตร์หรือเรียกดูอย่างปลอดภัยจากเครือข่ายที่ไม่น่าเชื่อถือ

## คำถามที่พบบ่อย

### SSH ย่อมาจากอะไร?

SSH ย่อมาจาก **Secure Shell** เป็นโปรโตคอลเครือข่ายการเข้ารหัสสำหรับการให้บริการเครือข่ายอย่างปลอดภัยผ่านเครือข่ายที่ไม่ปลอดภัย ชื่อนี้อ้างถึงกรณีการใช้งานหลัก: จัดให้มีเซสชันเชลล์แบบโต้ตอบที่ปลอดภัยบนคอมพิวเตอร์ระยะไกล

### SSH เหมือนกับ SSL หรือไม่

ไม่ SSH และ SSL/TLS เป็นโปรโตคอลการเข้ารหัสทั้งคู่ แต่มีวัตถุประสงค์ที่แตกต่างกัน **SSL/TLS** (Secure Sockets Layer / Transport Layer Security) ใช้เพื่อรักษาความปลอดภัยการรับส่งข้อมูลเว็บ (HTTPS) อีเมล และโปรโตคอลแอปพลิเคชันอื่นๆ **SSH** ได้รับการออกแบบมาโดยเฉพาะสำหรับการเข้าถึงเชลล์ระยะไกลที่ปลอดภัย การถ่ายโอนไฟล์ และการขุดอุโมงค์ พวกเขาใช้การเข้ารหัสแบบดั้งเดิมที่คล้ายกัน (การเข้ารหัสคีย์สาธารณะ การเข้ารหัสแบบสมมาตร) แต่มีกลไกการจับมือและกรณีการใช้งานที่แตกต่างกัน SSH ไม่ได้ใช้ SSL/TLS ภายใน

### SSH ใช้พอร์ตใด

SSH ใช้ **TCP พอร์ต 22** เป็นค่าเริ่มต้น นี่คือท่าเรือที่มีชื่อเสียงซึ่งได้รับมอบหมายจาก IANA คุณสามารถกำหนดค่าเซิร์ฟเวอร์ SSH ให้ฟังบนพอร์ตอื่นได้โดยใช้คำสั่ง `Port` ใน `/etc/ssh/sshd_config` และเชื่อมต่อกับพอร์ตที่ไม่ได้มาตรฐานด้วย `ssh -p PORT user@host`

### SSH สามารถแฮ็กได้หรือไม่?

เซิร์ฟเวอร์ SSH ที่กำหนดค่าอย่างเหมาะสมพร้อมการตรวจสอบสิทธิ์แบบใช้คีย์นั้นยากต่อการประนีประนอมอย่างยิ่ง การเข้ารหัส — คีย์ Ed25519, การเข้ารหัส AES-256, SHA-2 MAC — ถือว่าทนทานต่อควอนตัมสำหรับการรับรองความถูกต้องของคีย์ และเป็นไปไม่ได้ในการคำนวณด้วยกำลังดุร้าย โดยทั่วไปแล้ว การประนีประนอม SSH ในโลกแห่งความเป็นจริงจะเกี่ยวข้องกับ: ไฟล์คีย์ส่วนตัวที่ถูกขโมย, ข้อความรหัสผ่านที่ไม่รัดกุม, เซิร์ฟเวอร์ที่กำหนดค่าไม่ถูกต้องซึ่งอนุญาตให้มีการตรวจสอบรหัสผ่านด้วยรหัสผ่านที่ไม่รัดกุม, เวอร์ชันซอฟต์แวร์ที่มีช่องโหว่พร้อม CVE ที่ไม่ได้รับแพตช์ หรือวิศวกรรมสังคมมากกว่าการโจมตีด้วยการเข้ารหัส อัปเดตเซิร์ฟเวอร์ของคุณ ปิดใช้งานการตรวจสอบสิทธิ์ด้วยรหัสผ่าน และปกป้องไฟล์คีย์ส่วนตัวของคุณ

### SSH และ HTTPS แตกต่างกันอย่างไร

ทั้ง SSH และ HTTPS เข้ารหัสการรับส่งข้อมูลเครือข่าย แต่ได้รับการออกแบบมาเพื่อวัตถุประสงค์ที่แตกต่างกัน **HTTPS** รักษาความปลอดภัยในการท่องเว็บ — ใช้ TLS เพื่อเข้ารหัสคำขอ HTTP และการตอบกลับระหว่างเบราว์เซอร์และเว็บเซิร์ฟเวอร์ **SSH** รักษาความปลอดภัยการเข้าถึงการดูแลระบบระยะไกล การถ่ายโอนไฟล์ และการขุดอุโมงค์ เมื่อคุณพุชโค้ดไปที่ GitHub คุณสามารถใช้ SSH (พอร์ต 22, การตรวจสอบสิทธิ์แบบใช้คีย์) หรือ HTTPS (พอร์ต 443, การตรวจสอบสิทธิ์แบบโทเค็น) โดยทั่วไปแล้วนักพัฒนาต้องการ SSH เนื่องจากไม่จำเป็นต้องป้อนข้อมูลรับรองอีกครั้ง HTTPS ได้รับการสนับสนุนในระดับสากลผ่านไฟร์วอลล์ที่อาจบล็อกพอร์ต 22

## สร้างคีย์ SSH ของคุณทันที

การตรวจสอบสิทธิ์โดยใช้คีย์ SSH มีความปลอดภัยและสะดวกกว่ารหัสผ่าน หากคุณยังไม่ได้ตั้งค่าคีย์ SSH เครื่องมือสร้างออนไลน์ฟรีของเราจะสร้างคู่คีย์ Ed25519 หรือ RSA ทันทีในเบราว์เซอร์ของคุณ โดยไม่จำเป็นต้องติดตั้ง และไม่มีการส่งอะไรเลยไปยังเซิร์ฟเวอร์ของเรา
