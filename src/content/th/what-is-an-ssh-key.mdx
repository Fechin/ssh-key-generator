# คีย์ SSH คืออะไร อธิบายกุญแจสาธารณะ กุญแจส่วนตัว และคู่กุญแจ

**คีย์ SSH** คือข้อมูลรับรองการเข้ารหัสที่ใช้ในการตรวจสอบสิทธิ์เซิร์ฟเวอร์ SSH โดยไม่ต้องใช้รหัสผ่าน คีย์ SSH มาเป็นคู่กัน — **คีย์สาธารณะ** และ **คีย์ส่วนตัว** — เมื่อรวมกันแล้วทำให้การเข้าถึงระยะไกลมีความปลอดภัยและสะดวกยิ่งขึ้นกว่าการตรวจสอบสิทธิ์ด้วยรหัสผ่านแบบเดิม คู่มือนี้จะอธิบายว่าคีย์ SSH คืออะไร ระบบคู่คีย์ทำงานอย่างไร ประเภทคีย์ที่ควรเลือก และวิธีใช้คีย์ SSH กับบริการต่างๆ เช่น GitHub และ GitLab

## คีย์ SSH ทำงานอย่างไร

คีย์ SSH อิงตาม **การเข้ารหัสแบบอสมมาตร** หรือที่เรียกว่าการเข้ารหัสคีย์สาธารณะ ต่างจากการเข้ารหัสแบบสมมาตร (โดยที่คีย์ลับเพียงคีย์เดียวทั้งเข้ารหัสและถอดรหัสข้อมูล) ระบบที่ไม่สมมาตรใช้คู่คีย์ที่เชื่อมโยงทางคณิตศาสตร์ โดยที่ข้อมูลที่เข้ารหัสด้วยคีย์เดียวสามารถถอดรหัสได้ด้วยอีกคีย์เดียวเท่านั้น

ขั้นตอนการตรวจสอบสิทธิ์เมื่อคุณใช้คีย์ SSH เพื่อเข้าสู่ระบบเซิร์ฟเวอร์มีดังนี้

1. คุณเรียกใช้ `ssh user@server.example.com`
2. เซิร์ฟเวอร์ค้นหารหัสสาธารณะของคุณใน `~/.ssh/authorized_keys`
3. เซิร์ฟเวอร์สร้างความท้าทายแบบสุ่มและเข้ารหัสด้วยรหัสสาธารณะของคุณ
4. ไคลเอนต์ SSH ของคุณถอดรหัสความท้าทายโดยใช้รหัสส่วนตัวของคุณ
5. ไคลเอนต์ส่งหลักฐานการเข้ารหัส (ที่ได้มาจากความท้าทายที่ถอดรหัสและ ID เซสชัน) กลับไปที่เซิร์ฟเวอร์
6. เซิร์ฟเวอร์ตรวจสอบหลักฐาน มีเพียงผู้ถือคีย์ส่วนตัวเท่านั้นที่สามารถผลิตมันได้
7. การรับรองความถูกต้องสำเร็จ ไม่มีรหัสผ่านใดหลุดออกจากเครื่องของคุณ

นี่คือข้อมูลเชิงลึกที่สำคัญ: **คีย์ส่วนตัวของคุณไม่เคยเดินทางผ่านเครือข่าย** เซิร์ฟเวอร์จะมองเห็นคีย์สาธารณะของคุณและหลักฐานการเข้ารหัสที่คุณสร้างขึ้นเท่านั้น แม้ว่าบางคนจะสกัดกั้นทุกแพ็กเก็ตในเซสชัน SSH ของคุณ พวกเขาก็ไม่ได้รับอะไรเลยที่ช่วยให้พวกเขาปลอมตัวเป็นคุณได้

หากต้องการทราบความเป็นมาของโปรโตคอล SSH ที่กว้างขึ้น โปรดดูคำแนะนำของเราเกี่ยวกับ <a href="/th/what-is-ssh/">SSH คืออะไร</a>

## กุญแจสาธารณะกับกุญแจส่วนตัว: อะไรคือความแตกต่าง?

การทำความเข้าใจความแตกต่างระหว่างคีย์ทั้งสองเป็นสิ่งสำคัญสำหรับการใช้ SSH อย่างถูกต้อง

### กุญแจสาธารณะ — ล็อคของคุณ

**กุญแจสาธารณะ** ได้รับการออกแบบให้แชร์ได้อย่างอิสระ คิดว่ามันเป็นแม่กุญแจ: คุณแจกจ่ายสำเนาให้กับทุกคนที่คุณต้องการเข้าถึงระบบ เมื่อเซิร์ฟเวอร์มีรหัสสาธารณะของคุณในไฟล์ `authorized_keys` เซิร์ฟเวอร์จะสามารถตรวจสอบตัวตนของคุณได้โดยไม่ต้องรู้รหัสส่วนตัวของคุณ

กุญแจสาธารณะ:
- ปลอดภัยในการแบ่งปัน - ไม่สามารถใช้เพื่อแอบอ้างเป็นคุณได้
- ถูกวางไว้บนเซิร์ฟเวอร์ใน `~/.ssh/authorized_keys`
- อัปโหลดไปยัง GitHub, GitLab และบริการอื่น ๆ
- โดยทั่วไปจะมีนามสกุล `.pub` (เช่น `id_ed25519.pub`)
- ดูเหมือน: `ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... user@hostname`

### คีย์ส่วนตัว — คีย์ของคุณ

**คีย์ส่วนตัว** คือความลับครึ่งหนึ่งของคู่นี้ มันจะต้องไม่ออกจากเครื่องของคุณและไม่ควรแบ่งปันกับใครก็ตาม แม้แต่ผู้ดูแลระบบ ไม่ใช่ผู้ให้บริการคลาวด์ หรือเจ้าหน้าที่ฝ่ายสนับสนุน หากคีย์ส่วนตัวของคุณถูกบุกรุก ใครก็ตามที่ได้รับคีย์นี้สามารถปลอมตัวเป็นคุณบนเซิร์ฟเวอร์ทุกเครื่องที่ติดตั้งคีย์สาธารณะของคุณได้

คีย์ส่วนตัว:
- จะต้องเก็บเป็นความลับและปลอดภัย
- ถูกจัดเก็บไว้ใน `~/.ssh/` โดยมีสิทธิ์ที่จำกัด (600)
- สามารถเลือกเข้ารหัสด้วยข้อความรหัสผ่านได้
- ไม่มีนามสกุลหรือนามสกุลเฉพาะภาษา (เช่น `id_ed25519`)

### ไฟล์ `authorized_keys`

ทางฝั่งเซิร์ฟเวอร์ กุญแจสาธารณะของผู้ใช้แต่ละคนจะถูกเก็บไว้ใน `~/.ssh/authorized_keys` นี่คือไฟล์ข้อความธรรมดาที่แต่ละบรรทัดมีคีย์สาธารณะหนึ่งคีย์ เมื่อไคลเอนต์ SSH พยายามตรวจสอบสิทธิ์ เซิร์ฟเวอร์จะตรวจสอบว่าไคลเอนต์สามารถพิสูจน์การครอบครองคีย์ส่วนตัวที่สอดคล้องกับรายการใด ๆ ในไฟล์นี้ได้หรือไม่

คุณสามารถมีคีย์สาธารณะได้มากเท่าที่คุณต้องการในไฟล์นี้ ทำให้อุปกรณ์หรือระบบอัตโนมัติหลายเครื่องสามารถตรวจสอบสิทธิ์ไปยังเซิร์ฟเวอร์เดียวกันได้

## อัลกอริธึมคีย์ SSH: Ed25519 กับ RSA

เมื่อคุณสร้างคู่คีย์ SSH คุณจะต้องเลือกอัลกอริทึม สองตัวเลือกที่พบบ่อยที่สุดในปัจจุบันคือ **Ed25519** และ **RSA** นอกจากนี้ยังมี ECDSA และ DSA อีกด้วย แต่ DSA เลิกใช้งานแล้ว และ ECDSA มีข้อผิดพลาดในการใช้งาน ดังนั้นเราจะเน้นที่ตัวเลือกที่แนะนำ 2 ตัว

### Ed25519 — ทางเลือกที่ทันสมัย

Ed25519 เปิดตัวกับ OpenSSH ในเวอร์ชัน 6.5 (2014) และตอนนี้เป็นค่าเริ่มต้นที่แนะนำแล้ว ใช้การเข้ารหัสเส้นโค้งวงรีบนเส้นโค้ง Twisted Edwards (Curve25519) ซึ่งออกแบบโดย Daniel J. Bernstein

**ข้อดีของ Ed25519:**
- **กะทัดรัด** — คีย์มีความยาวเพียง 68 ตัวอักษร (คีย์สาธารณะ) และไฟล์มีขนาดเล็ก
- **รวดเร็ว** — การสร้างลายเซ็นและการตรวจสอบทำได้อย่างรวดเร็วมาก
- **ปลอดภัยด้วยการออกแบบ** — ไม่เสี่ยงต่อการโจมตีช่องทางด้านข้างที่ส่งผลต่อการใช้งาน RSA บางอย่าง
- **ขนาดคีย์คงที่** — ระดับความปลอดภัย 256 บิต ไม่จำเป็นต้องเลือกขนาดคีย์
- **กำหนด** — ลายเซ็นถูกกำหนดไว้ ซึ่งช่วยลดความล้มเหลวของตัวสร้างตัวเลขสุ่ม

@@รหัส1@@

### RSA — ตัวเลือกสากล

RSA เป็นอัลกอริธึมรุ่นเก่า (คิดค้นในปี 1977) และถึงแม้จะยังคงมีความปลอดภัยอยู่ที่ 4,096 บิต แต่ก็ช้ากว่าและสร้างคีย์ที่ใหญ่กว่า Ed25519 อย่างไรก็ตาม RSA มีความเข้ากันได้กว้างกว่ามาก — ทำงานได้กับเซิร์ฟเวอร์ SSH เก่ามาก ระบบเดิม และคีย์ความปลอดภัยฮาร์ดแวร์บางตัว

**เมื่อใดจึงควรใช้ RSA:**
- คุณต้องเชื่อมต่อกับระบบเดิมที่ใช้ OpenSSH เวอร์ชันเก่า (ก่อน 6.5)
- คุณกำลังใช้สมาร์ทการ์ดหรือ HSM ที่ไม่รองรับ Ed25519
- จำเป็นต้องมีความเข้ากันได้กับเครื่องมือรุ่นเก่า

@@รหัส2@@

**อย่าใช้ RSA ที่น้อยกว่า 4096 บิต** RSA-1024 และ RSA-2048 ถือว่าไม่เพียงพอมากขึ้นเรื่อยๆ เมื่อพลังในการคำนวณเพิ่มมากขึ้น Ed25519 ที่ 256 บิตให้ความปลอดภัยที่เทียบเท่าหรือดีกว่าโดยมีขนาดเพียงเศษเสี้ยวของคีย์

### การเปรียบเทียบโดยสรุป

| คุณสมบัติ | Ed25519 | อาร์เอสเอ 4096 |
|---|---|---|
| ขนาดคีย์ (สาธารณะ) | ~68 ตัวอักษร | ~725 ตัวอักษร |
| ระดับความปลอดภัย | 128 บิต (โครงสร้างต้านทานหลังควอนตัม) | ~ 140 บิตคลาสสิก |
| ความเร็ว | เร็วมาก | ช้าลง |
| ความเข้ากันได้ | SSH สมัยใหม่ (OpenSSH 6.5+) | สากล |
| แนะนำสำหรับ | กุญแจใหม่ทั้งหมด | ระบบเดิมเท่านั้น |

**สิ่งสำคัญที่สุด: สร้างคีย์ Ed25519 สำหรับกรณีการใช้งานสมัยใหม่** ใช้ RSA เมื่อบังคับตามข้อกำหนดด้านความเข้ากันได้เท่านั้น

## วิธีเพิ่มคีย์ SSH ใน GitHub

การเพิ่มคีย์สาธารณะ SSH ของคุณลงใน GitHub ช่วยให้คุณสามารถส่งและดึง SSH ได้โดยไม่ต้องป้อนชื่อผู้ใช้และรหัสผ่านในทุกการดำเนินการ นี่คือกระบวนการทีละขั้นตอน

หากคุณยังไม่มีคู่คีย์ SSH ให้สร้างขึ้นใหม่ คุณสามารถใช้<a href="/">เครื่องมือสร้างคีย์ SSH ออนไลน์ฟรี</a>ของเรา หรือเรียกใช้:

### ขั้นตอนที่ 2: คัดลอกคีย์สาธารณะของคุณ

แสดงกุญแจสาธารณะของคุณและคัดลอกไปยังคลิปบอร์ดของคุณ:

### ขั้นตอนที่ 3: เพิ่มรหัสลงใน GitHub

1. เข้าสู่ระบบ **GitHub.com**
2. คลิกที่รูปโปรไฟล์ของคุณ → **การตั้งค่า**
3. ในแถบด้านข้างซ้าย คลิก **คีย์ SSH และ GPG**
4. คลิก **คีย์ SSH ใหม่**
5. ตั้งชื่อที่สื่อความหมาย (เช่น "MacBook Pro 2024" หรือ "แล็ปท็อปสำหรับทำงาน")
6. วางคีย์สาธารณะของคุณลงในช่อง **คีย์**
7. คลิก **เพิ่มคีย์ SSH**

## ข้อความรหัสผ่านคีย์ SSH

เมื่อคุณสร้างคู่คีย์ SSH `ssh-keygen` จะถามว่าคุณต้องการตั้งค่า **ข้อความรหัสผ่าน** หรือไม่ นี่เป็นรหัสผ่านทางเลือกที่เข้ารหัสไฟล์คีย์ส่วนตัวบนดิสก์

### ทำไมต้องใช้ข้อความรหัสผ่าน?

หากไฟล์คีย์ส่วนตัวของคุณได้รับการปกป้องด้วยข้อความรหัสผ่านที่รัดกุม การขโมยไฟล์นั้นไม่เพียงพอที่จะปลอมตัวเป็นคุณ ผู้โจมตีจะต้องถอดรหัสข้อความรหัสผ่านด้วย นี่เป็นการรักษาความปลอดภัยชั้นที่สองที่สำคัญ โดยเฉพาะอย่างยิ่งบนแล็ปท็อปที่อาจถูกขโมยทางกายภาพ

หากไม่มีข้อความรหัสผ่าน คีย์ส่วนตัวของคุณจะถูกจัดเก็บไว้ในข้อความธรรมดาบนดิสก์ ใครก็ตามที่สามารถอ่านไฟล์ได้ (ผ่านมัลแวร์ การเข้าถึงทางกายภาพ หรือการสำรองข้อมูลที่ไม่ปลอดภัยอย่างเหมาะสม) จะสามารถใช้งานได้ทันที

## ลายนิ้วมือคีย์ SSH

คู่คีย์ SSH ทุกคู่มี **ลายนิ้วมือ** ซึ่งเป็นแฮชแบบสั้นของคีย์สาธารณะที่ระบุได้โดยไม่ซ้ำกัน ลายนิ้วมือมีจุดประสงค์หลายประการ:

- **ยืนยันตัวตนของโฮสต์** — เมื่อคุณเชื่อมต่อกับเซิร์ฟเวอร์เป็นครั้งแรก SSH จะแสดงลายนิ้วมือของคีย์โฮสต์ให้คุณเห็น หากคุณยืนยันว่าตรงกับที่คุณคาดหวัง (เช่น จัดทำโดยผู้ให้บริการโฮสติ้งของคุณ) คุณจะรู้ว่าคุณกำลังเชื่อมต่อกับเซิร์ฟเวอร์ที่ถูกต้อง
- **ระบุคีย์** — เมื่อจัดการหลายคีย์ ลายนิ้วมือจะช่วยคุณระบุคีย์จริงที่สอดคล้องกับรายการใดใน `authorized_keys` หรือบนบริการ เช่น GitHub
- **การตรวจสอบและการเพิกถอน** — บันทึกการตรวจสอบมักจะบันทึกลายนิ้วมือมากกว่าคีย์ทั้งหมด การรู้ลายนิ้วมือของกุญแจช่วยให้คุณค้นหาและลบออกได้หากจำเป็น

## คำถามที่พบบ่อย

### คีย์ SSH ใช้ทำอะไร?

คีย์ SSH จะตรวจสอบสิทธิ์คุณกับเซิร์ฟเวอร์ระยะไกลโดยไม่ต้องใช้รหัสผ่าน การใช้งานที่พบบ่อยที่สุดคือ: การเข้าสู่ระบบเซิร์ฟเวอร์ Linux/Unix เพื่อการดูแลระบบ การตรวจสอบสิทธิ์กับบริการโฮสติ้ง Git (GitHub, GitLab, Bitbucket) สำหรับการเข้าถึงแบบพุช/พุล สคริปต์การปรับใช้อัตโนมัติที่ต้องใช้การเข้าถึงเซิร์ฟเวอร์แบบไม่ใช้รหัสผ่าน และไปป์ไลน์ CI/CD ที่ปรับใช้โค้ดกับการใช้งานจริง แนะนำให้ใช้คีย์ SSH มากกว่ารหัสผ่าน เนื่องจากมีภูมิคุ้มกันต่อการใช้กำลังดุร้าย ไม่สามารถนำมาใช้ซ้ำในการละเมิด และสามารถใช้ได้โดยไม่ต้องมีปฏิสัมพันธ์กับมนุษย์

### ฉันควรมีคีย์ SSH จำนวนเท่าใด

ไม่มีคำตอบที่ถูกต้องเพียงข้อเดียว แต่แนวทางปฏิบัติที่ดีที่สุดโดยทั่วไปคือ **หนึ่งคีย์ต่ออุปกรณ์** (ไม่ใช่ต่อบริการ) แล็ปท็อปของคุณมีคู่คีย์หนึ่งคู่ เดสก์ท็อปของคุณมีอีกอันหนึ่ง เซิร์ฟเวอร์ CI ของคุณมีเซิร์ฟเวอร์อื่น ด้วยวิธีนี้ หากอุปกรณ์ถูกขโมยหรือถูกบุกรุก คุณจะเพิกถอนเฉพาะคีย์ของอุปกรณ์นั้นเท่านั้น บางคนยังแยกตามบริบท (ส่วนตัวกับที่ทำงาน) ส่งผลให้มีคีย์สองคีย์ต่ออุปกรณ์ หลีกเลี่ยงการแชร์คีย์เดียวในหลายเครื่อง หากคีย์นั้นถูกบุกรุก การเข้าถึงทั้งหมดของคุณก็จะตกอยู่ในความเสี่ยงพร้อมกัน

### ฉันสามารถแชร์คีย์สาธารณะ SSH ของฉันได้หรือไม่

ใช่. กุญแจสาธารณะได้รับการออกแบบเพื่อใช้ร่วมกัน สามารถส่งอีเมล โพสต์ในเอกสาร GitHub หรือรวมไว้ในตั๋วเพื่อขอการเข้าถึงเซิร์ฟเวอร์ได้อย่างปลอดภัย คีย์สาธารณะเพียงอย่างเดียวไม่สามารถใช้ในการตรวจสอบสิทธิ์ในฐานะคุณได้ ซึ่งต้องใช้คีย์ส่วนตัวที่เกี่ยวข้อง ซึ่งมีเพียงคุณเท่านั้นที่มี นี่คือคุณสมบัติพื้นฐานของการเข้ารหัสแบบอสมมาตร อย่าเปิดเผยรหัส **ส่วนตัว** ของคุณกับใครเลย

### จะเกิดอะไรขึ้นหากฉันทำคีย์ส่วนตัว SSH หาย

หากคุณทำคีย์ส่วนตัวหายและไม่มีข้อมูลสำรอง คุณจะสูญเสียความสามารถในการตรวจสอบสิทธิ์ด้วยคีย์ดังกล่าว รหัสสาธารณะของคุณที่ยังคงติดตั้งอยู่บนเซิร์ฟเวอร์และบริการต่างๆ จะกลายเป็นเด็กกำพร้า คุณจะต้องมีทางเลือกอื่นในการเข้าถึงเซิร์ฟเวอร์ (หากเปิดใช้งานการตรวจสอบรหัสผ่าน การเข้าถึงคอนโซล กลไกการเข้าถึงฉุกเฉินของผู้ให้บริการโฮสต์ของคุณ) เพื่อลบรหัสสาธารณะเก่าออกจาก `authorized_keys` และเพิ่มรหัสใหม่ นี่คือเหตุผลว่าทำไมการสำรองข้อมูลคีย์ส่วนตัวของคุณ (ในตัวจัดการรหัสผ่านหรือการสำรองข้อมูลที่เข้ารหัส) จึงมีความสำคัญ ไม่สามารถสร้างคีย์ส่วนตัวขึ้นใหม่จากคีย์สาธารณะได้

### Ed25519 หรือ RSA ดีกว่ากัน?

**Ed25519 ดีกว่าสำหรับการสร้างคีย์ใหม่ทั้งหมด** เร็วกว่า สร้างคีย์ที่เล็กกว่า ไม่เสี่ยงต่อจุดอ่อนในการใช้งานบางอย่างที่ส่งผลต่อ RSA และให้ความปลอดภัยที่เทียบเท่าหรือเหนือกว่าด้วยคีย์ 256 บิต เมื่อเทียบกับ RSA-4096 เหตุผลเดียวที่ควรใช้ RSA ในปัจจุบันก็คือความเข้ากันได้กับซอฟต์แวร์เซิร์ฟเวอร์ SSH รุ่นเก่าๆ (OpenSSH ก่อนเวอร์ชัน 6.5 เปิดตัวในปี 2014) หรือโมดูลความปลอดภัยของฮาร์ดแวร์รุ่นเก่า หากคุณกำลังเข้าถึงเซิร์ฟเวอร์หรือบริการสมัยใหม่เช่น GitHub ให้ใช้ Ed25519

## สร้างคู่คีย์ SSH ของคุณ
