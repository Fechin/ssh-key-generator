# Команда SSH: повний довідник з прикладами для Linux, Mac і Windows

Команда `ssh` — основний інструмент для встановлення SSH-з'єднань з командного рядка. Чи входите ви на віддалений Linux-сервер, перенаправляєте порт для доступу до бази даних або переходите через бастіонний хост у внутрішню мережу — команда `ssh` є вашою точкою входу. У цьому посібнику розглядається все: від базового синтаксису до просунутих прапорців, тунелювання та конфігурації ProxyJump — з реальними прикладами для Linux, macOS і Windows.

Перш ніж виконувати SSH-команди, вам знадобляться SSH-ключі. Перегляньте наш посібник <a href="/uk/what-is-an-ssh-key/">що таке SSH-ключ</a> або скористайтеся <a href="/">безкоштовним онлайн-генератором</a> для створення пари ключів.

## Синтаксис команди SSH

Базовий синтаксис команди `ssh`:

```
ssh [параметри] [користувач@]хост [команда]
```

Розберемо по частинах:

- **`[параметри]`** — нуль або більше прапорців, що змінюють поведінку (розглядаються по всьому посібнику)
- **`[користувач@]`** — необов'язкове ім'я користувача. Якщо опущено, SSH використовує поточне локальне ім'я користувача
- **`хост`** — віддалений хост для підключення. Може бути IP-адресою, доменним іменем або псевдонімом, визначеним у `~/.ssh/config`
- **`[команда]`** — необов'язкова команда для виконання на віддаленому хості. Якщо опущена, SSH запускає інтерактивний сеанс оболонки

**Базові приклади:**

```bash
# Підключитися з поточним ім'ям користувача
ssh server.example.com

# Підключитися як конкретний користувач
ssh alice@server.example.com

# Підключитися за IP-адресою
ssh ubuntu@192.168.1.100

# Виконати одну команду і негайно відключитися
ssh user@server.example.com "df -h"

# Підключитися й залишатися в з'єднанні
ssh user@server.example.com
```

## SSH у Linux і macOS

Дистрибутиви Linux і macOS постачаються з передвстановленим **OpenSSH**. У Linux він входить до пакета `openssh-client` (Debian/Ubuntu) або `openssh-clients` (RHEL/CentOS/Fedora). У macOS він вбудований у систему і доступний відразу з Terminal або iTerm2.

### Перевірити наявність SSH

```bash
ssh -V
# Виведення: OpenSSH_9.6p1, LibreSSL 3.3.6
```

### Базове підключення

```bash
# Підключитися до сервера
ssh user@myserver.example.com

# SSH перевіряє ~/.ssh/known_hosts при першому підключенні і запитує:
# The authenticity of host 'myserver.example.com' can't be established.
# ED25519 key fingerprint is SHA256:abc123...
# Are you sure you want to continue connecting (yes/no)? yes

# Відбиток хоста зберігається в ~/.ssh/known_hosts
```

### Використання конфігураційного файлу SSH для зручності

Замість того щоб щоразу вводити повні дані підключення, створіть `~/.ssh/config`:

```
Host myserver
    HostName myserver.example.com
    User alice
    Port 22
    IdentityFile ~/.ssh/id_ed25519

Host dev
    HostName dev.example.com
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_work
```

Тепер `ssh myserver` розгортається в повне з'єднання — жодних прапорців не потрібно.

## SSH у Windows (PowerShell і OpenSSH)

Windows 10 (версія 1809) і Windows 11 включають **OpenSSH-клієнт** як додатковий компонент, роблячи команду `ssh` доступною в PowerShell, Командному рядку та Windows Terminal.

### Перевірити, чи встановлений OpenSSH

```powershell
# У PowerShell
ssh -V
```

Якщо команду не знайдено, встановіть її:

**Через Параметри:**
1. Відкрийте **Параметри → Застосунки → Додаткові можливості**
2. Натисніть **Додати компонент**
3. Знайдіть **Клієнт OpenSSH** і встановіть

**Через PowerShell (від адміністратора):**

```powershell
Add-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0
```

### Використання SSH у PowerShell

Після встановлення синтаксис ідентичний Linux і macOS:

```powershell
# Підключитися до сервера
ssh user@server.example.com

# Підключитися на нестандартному порту
ssh -p 2222 user@server.example.com

# Використовувати конкретний файл ключа
ssh -i C:\Users\YourName\.ssh\id_ed25519 user@server.example.com
```

### Розташування SSH-ключів у Windows

SSH-ключі у Windows зберігаються в `C:\Users\YourUsername\.ssh\`. PowerShell розпізнає `~` як скорочення:

```powershell
# Ці команди еквівалентні
ssh -i ~/.ssh/id_ed25519 user@server
ssh -i C:\Users\YourUsername\.ssh\id_ed25519 user@server
```

### PuTTY і OpenSSH у Windows

До появи OpenSSH у Windows стандартним SSH-клієнтом був **PuTTY**. PuTTY використовує власний формат ключів (файли `.ppk`) і власний генератор ключів (PuTTYgen). Якщо у вас є ключ `.ppk` і потрібно конвертувати його в стандартний формат OpenSSH:

```bash
# Використовуючи інструменти PuTTY
puttygen mykey.ppk -O private-openssh -o id_ed25519
```

Для нових налаштувань надавайте перевагу вбудованому OpenSSH-клієнту у Windows — він підтримується проєктом OpenSSH і використовує той самий формат ключів і конфігураційний файл, що й Linux і macOS.

## SSH з автентифікацією за ключем: прапорець -i

За замовчуванням SSH шукає файли ключів у `~/.ssh/` зі стандартними іменами (`id_ed25519`, `id_rsa`, `id_ecdsa`, `id_dsa`). **Прапорець `-i`** (файл ідентифікатора) дозволяє явно вказати інший файл ключа.

```bash
# Використовувати конкретний файл ключа
ssh -i ~/.ssh/id_ed25519_production user@prod.example.com

# Використовувати ключ, що зберігається в нестандартному розташуванні
ssh -i /path/to/my/custom_key deploy@server.example.com

# Поєднувати з іншими параметрами
ssh -i ~/.ssh/deploy_key -p 2222 -l deploy server.example.com
```

Прапорець `-i` необхідний у кількох ситуаціях:
- У вас кілька пар ключів для різних серверів або цілей
- Ключ не знаходиться в директорії `~/.ssh/` за замовчуванням
- Ви вказуєте ключ у середовищі CI/CD, де шлях задано явно
- Ви завантажили файл `.pem` з AWS або іншого хмарного провайдера

### Приклад AWS EC2

AWS надає файл ключа `.pem` при запуску екземпляра EC2:

```bash
# Підключитися до екземпляра AWS EC2 з ключем .pem
ssh -i ~/Downloads/my-aws-key.pem ec2-user@54.123.456.789

# Виправити права за необхідності (обов'язково — SSH відхиляє ключі з відкритими правами)
chmod 400 ~/Downloads/my-aws-key.pem
```

## Основні параметри та прапорці команди SSH

Команда `ssh` має десятки параметрів. Ось ті, які вам дійсно знадобляться.

| Прапорець | Опис | Приклад |
|---|---|---|
| `-p ПОРТ` | Підключитися на нестандартному порту | `ssh -p 2222 user@host` |
| `-i ФАЙЛ` | Використовувати конкретний файл ідентифікатора (ключа) | `ssh -i ~/.ssh/mykey user@host` |
| `-l КОРИСТУВАЧ` | Вказати ім'я користувача для входу (альтернатива `user@`) | `ssh -l alice host` |
| `-L локальний:хост:віддалений` | Локальне перенаправлення портів | `ssh -L 8080:localhost:80 host` |
| `-R віддалений:хост:локальний` | Віддалене перенаправлення портів | `ssh -R 9090:localhost:3000 host` |
| `-D порт` | Динамічний (SOCKS) проксі | `ssh -D 1080 host` |
| `-N` | Не виконувати віддалену команду (тільки тунелювання) | `ssh -N -L 3307:db:3306 host` |
| `-f` | Перейти у фоновий режим після автентифікації | `ssh -f -N -D 1080 host` |
| `-v` | Детальний (налагоджувальний) виведення | `ssh -v user@host` |
| `-vv` або `-vvv` | Ще детальніший виведення | `ssh -vvv user@host` |
| `-A` | Увімкнути перенаправлення SSH-агента | `ssh -A user@jump-host` |
| `-X` | Увімкнути перенаправлення X11 | `ssh -X user@host xclock` |
| `-C` | Увімкнути стиснення | `ssh -C user@host` |
| `-T` | Вимкнути виділення псевдотермінала | `ssh -T git@github.com` |
| `-q` | Тихий режим (придушити попередження) | `ssh -q user@host` |
| `-o ПАРАМЕТР=ЗНАЧЕННЯ` | Задати параметр конфігурації inline | `ssh -o StrictHostKeyChecking=no host` |
| `-J СТРИБОК` | ProxyJump через бастіонний хост | `ssh -J jump.host user@internal` |
| `-W хост:порт` | Перенаправити stdio на хост:порт (для ProxyJump) | Використовується всередині `-J` |

### Детальний режим для налагодження

Коли SSH не працює, `-v` (або `-vv` / `-vvv`) показує, що саме відбувається:

```bash
ssh -v user@server.example.com
# Показує: які ключі перебираються, які методи автентифікації приймає сервер,
# чи задіяний агент і де відбувається збій підключення

ssh -vvv user@server.example.com
# Максимальна деталізація — показує детальне криптографічне узгодження
```

### Вимкнення перевірки хост-ключа (використовуйте обережно)

У середовищах автоматизації та тестування, де перевірка хост-ключа незручна:

```bash
ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null user@host
```

**Не використовуйте це в продакшні.** Це вимикає захист від атак «людина посередині». Натомість попередньо заповнюйте `known_hosts` за допомогою `ssh-keyscan` у скриптах підготовки.

### Перенаправлення агента

```bash
# Перенаправити локальний SSH-агент на віддалений сервер
ssh -A user@jump-server

# З jump-сервера тепер можна SSH-підключатися до внутрішніх хостів з локальними ключами
# Не потрібно копіювати приватні ключі на jump-сервер
ssh internal-server.private
```

Перенаправлення агента корисне, але має наслідки для безпеки: будь-хто, хто має root-доступ на проміжному сервері, може використовувати перенаправлений агент. Використовуйте його тільки з довіреними серверами і надавайте перевагу **ProxyJump** (він не розкриває агент проміжним хостам).

## Команди SSH-тунелювання

SSH-тунелювання маршрутизує TCP-з'єднання через зашифрований SSH-канал. Загальну інформацію про тунелювання — у <a href="/uk/what-is-ssh/">розділі про тунелювання в нашому SSH-посібнику</a>.

### Локальне перенаправлення портів

Робить віддалений сервіс доступним на вашій локальній машині.

```bash
# Отримати доступ до віддаленої бази даних MySQL на локальному порту 3307
ssh -L 3307:localhost:3306 user@db-server.example.com -N

# Підключитися до віддаленого вебсервера, доступного тільки з SSH-хоста
ssh -L 8080:intranet.company.com:80 user@vpn.company.com -N

# Отримати доступ до екземпляра Redis у приватній мережі
ssh -L 6380:10.0.1.50:6379 user@bastion.example.com -N
```

Після виконання будь-якої з наведених команд підключайте локальний застосунок до `127.0.0.1:ЛОКАЛЬНИЙ_ПОРТ`.

### Віддалене перенаправлення портів

Відкриває локальний сервіс на віддаленому сервері.

```bash
# Відкрити локальний сервер розробки (порт 3000) на порту 9090 віддаленого сервера
ssh -R 9090:localhost:3000 user@remote.example.com -N

# Дозволити будь-якому інтерфейсу на віддаленому сервері підключатися (потребує GatewayPorts yes у sshd_config)
ssh -R 0.0.0.0:9090:localhost:3000 user@remote.example.com -N
```

### Динамічне перенаправлення портів (SOCKS-проксі)

```bash
# Створити SOCKS5-проксі на локальному порту 1080, що працює у фоні
ssh -D 1080 -N -f user@remote.example.com

# Тепер налаштуйте браузер або систему на використання SOCKS5-проксі: 127.0.0.1:1080
# Весь трафік маршрутизується через віддалений сервер
```

Для використання з curl:

```bash
curl --socks5 127.0.0.1:1080 https://api.example.com/
```

### Запуск тунелю у фоновому режимі

```bash
# -f відправляє SSH у фоновий режим, -N означає без віддаленої команди
ssh -f -N -L 3307:localhost:3306 user@db-server

# Знайти фоновий SSH-процес
ps aux | grep ssh

# Завершити його
kill PID
```

## SSH ProxyJump і бастіонні хости

У багатьох організаціях продакшн-сервери знаходяться в приватних мережах, недоступних безпосередньо з інтернету. **Бастіонний хост** (або jump-хост) — це захищений сервер на периметрі мережі, до якого ви спочатку підключаєтеся по SSH, а потім переходите на внутрішні сервери. Прапорець `ssh -J` (ProxyJump, доданий в OpenSSH 7.3) робить це безшовним.

### Базовий ProxyJump

```bash
# Перейти через один бастіонний хост для доступу до внутрішнього сервера
ssh -J ec2-user@bastion.example.com ubuntu@10.0.1.50

# Скорочена форма
ssh -J user@jump-host user@internal-server
```

Це створює два SSH-з'єднання: спочатку до бастіона, потім з бастіона до внутрішнього сервера — все в одній команді, з наскрізним шифруванням. Ваш локальний SSH-агент обробляє автентифікацію на обох хостах.

### Ланцюжок із кількох переходів

```bash
# Перейти через два проміжні хости
ssh -J user@jump1.example.com,user@jump2.example.com user@final-destination
```

### ProxyJump у файлі конфігурації SSH

Для часто використовуваних серверів внесіть конфігурацію ProxyJump до `~/.ssh/config`:

```
Host bastion
    HostName bastion.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host prod-web
    HostName 10.0.1.100
    User ubuntu
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519_aws

Host prod-db
    HostName 10.0.2.50
    User postgres
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519_aws
```

Тепер `ssh prod-web` автоматично підключається через бастіон. Ручні прапорці `-J` не потрібні.

### ProxyJump проти перенаправлення агента

ProxyJump кращий за перенаправлення агента (`-A`), тому що:
- Приватний ключ ніколи не передається проміжному хосту
- Бастіонний хост не може використовувати ваші облікові дані для автентифікації деінде
- З'єднання автентифіковане наскрізним чином — сервер призначення безпосередньо перевіряє вас

Детальна інформація про налаштування — у нашому повному посібнику <a href="/uk/how-to-set-up-ssh/">як налаштувати SSH</a>.

## Часті запитання

### Як виконати SSH-команду без пароля?

Налаштуйте автентифікацію за SSH-ключем. Згенеруйте пару ключів (`ssh-keygen -t ed25519`), скопіюйте відкритий ключ до файлу `~/.ssh/authorized_keys` на сервері за допомогою `ssh-copy-id user@host`, і наступні підключення використовуватимуть ключ без запиту пароля. Якщо ключ захищений парольною фразою, використовуйте `ssh-agent` і `ssh-add` для її кешування — вводити доведеться тільки один раз за сеанс.

### Як завершити SSH-сеанс?

Введіть `exit` або натисніть **Ctrl+D**, щоб закрити віддалену оболонку та відключитися. Якщо сеанс завис (наприклад, мережа перервалася), використовуйте escape-послідовність SSH: натисніть **Enter**, потім введіть `~.` (тильда і крапка). SSH негайно завершить з'єднання. Інші escape-послідовності: `~^Z` переводить SSH у фоновий режим, `~?` показує всі escape-послідовності.

### Як виконати команду на віддаленому сервері через SSH?

Додайте команду після імені хоста:

```bash
ssh user@server "command to run"
ssh user@server "ls -la /var/log/"
ssh user@server "sudo systemctl status nginx"
```

Для багаторядкових команд або скриптів використовуйте heredoc:

```bash
ssh user@server << 'EOF'
cd /var/www/html
git pull origin main
systemctl reload nginx
EOF
```

Для інтерактивних команд (що потребують термінала) додайте прапорець `-t` для виділення псевдотермінала: `ssh -t user@server sudo apt upgrade`.

### У чому різниця між ssh і scp?

`ssh` відкриває сеанс термінала (або виконує одну команду) на віддаленій машині. `scp` (Secure Copy Protocol) копіює файли між вашою локальною машиною та віддаленою, або між двома віддаленими машинами, використовуючи SSH для шифрування та автентифікації. Обидва використовують однаковий протокол SSH та інфраструктуру ключів. Для передачі файлів `scp` — спеціалізований інструмент:

```bash
# scp з локального на віддалений
scp file.txt user@server:/destination/path/

# scp з віддаленого на локальний
scp user@server:/remote/file.txt ./local-copy.txt
```

Для більш просунутої передачі файлів (рекурсивної, відновлюваної, з навігацією по директоріях) використовуйте `sftp`. Для великих передач `rsync -avz -e ssh` ще ефективніший.

### Як використовувати SSH з файлом конфігурації?

Створіть `~/.ssh/config` і додайте записи хостів. Кожен блок `Host` визначає псевдонім підключення з повними налаштуваннями:

```
Host alias-name
    HostName actual.server.com
    User username
    Port 22
    IdentityFile ~/.ssh/specific_key
```

Тоді `ssh alias-name` автоматично використовує всі задані налаштування. Файл конфігурації підтримує wildcards: `Host *.example.com` застосовує налаштування до всіх відповідних хостів. `Host *` у кінці застосовує налаштування за замовчуванням до всіх з'єднань. Параметри в конкретному блоці `Host` мають пріоритет над глобальними значеннями за замовчуванням.

## Потрібні SSH-ключі? Створіть їх безкоштовно

Для використання автентифікації за SSH-ключем — яка усуває необхідність у паролях — вам потрібна пара ключів. Створіть ключі Ed25519 або RSA миттєво в браузері, нічого не надсилаючи на наші сервери.
