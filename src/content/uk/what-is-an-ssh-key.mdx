# Що таке SSH-ключ? Відкриті ключі, приватні ключі та пари ключів

**SSH-ключ** — це криптографічний обліковий запис для автентифікації на SSH-серверах без пароля. SSH-ключі постачаються парами: **відкритий ключ** і **приватний ключ**, — і разом вони роблять віддалений доступ безпечнішим і зручнішим, ніж традиційна автентифікація за паролем. У цьому посібнику пояснюється, що таке SSH-ключ, як працює система пар ключів, які типи ключів обирати і як використовувати SSH-ключі з такими сервісами, як GitHub і GitLab.

## Як працюють SSH-ключі

SSH-ключі засновані на **асиметричній криптографії**, також відомій як криптографія з відкритим ключем. На відміну від симетричної криптографії (де один секретний ключ і шифрує, і розшифровує дані), асиметричні системи використовують математично пов'язані пари ключів, де дані, зашифровані одним ключем, можна розшифрувати тільки іншим.

Ось як виглядає процес автентифікації при вході на сервер за допомогою SSH-ключа:

1. Ви виконуєте `ssh user@server.example.com`.
2. Сервер шукає ваш відкритий ключ у `~/.ssh/authorized_keys`.
3. Сервер генерує випадкове завдання і шифрує його вашим відкритим ключем.
4. Ваш SSH-клієнт розшифровує завдання за допомогою приватного ключа.
5. Клієнт надсилає серверу криптографічне доказування (отримане з розшифрованого завдання та ідентифікатора сесії).
6. Сервер перевіряє доказування. Тільки власник приватного ключа міг його створити.
7. Автентифікація успішна. Жоден пароль не залишає вашу машину.

Ключовий момент: **приватний ключ ніколи не передається мережею**. Сервер бачить лише ваш відкритий ключ і криптографічні доказування, які ви створюєте. Навіть якщо хтось перехопить кожен пакет вашого SSH-сеансу, він не отримає нічого, що допоможе йому видати себе за вас.

Загальну інформацію про протокол SSH — у нашому посібнику <a href="/uk/what-is-ssh/">що таке SSH</a>.

## Відкритий ключ і приватний ключ: у чому різниця?

Розуміння відмінності між двома ключами необхідне для правильного використання SSH.

### Відкритий ключ — ваш замок

**Відкритий ключ** призначений для вільного поширення. Думайте про нього як про навісний замок: ви роздаєте копії всім, чиїми системами хочете користуватися для входу. Коли сервер зберігає ваш відкритий ключ у файлі `authorized_keys`, він може перевірити вашу особу, не знаючи приватного ключа.

Відкриті ключі:
- Безпечно передавати — їх не можна використати для вашої імперсонації
- Розміщуються на серверах у `~/.ssh/authorized_keys`
- Завантажуються до GitHub, GitLab та інших сервісів
- Зазвичай мають розширення `.pub` (наприклад, `id_ed25519.pub`)
- Виглядають так: `ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... user@hostname`

### Приватний ключ — ваш ключ

**Приватний ключ** — секретна половина пари. Він ніколи не повинен залишати вашу машину і не повинен передаватися нікому — ні адміністраторам, ні хмарним провайдерам, ні співробітникам служби підтримки. Якщо ваш приватний ключ скомпрометований, будь-хто, хто його отримає, може видати себе за вас на кожному сервері, де встановлений ваш відкритий ключ.

Приватні ключі:
- Повинні зберігатися в таємниці й безпеці
- Зберігаються локально в `~/.ssh/` з обмеженими правами доступу (600)
- Необов'язково шифруються парольною фразою
- Не мають розширення або мають специфічне розширення (наприклад, `id_ed25519`)

### Файл `authorized_keys`

На боці сервера відкриті ключі кожного користувача зберігаються в `~/.ssh/authorized_keys`. Це текстовий файл, де кожен рядок містить один відкритий ключ. При спробі автентифікації SSH-клієнта сервер перевіряє, чи може клієнт довести володіння приватним ключем, що відповідає будь-якому запису в цьому файлі.

```
# Формат ~/.ssh/authorized_keys
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGq... alice@laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHr... alice@workstation
ssh-rsa AAAAB3NzaC1yc2EAAAA... backup-deploy-key
```

У цьому файлі можна зберігати стільки відкритих ключів, скільки потрібно, дозволяючи кільком пристроям або системам автоматизації автентифікуватися на одному сервері.

## Алгоритми SSH-ключів: Ed25519 проти RSA

При генерації пари SSH-ключів ви обираєте алгоритм. Два найпоширеніші варіанти сьогодні — **Ed25519** і **RSA**. Існують також ECDSA і DSA, але DSA застарів, а ECDSA має підводні камені в реалізації, тому зосередимося на двох рекомендованих варіантах.

### Ed25519 — сучасний вибір

Ed25519 був запроваджений в OpenSSH у версії 6.5 (2014) і зараз є рекомендованим стандартом. Він використовує еліптичну криптографію на скрученій кривій Едвардса (Curve25519), розробленій Деніелом Дж. Бернштейном.

**Переваги Ed25519:**
- **Компактність** — ключі займають лише 68 символів (відкритий ключ), файли мізерного розміру
- **Швидкість** — генерація та перевірка підпису надзвичайно швидкі
- **Безпечна конструкція** — не вразливий до атак по сторонніх каналах, що впливають на деякі реалізації RSA
- **Фіксований розмір ключа** — рівень безпеки 256 біт, не потрібно обирати розмір
- **Детермінованість** — підписи детерміновані, що усуває цілий клас збоїв генератора випадкових чисел

```bash
# Згенерувати пару ключів Ed25519
ssh-keygen -t ed25519 -C "your@email.com"
```

### RSA — універсальний варіант

RSA — старіший алгоритм (винайдений у 1977 році), і хоча він досі безпечний при 4096 бітах, він повільніший і виробляє більші ключі, ніж Ed25519. Однак RSA має значно ширшу сумісність — працює зі старими SSH-серверами, застарілими системами та деякими апаратними ключами безпеки.

**Коли використовувати RSA:**
- Потрібно підключитися до застарілих систем зі старими версіями OpenSSH (до 6.5)
- Використовується смарт-картка або HSM без підтримки Ed25519
- Потрібна сумісність зі старими інструментами

```bash
# Згенерувати пару ключів RSA 4096 біт
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

**Ніколи не використовуйте RSA менш ніж з 4096 бітами.** RSA-1024 і RSA-2048 дедалі більше вважаються недостатніми по мірі зростання обчислювальних потужностей. Ed25519 з 256 бітами забезпечує еквівалентну або кращу безпеку при частці розміру ключа.

### Порівняльна таблиця

| Характеристика | Ed25519 | RSA 4096 |
|---|---|---|
| Розмір ключа (відкритий) | ~68 символів | ~725 символів |
| Рівень безпеки | 128-біт (постквантово-стійка структура) | ~140-біт класичний |
| Швидкість | Дуже висока | Повільніше |
| Сумісність | Сучасний SSH (OpenSSH 6.5+) | Універсальна |
| Рекомендується для | Всіх нових ключів | Тільки застарілих систем |

**Висновок: генеруйте ключі Ed25519 для будь-якого сучасного застосування.** RSA використовуйте тільки за примусових вимог сумісності.

## Як додати SSH-ключ до GitHub

Додавання вашого відкритого SSH-ключа до GitHub дозволяє виконувати push і pull по SSH без введення імені користувача та пароля при кожній операції. Ось покроковий процес.

### Крок 1: Згенеруйте пару ключів

Якщо у вас ще немає пари SSH-ключів, згенеруйте її. Ви можете скористатися нашим <a href="/">безкоштовним онлайн-генератором SSH-ключів</a> або виконати:

```bash
ssh-keygen -t ed25519 -C "your-github-email@example.com"
```

Прийміть розташування файлу за замовчуванням (`~/.ssh/id_ed25519`) і за бажанням задайте парольну фразу.

### Крок 2: Скопіюйте відкритий ключ

Відобразіть відкритий ключ і скопіюйте його в буфер обміну:

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows (PowerShell)
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

Або просто відобразіть і скопіюйте вручну:

```bash
cat ~/.ssh/id_ed25519.pub
```

### Крок 3: Додайте ключ до GitHub

1. Увійдіть до **GitHub.com**
2. Натисніть на аватар → **Settings**
3. На лівій панелі натисніть **SSH and GPG keys**
4. Натисніть **New SSH key**
5. Дайте йому зрозумілу назву (наприклад, «MacBook Pro 2024» або «Робочий ноутбук»)
6. Вставте відкритий ключ у поле **Key**
7. Натисніть **Add SSH key**

### Крок 4: Перевірте з'єднання

```bash
ssh -T git@github.com
```

Ви маєте побачити: `Hi username! You've successfully authenticated, but GitHub does not provide shell access.`

### Перемикання наявного репозиторію на SSH

Якщо ви клонували репозиторій по HTTPS і хочете перейти на SSH:

```bash
# Перевірити поточну URL-адресу віддаленого репозиторію
git remote -v

# Оновити на SSH
git remote set-url origin git@github.com:username/repository.git
```

Для GitLab процес ідентичний — перейдіть до **Preferences → SSH Keys** і виконайте ті самі кроки.

## Де зберігаються SSH-ключі?

SSH-ключі зберігаються в директорії `.ssh` у домашній теці користувача. Розташування залежить від операційної системи.

### Linux і macOS

```
~/.ssh/
├── id_ed25519          # Ваш приватний ключ (права: 600)
├── id_ed25519.pub      # Ваш відкритий ключ (права: 644)
├── id_rsa              # RSA приватний ключ (якщо є)
├── id_rsa.pub          # RSA відкритий ключ
├── authorized_keys     # Відкриті ключі для входу на цю машину
├── known_hosts         # Відбитки серверів, до яких ви підключалися
└── config              # Файл конфігурації клієнта
```

Сама директорія `~/.ssh/` повинна мати права **700** (тільки власник: читання/запис/виконання). Файли приватних ключів повинні мати права **600** (тільки власник: читання/запис). SSH відмовиться використовувати ключі з надмірно відкритими правами.

```bash
# Виправити права за необхідності
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

### Windows

У Windows 10 і 11 зі встановленим OpenSSH SSH-ключі зберігаються в:

```
C:\Users\YourUsername\.ssh\
├── id_ed25519
├── id_ed25519.pub
├── authorized_keys
├── known_hosts
└── config
```

У Windows директорія `.ssh` доступна з Провідника файлів (може знадобитися увімкнути «Показувати приховані файли») або з PowerShell за допомогою `~\.ssh\`.

### Кілька файлів ключів

Якщо у вас є ключі для різних цілей (особистий GitHub, робочий GitLab, продакшн-сервери), використовуйте файл `~/.ssh/config` для зіставлення кожного хоста з його ключем:

```
Host github.com
    IdentityFile ~/.ssh/id_ed25519_github

Host gitlab.work.com
    IdentityFile ~/.ssh/id_ed25519_work

Host prod-server
    HostName 203.0.113.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_production
```

## Парольні фрази для SSH-ключів

При генерації пари SSH-ключів `ssh-keygen` запитує, чи хочете ви задати **парольну фразу**. Це необов'язковий пароль, який шифрує файл приватного ключа на диску.

### Навіщо використовувати парольну фразу?

Якщо файл приватного ключа захищений надійною парольною фразою, крадіжки файлу недостатньо для вашої імперсонації — зловмиснику також потрібно буде зламати парольну фразу. Це створює критично важливий другий рівень захисту, особливо для ноутбуків, які можуть бути фізично викрадені.

Без парольної фрази приватний ключ зберігається у відкритому вигляді на диску. Будь-хто, хто може прочитати файл (через шкідливе ПЗ, фізичний доступ або неналежним чином захищену резервну копію), може негайно його використати.

### Взаємодія парольних фраз із ssh-agent

Недолік парольної фрази — необхідність вводити її кожного разу при використанні ключа. **ssh-agent** вирішує цю проблему, зберігаючи розшифровані ключі в пам'яті протягом сеансу.

```bash
# Запустити ssh-agent і додати ключ
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
# Введіть парольну фразу один раз — вона кешується на час сеансу
```

У macOS системний Keychain інтегрований із ssh-agent. Додайте `--apple-use-keychain` для збереження між перезавантаженнями:

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

### Зміна парольної фрази

Ви можете змінити парольну фразу наявного ключа без перегенерації пари:

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

Буде запитана стара парольна фраза, потім запропоновано задати нову. Відкритий ключ при цьому не змінюється, тому оновлювати його на серверах не потрібно.

## Відбитки SSH-ключів

Кожна пара SSH-ключів має **відбиток** — короткий хеш відкритого ключа, що унікально його ідентифікує. Відбитки слугують кільком цілям:

- **Перевірка автентичності хоста** — при першому підключенні до сервера SSH показує відбиток його хост-ключа. Якщо ви підтверджуєте відповідність очікуваному (наприклад, наданому вашим хостинг-провайдером), значить, ви підключаєтеся до потрібного сервера.
- **Ідентифікація ключів** — при керуванні багатьма ключами відбитки допомагають визначити, який фізичний ключ відповідає якому запису в `authorized_keys` або на такому сервісі, як GitHub.
- **Аудит і відкликання** — у журналах аудиту часто записуються відбитки, а не повні ключі. Знання відбитка ключа допомагає знайти й видалити його за необхідності.

### Перегляд відбитка ключа

```bash
# Переглянути відбиток відкритого ключа
ssh-keygen -lf ~/.ssh/id_ed25519.pub
# Виведення: 256 SHA256:abcdef1234567890... your@email.com (ED25519)

# Переглянути відбиток хост-ключа сервера
ssh-keyscan server.example.com | ssh-keygen -lf -
```

GitHub відображає відбитки всіх доданих SSH-ключів у **Settings → SSH and GPG keys**, дозволяючи перевірити, що видимий там ключ збігається з ключем на вашій машині.

### Файл `known_hosts`

При першому підключенні до нового SSH-сервера ви бачите повідомлення:

```
The authenticity of host 'example.com (203.0.113.10)' can't be established.
ED25519 key fingerprint is SHA256:abcdef1234567890...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Введіть `yes`, і відбиток сервера буде додано до `~/.ssh/known_hosts`. При наступних підключеннях SSH мовчки перевіряє, чи збігається ключ сервера зі збереженим відбитком. Якщо ні — що може вказувати на неправильну конфігурацію або атаку «людина посередині» — SSH голосно попередить вас і відмовиться підключатися.

## Часті запитання

### Для чого використовується SSH-ключ?

SSH-ключі автентифікують вас на віддалених серверах без пароля. Найпоширеніші випадки використання: вхід на сервери Linux/Unix для адміністрування, автентифікація в сервісах хостингу Git (GitHub, GitLab, Bitbucket) для push/pull-операцій, скрипти автоматичного розгортання, що потребують беспарольного доступу до серверів, і CI/CD-пайплайни, що деплоять код у продакшн. SSH-ключі кращі за паролі, тому що вони захищені від брутфорсу, не використовуються повторно при витоках і можуть застосовуватися без участі людини.

### Скільки SSH-ключів мені потрібно?

Єдиної правильної відповіді немає, але поширена найкраща практика — **один ключ на пристрій** (не на сервіс). На вашому ноутбуці — одна пара ключів, на десктопі — інша, на CI-сервері — ще одна. Тоді при крадіжці або компрометації пристрою ви відкликаєте тільки ключ цього пристрою. Дехто додатково розділяє за контекстом (особистий vs робочий), отримуючи два ключі на пристрій. Уникайте використання одного ключа на кількох машинах — при його компрометації весь ваш доступ опиниться під загрозою одночасно.

### Чи можна передавати відкритий SSH-ключ?

Так. Відкритий ключ призначений для передачі. Його можна надіслати електронною поштою, опублікувати в документації GitHub або включити до заявки на отримання доступу до сервера. Один лише відкритий ключ не може бути використаний для автентифікації від вашого імені — для цього потрібен відповідний приватний ключ, який є тільки у вас. Це фундаментальна властивість асиметричної криптографії. Ніколи нікому не передавайте **приватний** ключ.

### Що станеться, якщо я втрачу приватний SSH-ключ?

Якщо ви втратите приватний ключ і не маєте резервної копії, ви втратите можливість автентифікуватися з його допомогою. Ваш відкритий ключ, досі встановлений на серверах і сервісах, стане «сиротою». Вам знадобиться альтернативний спосіб доступу до сервера (автентифікація за паролем, якщо вона увімкнена, консольний доступ, екстрений доступ хостинг-провайдера) для видалення старого відкритого ключа з `authorized_keys` і додавання нового. Саме тому важливо робити резервну копію приватного ключа (у менеджері паролів або зашифрованій резервній копії). Приватний ключ не може бути відновлений з відкритого.

### Що краще: Ed25519 чи RSA?

**Ed25519 кращий для всіх нових ключів.** Він швидший, виробляє менші ключі, не вразливий до певних слабкостей реалізації, що впливають на RSA, і забезпечує еквівалентну або вищу безпеку з 256-бітним ключем порівняно з RSA-4096. Єдина причина використовувати RSA сьогодні — сумісність із дуже старим SSH-серверним ПЗ (OpenSSH до версії 6.5, випущеної у 2014 році) або застарілими апаратними модулями безпеки. Якщо ви підключаєтеся до будь-якого сучасного сервера або сервісу на кшталт GitHub — використовуйте Ed25519.
