# Як налаштувати SSH: повний покроковий посібник для Linux, Mac і Windows

Правильне налаштування SSH займає близько десяти хвилин і окупається роками. Після налаштування ви зможете входити на сервери без введення паролів, надсилати код до GitHub без введення облікових даних при кожній операції та автоматизувати розгортання з упевненістю. Цей посібник проведе вас через весь процес — від генерації пари ключів до усунення поширених помилок — для Linux, macOS і Windows.

Якщо ви спочатку хочете розібратися в технологіях, що лежать в основі, прочитайте наші посібники про те, <a href="/uk/what-is-ssh">що таке SSH</a> і <a href="/uk/what-is-an-ssh-key">що таке SSH-ключі</a>.

## Передумови

Перш ніж починати, переконайтеся, що у вас є:

**Для доступу до сервера:**
- Термінальний застосунок (Terminal у macOS/Linux, Windows Terminal або PowerShell у Windows)
- Віддалений сервер на Linux/Unix зі встановленим програмним забезпеченням SSH-сервера (sshd)
- Обліковий запис користувача на цьому сервері (або root-доступ для його створення)
- IP-адреса або ім'я хоста сервера

**Для доступу до GitHub/GitLab:**
- Акаунт на GitHub або GitLab
- Термінал з доступною командою `ssh`
- Необов'язково: git, встановлений локально

**Вимоги до операційної системи:**
- **Linux**: SSH-клієнт передвстановлений практично у всіх дистрибутивах
- **macOS**: SSH-клієнт вбудований, доступний з Terminal
- **Windows 10 версії 1809+ / Windows 11**: OpenSSH-клієнт доступний як додатковий компонент

Встановлювати нічого не потрібно ні в Linux, ні в macOS. У Windows перевірте наявність OpenSSH, відкривши PowerShell і виконавши `ssh -V`. Якщо отримаєте рядок версії — готово. Якщо ні — дивіться Крок 4 з інструкцією по встановленню.

## Крок 1: Згенеруйте пару SSH-ключів

Вам потрібна пара ключів, перш ніж ви зможете щось зробити. Пара ключів складається з **приватного ключа** (залишається на вашій машині) і **відкритого ключа** (передається на сервери або сервіси типу GitHub).

### Варіант А: Використайте онлайн-генератор

Якщо надаєте перевагу графічному інтерфейсу, наш <a href="/">безкоштовний онлайн-генератор SSH-ключів</a> створює пари ключів Ed25519 або RSA в браузері. Приватний ключ генерується локально за допомогою Web Crypto API — нічого не надсилається на наші сервери. Завантажте обидва файли і збережіть їх у `~/.ssh/` (Linux/macOS) або `C:\Users\YourName\.ssh\` (Windows).

### Варіант Б: Використайте ssh-keygen у командному рядку

Команда `ssh-keygen` доступна в Linux, macOS і Windows з OpenSSH:

```bash
# Згенерувати пару ключів Ed25519 (рекомендується)
ssh-keygen -t ed25519 -C "your@email.com"
```

**Що означає кожен прапорець:**
- `-t ed25519` — тип ключа (Ed25519 сучасний, швидкий і безпечний)
- `-C "коментар"` — мітка, вбудована у відкритий ключ, зазвичай ваша email-адреса

Вам буде запропоновано:

```
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/user/.ssh/id_ed25519):
```

Натисніть **Enter** для прийняття розташування за замовчуванням. За наявності кількох ключів можна дати їм описові імена (наприклад, `/home/user/.ssh/id_ed25519_github`).

```
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
```

Введіть надійну парольну фразу. Вона шифрує файл приватного ключа на диску — при крадіжці машини ключ не можна буде використати без парольної фрази. Можна залишити порожньою для сценаріїв автоматизації, де участь людини неможлива, але для персональних ключів завжди використовуйте парольну фразу.

**Виведення:**

```
Your identification has been saved in /home/user/.ssh/id_ed25519
Your public key has been saved in /home/user/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:abc123defghijk... your@email.com
The key's randomart image is:
+--[ED25519 256]--+
...
```

Створюються два файли:
- `~/.ssh/id_ed25519` — ваш **приватний ключ** (ніколи не передавайте)
- `~/.ssh/id_ed25519.pub` — ваш **відкритий ключ** (можна передавати)

### Для RSA (сумісність із застарілими системами)

Якщо потрібно підключитися до дуже старих систем без підтримки Ed25519:

```bash
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

Використовуйте щонайменше 4096 біт. RSA-ключі менше 4096 біт більше не вважаються достатньо захищеними.

## Крок 2: Налаштування SSH у Linux

У цьому розділі описано копіювання відкритого ключа на Linux-сервер для автентифікації без пароля.

### Метод 1: ssh-copy-id (найпростіший)

`ssh-copy-id` — найпростіший спосіб встановити відкритий ключ на віддалений сервер. Він підключається з використанням автентифікації за паролем, додає відкритий ключ до файлу `~/.ssh/authorized_keys` сервера і встановлює правильні права доступу.

```bash
ssh-copy-id user@server.example.com

# Явно вказати файл ключа
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server.example.com

# Підключитися на нестандартному порту
ssh-copy-id -p 2222 user@server.example.com
```

Вас попросять ввести пароль сервера востаннє. Після цього ви зможете підключатися за допомогою `ssh user@server.example.com` використовуючи ключ.

### Метод 2: Ручне копіювання

Якщо `ssh-copy-id` недоступний (поширено в деяких BSD-системах або мінімалістичних встановленнях Linux):

```bash
# Відобразити відкритий ключ і скопіювати виведення
cat ~/.ssh/id_ed25519.pub

# Потім підключитися до сервера (з паролем) і виконати:
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "ВСТАВТЕ_ВІДКРИТИЙ_КЛЮЧ_ТУТ" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

Або все в одному конвеєрі:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com \
  "mkdir -p ~/.ssh && chmod 700 ~/.ssh && \
   cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

### Перевірка встановлення ключа

```bash
ssh user@server.example.com
# Має підключитися без запиту пароля (тільки парольна фраза, якщо ви її задали)
```

### Налаштування SSH-сервера (sshd)

На віддаленому сервері конфігурація SSH-демона знаходиться в `/etc/ssh/sshd_config`. Для безпечного налаштування:

```bash
sudo nano /etc/ssh/sshd_config
```

Рекомендовані налаштування:

```
# Увімкнути автентифікацію за відкритим ключем
PubkeyAuthentication yes

# Вимкнути автентифікацію за паролем (після підтвердження роботи ключів!)
 PasswordAuthentication no

# Вимкнути вхід root по SSH
 PermitRootLogin no

# Дозволити тільки певних користувачів
AllowUsers alice bob deploy

# Підтримувати сеанси активними
ClientAliveInterval 60
ClientAliveCountMax 3
```

Після редагування перезавантажте SSH-демон:

```bash
sudo systemctl reload sshd
# Або в старих системах:
sudo service ssh reload
```

**Важливо**: Не вимикайте `PasswordAuthentication`, поки не переконаєтеся в роботі автентифікації за ключем. Якщо ви заблокуєте себе, знадобиться консольний доступ (або екстрений доступ хостинг-провайдера) для відновлення.

### Налаштування ssh-agent у Linux

Якщо ключ захищений парольною фразою, додайте його до `ssh-agent` на початку сеансу, щоб не вводити її повторно:

```bash
# Запустити агент (часто автоматично запускається середовищами робочого стола)
eval "$(ssh-agent -s)"

# Додати ключ (введіть парольну фразу один раз)
ssh-add ~/.ssh/id_ed25519

# Перевірити завантажені ключі
ssh-add -l
```

Для автоматичного запуску `ssh-agent` і додавання ключа при вході додайте до `~/.bashrc` або `~/.zshrc`:

```bash
# Автозапуск ssh-agent
if [ -z "$SSH_AUTH_SOCK" ]; then
  eval "$(ssh-agent -s)"
  ssh-add ~/.ssh/id_ed25519
fi
```

## Крок 3: Налаштування SSH у macOS

macOS включає повний OpenSSH-клієнт. Процес налаштування автентифікації за ключем на віддаленому сервері ідентичний Linux (див. Крок 2 вище). Специфіка macOS стосується інтеграції з Keychain.

### Інтеграція з Keychain

macOS може зберегти вашу парольну фразу в системному ланцюжку ключів, щоб після першого введення більше не запитувати її:

```bash
# Додати ключ до ssh-agent з інтеграцією Keychain
ssh-add --apple-use-keychain ~/.ssh/id_ed25519

# Для старих версій macOS (до Monterey)
ssh-add -K ~/.ssh/id_ed25519
```

Щоб macOS завжди завантажувала ключі з Keychain, додайте до `~/.ssh/config`:

```
Host *
    AddKeysToAgent yes
    UseKeychain yes
    IdentityFile ~/.ssh/id_ed25519
```

Після цього парольна фраза зберігається в Keychain і завантажується автоматично — при наступних завантаженнях запитуватися не буде.

### Перевірити версію SSH-клієнта

```bash
ssh -V
# OpenSSH_9.x, LibreSSL x.x.x
```

macOS постачається з достатньо свіжою версією OpenSSH, але не завжди з найновішою. Якщо потрібні нові функції, встановіть через Homebrew: `brew install openssh`.

## Крок 4: Налаштування SSH у Windows

Windows 10 (1809+) і Windows 11 включають OpenSSH. Ось як змусити його працювати.

### Встановити OpenSSH-клієнт

**Перевірити, чи встановлений:**

```powershell
ssh -V
```

**Встановити через Параметри:**
1. Відкрийте **Параметри → Застосунки → Додаткові можливості**
2. Знайдіть **Клієнт OpenSSH**
3. Натисніть **Встановити**

**Встановити через PowerShell (від адміністратора):**

```powershell
Add-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0
```

### Згенерувати ключі у Windows

Відкрийте PowerShell і виконайте:

```powershell
ssh-keygen -t ed25519 -C "your@email.com"
```

Ключі зберігаються в `C:\Users\YourUsername\.ssh\`.

### Скопіювати відкритий ключ на Linux-сервер

Windows не включає `ssh-copy-id`, але це можна зробити вручну з PowerShell:

```powershell
# Відобразити відкритий ключ
Get-Content ~/.ssh/id_ed25519.pub

# Скопіювати на віддалений сервер (буде запитаний пароль сервера)
$pubkey = Get-Content ~/.ssh/id_ed25519.pub -Raw
ssh user@server.example.com "mkdir -p ~/.ssh && echo '$pubkey' >> ~/.ssh/authorized_keys && chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys"
```

Або за допомогою `Add-Content` для передачі через конвеєр:

```powershell
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh user@server.example.com "cat >> ~/.ssh/authorized_keys"
```

### Налаштування ssh-agent у Windows

Щоб не вводити парольну фразу повторно, запустіть службу OpenSSH Authentication Agent:

```powershell
# Запустити службу
Start-Service ssh-agent

# Налаштувати автозапуск
Set-Service -Name ssh-agent -StartupType Automatic

# Додати ключ
ssh-add ~/.ssh/id_ed25519
```

### PuTTY як альтернатива

Якщо надаєте перевагу графічному SSH-клієнту, PuTTY — безкоштовний варіант для Windows. Використовуйте **PuTTYgen** для генерації ключів (зберігайте у форматі `.ppk` для PuTTY і експортуйте у формат OpenSSH). Однак для нових налаштувань вбудований OpenSSH-клієнт простіший і сумісний з рештою екосистеми.

## Як налаштувати SSH-ключі для GitHub і GitLab

Автентифікація за SSH-ключем у GitHub і GitLab працює однаково — ви завантажуєте відкритий ключ до сервісу і використовуєте відповідний приватний ключ при Git-операціях.

### Налаштування SSH для GitHub

**1. Скопіюйте відкритий ключ:**

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows PowerShell
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

Або просто відобразіть і скопіюйте вручну:

```bash
cat ~/.ssh/id_ed25519.pub
```

**2. Додайте до GitHub:**

1. Перейдіть на **github.com** і увійдіть
2. Натисніть аватар → **Settings**
3. На лівій панелі: **SSH and GPG keys**
4. Натисніть **New SSH key**
5. Дайте зрозумілу **Назву** (наприклад, «MacBook Pro» або «Робочий ПК»)
6. Встановіть **Key type** у **Authentication Key**
7. Вставте відкритий ключ
8. Натисніть **Add SSH key**

**3. Перевірте з'єднання:**

```bash
ssh -T git@github.com
# Hi username! You've successfully authenticated, but GitHub does not provide shell access.
```

**4. Використовуйте SSH для Git-операцій:**

```bash
# Клонувати по SSH замість HTTPS
git clone git@github.com:username/repository.git

# Якщо наявний репозиторій використовує HTTPS, переключіть на SSH
git remote set-url origin git@github.com:username/repository.git
```

### Налаштування SSH для GitLab

Процес ідентичний:

1. Перейдіть на **gitlab.com** → Натисніть аватар → **Preferences**
2. На лівій панелі: **SSH Keys**
3. Вставте відкритий ключ, дайте назву і за бажанням встановіть дату закінчення
4. Натисніть **Add key**

Перевірте:

```bash
ssh -T git@gitlab.com
# Welcome to GitLab, @username!
```

Для self-hosted GitLab замініть `gitlab.com` іменем хоста вашого екземпляра.

### Використання окремого ключа для GitHub та інших сервісів

Якщо вам потрібен окремий ключ для GitHub (рекомендується для гігієни ключів), згенеруйте іменований ключ:

```bash
ssh-keygen -t ed25519 -C "github" -f ~/.ssh/id_ed25519_github
```

Потім додайте блок `Host` до `~/.ssh/config`:

```
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github
    AddKeysToAgent yes
```

Тепер `git clone git@github.com:...` автоматично використовує ключ, специфічний для GitHub.

## Тестування та усунення несправностей SSH

### Базова перевірка підключення

```bash
ssh -v user@server.example.com
```

Прапорець `-v` (детальний режим) показує, що саме відбувається на кожному кроці. Шукайте рядки вигляду:
- `Offering public key: ~/.ssh/id_ed25519` — SSH перевіряє ваш ключ
- `Server accepts key` — ключ є в `authorized_keys`
- `Authenticated to server.example.com` — успіх

### Поширені помилки та способи їх усунення

**«Permission denied (publickey)»**

Найпоширеніша помилка. Причини та рішення:

```bash
# Перевірити, чи завантажений ключ у ssh-agent
ssh-add -l

# Якщо ключі не перелічені, додати ключ
ssh-add ~/.ssh/id_ed25519

# На віддаленому сервері перевірити authorized_keys
cat ~/.ssh/authorized_keys
# Переконатися, що ваш відкритий ключ там є

# Перевірити права на віддаленому сервері
ls -la ~/.ssh/
# ~/.ssh повинна мати права 700
# authorized_keys повинна мати права 600
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

# Перевірити, чи дозволяє сервер автентифікацію за відкритим ключем
sudo grep PubkeyAuthentication /etc/ssh/sshd_config
# Має бути: PubkeyAuthentication yes
```

**«WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED»**

Хост-ключ сервера змінився — SSH розцінює це як потенційну загрозу безпеці. Якщо ви знаєте, що це легітимно (сервер перебудований, IP переназначений), видаліть старий запис:

```bash
# Видалити конкретний запис хоста
ssh-keygen -R server.example.com

# Або відредагувати known_hosts вручну і видалити відповідний рядок
nano ~/.ssh/known_hosts
```

**«Bad permissions» або «Permissions too open»**

SSH строгий щодо прав доступу до файлів ключів:

```bash
# Виправити права приватного ключа
chmod 600 ~/.ssh/id_ed25519

# Виправити права відкритого ключа
chmod 644 ~/.ssh/id_ed25519.pub

# Виправити права директорії .ssh
chmod 700 ~/.ssh
```

У Windows права приватних ключів керуються інакше. Клацніть правою кнопкою миші по файлу ключа → Властивості → Безпека → Додатково → Вимкнути успадкування → Видалити всіх, крім вашого облікового запису, надавши повний контроль тільки собі.

### Налаштування та використання ssh-agent

Якщо вас постійно запитують парольну фразу, `ssh-agent` не запущений або ключ не доданий:

```bash
# Запустити агент
eval "$(ssh-agent -s)"
# Agent pid 12345

# Додати ключ
ssh-add ~/.ssh/id_ed25519
# Enter passphrase for /home/user/.ssh/id_ed25519:
# Identity added: /home/user/.ssh/id_ed25519

# Перевірити
ssh-add -l
# 256 SHA256:abc123... your@email.com (ED25519)
```

### Попереднє заповнення known_hosts в автоматизації

У CI/CD-пайплайнах використовуйте `ssh-keyscan` для додавання відбитка віддаленого хоста перед підключенням:

```bash
ssh-keyscan -H server.example.com >> ~/.ssh/known_hosts
ssh-keyscan -H github.com >> ~/.ssh/known_hosts
```

Це запобігає інтерактивним запитам відбитка в автоматизованих скриптах.

## Часті запитання

### Як виправити «Permission denied (publickey)»?

Пройдіть по цьому чеклісту: (1) Переконайтеся, що ключ є в SSH-агенті: `ssh-add -l`. Якщо ні — виконайте `ssh-add ~/.ssh/id_ed25519`. (2) Перевірте, що ваш відкритий ключ є у файлі `~/.ssh/authorized_keys` на сервері. (3) Перевірте права: `~/.ssh` повинна мати права `700`, `authorized_keys` — `600`. (4) Переконайтеся, що на сервері `PubkeyAuthentication yes` у `/etc/ssh/sshd_config`. (5) Використайте `ssh -vvv user@server`, щоб побачити, де саме відбувається збій автентифікації. Детальний виведення покаже, які ключі перебираються і чи прийняв їх сервер.

### Як додати SSH-ключ до ssh-agent?

Виконайте `ssh-add ~/.ssh/id_ed25519` (замініть іменем вашого файлу ключа). Якщо агент не запущений, спочатку запустіть його за допомогою `eval "$(ssh-agent -s)"`. У macOS використовуйте `ssh-add --apple-use-keychain ~/.ssh/id_ed25519` для збереження парольної фрази в Keychain. У Windows увімкніть службу `ssh-agent` за допомогою `Start-Service ssh-agent`, потім виконайте `ssh-add`. Запускати `ssh-add` потрібно один раз за сеанс (або один раз назавжди з Keychain у macOS).

### Чи можна мати кілька SSH-ключів?

Так, і це хороша практика. Генеруйте окремі ключі для різних цілей — наприклад, один для особистого GitHub, один для робочого GitLab, один для продакшн-серверів. Використовуйте `~/.ssh/config` для зіставлення кожного хоста з конкретним ключем через `IdentityFile`. Наявність окремих ключів означає, що відкликання одного (видаленням із сервісу або сервера) не впливає на інші, обмежуючи наслідки компрометації ключа.

### Як змінити парольну фразу SSH-ключа?

Використайте `ssh-keygen -p`:

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

Вас попросять ввести поточну парольну фразу, потім задати і підтвердити нову. Сам ключ (і, відповідно, відкритий ключ) не зміниться — оновлювати його ніде не потрібно. Це корисно, якщо ви підозрюєте компрометацію парольної фрази або хочете змінити її в рамках планової ротації безпеки.

### Як видалити старий SSH-ключ із сервера?

Відкрийте `~/.ssh/authorized_keys` на сервері і видаліть рядок зі старим відкритим ключем:

```bash
# Відредагувати authorized_keys на віддаленому сервері
ssh user@server.example.com
nano ~/.ssh/authorized_keys
# Видалити рядок з ключем, який потрібно прибрати
# Зберегти та вийти
```

Кожен рядок у `authorized_keys` — один відкритий ключ. Коментар ключа (останнє поле, зазвичай `user@hostname` або email) допомагає визначити, який ключ є яким. Якщо не знаєте, який рядок видалити, порівняйте відбитки:

```bash
# На локальній машині отримати відбиток
ssh-keygen -lf ~/.ssh/id_ed25519.pub

# На сервері перевірити відбитки всіх авторизованих ключів
ssh-keygen -lf ~/.ssh/authorized_keys
```

Зіставте відбитки, щоб знайти й видалити потрібний ключ.

## Створіть SSH-ключі безкоштовно

Якщо ви ще не згенерували пару SSH-ключів, наш безкоштовний онлайн-генератор створює ключі Ed25519 або RSA прямо в браузері за лічені секунди. Встановлення не потрібне, нічого не надсилається на наші сервери — генерація відбувається повністю на вашому пристрої.
