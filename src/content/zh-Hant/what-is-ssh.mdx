# 什麼是 SSH？安全外殼協定完整指南

SSH 是 **Secure Shell（安全外殼）** 的縮寫，是一種加密網路協定，讓您能夠透過不安全的網路，安全地存取和管理遠端電腦。無論是登入 Linux 伺服器、將程式碼推送到 GitHub，還是在機器之間傳輸檔案，SSH 幾乎都是在幕後負責這一切的協定。本指南將詳細說明您需要了解的 SSH 相關知識：它是什麼、如何運作，以及為何它已成為安全遠端存取的通用標準。

## SSH 是什麼的縮寫？

**SSH 是 Secure Shell 的縮寫**。這個名稱準確描述了該協定的功能——它為遠端機器提供安全、加密的外殼會話。在 SSH 出現之前，管理員使用 Telnet 和 rsh（遠端外殼）等工具來管理遠端伺服器。這些協定以明文傳輸所有內容，包括使用者名稱和密碼，在任何共享網路上都極易遭到攔截。

SSH 由芬蘭電腦科學家 **Tatu Ylönen** 於 1995 年創建，直接動機是他親眼目睹了大學網路上發生的密碼竊取攻擊。他以自由軟體的形式發布了 SSH-1，該協定很快被整個網際網路採用。後來，IETF 將該協定標準化為 SSH-2（RFC 4251–4254），引入了更強的加密機制，並修復了原始設計中的幾個安全漏洞。

如今，人們說的「SSH」幾乎都指 SSH-2。最廣泛使用的實作是 **OpenSSH**，這是一個由 OpenBSD 團隊維護的開源專案，幾乎所有 Linux 發行版和 macOS 都已內建。

**SSH 的實際意義**遠不止是一個登入工具，它是一套完整的協定套件，支撐著：
- 安全的遠端終端機存取
- 加密的檔案傳輸（SFTP 和 SCP）
- Git 驗證（GitHub、GitLab、Bitbucket）
- 安全的連接埠轉送與通道
- 自動化部署流水線

## SSH 的運作原理

**安全外殼協定**採用分層架構，分別處理加密、驗證和通道多工。了解這些層次有助於理解為何 SSH 能夠勝任敏感操作。

### 三個協定層次

**1. 傳輸層（RFC 4253）**

當您發起 SSH 連線時，用戶端和伺服器首先執行握手，協商加密演算法並建立共享金鑰。現代 SSH 預設使用基於 Curve25519 的金鑰交換演算法。握手完成後，後續所有通訊都會被加密。傳輸層還透過訊息驗證碼（MAC）提供資料完整性，確保資料在傳輸過程中不被篡改。

**2. 使用者驗證層（RFC 4252）**

建立加密通道後，伺服器要求用戶端進行身份驗證。SSH 支援多種驗證方式：

- **公開金鑰驗證** — 用戶端証明擁有私密金鑰，而無需透過網路傳送。這是最安全也是最常見的方式。
- **密碼驗證** — 使用者密碼透過加密通道傳送。較為便利，但容易受到暴力破解攻擊。
- **鍵盤互動式** — 挑戰/回應機制，常用於多重要素驗證。
- **GSSAPI/Kerberos** — 用於具有集中式身份管理的企業環境。

**3. 連線層（RFC 4254）**

驗證完成後，連線層將單一加密連線多工為多個邏輯通道。這使您可以透過單一 TCP 連線同時進行互動式外殼、執行遠端命令、轉送連接埠和傳輸檔案。

### 金鑰交換詳解

當您輸入 `ssh user@example.com` 時，以下事情在最初幾毫秒內發生：

1. 用戶端連線到伺服器的 TCP 22 埠。
2. 雙方交換協定版本字串。
3. 雙方傳送支援的演算法清單（金鑰交換、加密、MAC、壓縮）。
4. 執行 Diffie-Hellman 或 ECDH 金鑰交換以推導共享會話金鑰——這就是為何即使有人錄製了加密流量，之後也無法解密。
5. 伺服器傳送其**主機金鑰**（公開金鑰），用戶端將其與 `~/.ssh/known_hosts` 檔案中的記錄進行驗證。
6. 按上述方式進行身份驗證。
7. 會話開始。

## SSH 與 Telnet：為何 SSH 勝出

在 SSH 之前，**Telnet** 是遠端終端機存取的標準協定。Telnet 簡單且仍然可用，但它有一個致命缺陷：所有內容都以明文傳輸。您的使用者名稱、密碼、每一個輸入的命令以及每一條輸出——任何能夠攔截網路流量的人都能看到。

1990 年代，當網際網路主幹是受信任的學術和研究網路時，這並不被視為嚴重問題。隨著網際網路商業化並出現對抗性環境，Dsniff 和 Ettercap 等工具讓在區域網路上擷取 Telnet 憑證變得輕而易舉。

SSH 透過加密所有內容解決了這個問題。以下是直接比較：

| 功能 | Telnet | SSH |
|---|---|---|
| 加密 | 無 | 強加密（AES、ChaCha20） |
| 驗證 | 明文密碼 | 公開金鑰 + 加密密碼 |
| 資料完整性 | 無 | HMAC 驗證 |
| 連接埠 | 23 | 22 |
| 仍建議使用 | 否 | 是 |

rsh（遠端外殼）和 rlogin 也有類似問題，它們依賴於基於 IP 的信任，而這很容易被偽造。SSH 取代了所有這些協定，並在超過 25 年的時間裡一直是無可爭議的標準。

## 什麼是 SSH 金鑰？

SSH 支援兩種主要的驗證方式：密碼和 **SSH 金鑰**。SSH 金鑰是基於非對稱加密的加密金鑰對——**公開金鑰**和**私密金鑰**。

**私密金鑰**保存在您的本機，可選擇使用密碼短語加以保護。**公開金鑰**放置在您想要存取的任何伺服器上，存放在名為 `~/.ssh/authorized_keys` 的檔案中。連線時，伺服器發出加密挑戰，只有持有對應私密金鑰的人才能回答。不會有密碼透過網路傳送。

這種方式有幾個優點：
- **不受暴力破解影響** — 沒有什麼可以猜測的。一個 256 位元的 Ed25519 金鑰有 2²⁵⁶ 個可能值。
- **無密碼可竊取** — 即使有人攔截握手過程，也得不到任何有用的資訊。
- **可自動化** — 腳本和 CI/CD 流水線可以在無人干預的情況下進行驗證。
- **可撤銷** — 從 `authorized_keys` 中移除公開金鑰即可立即撤銷存取權限。

最現代且推薦的金鑰類型是 **Ed25519**，它使用基於 Edwards 曲線的橢圓曲線加密。RSA 金鑰也被廣泛支援，但應至少使用 4096 位元。

如需深入了解金鑰對，請參閱我們的指南：<a href="/zh-Hant/what-is-an-ssh-key/">什麼是 SSH 金鑰</a>。

## SSH 常見使用情境

SSH 應用於非常廣泛的場景。以下是最重要的幾種。

### 遠端伺服器管理

最基本的使用情境。系統管理員透過 SSH 終端機會話登入 Linux 和 Unix 伺服器，安裝軟體、編輯設定檔、管理服務並診斷問題。

```bash
ssh user@192.168.1.100
ssh admin@myserver.example.com
ssh -p 2222 deploy@staging.example.com
```

### 使用 SFTP 和 SCP 傳輸檔案

SSH 提供兩種建立在相同加密通道上的檔案傳輸協定：

**SCP（安全複製協定）** 是一個用於複製檔案的簡單命令列工具：

```bash
# 將本地檔案複製到遠端伺服器
scp report.pdf user@server:/home/user/documents/

# 將遠端檔案複製到本機
scp user@server:/var/log/app.log ./local-copy.log

# 遞迴複製整個目錄
scp -r ./project/ user@server:/var/www/
```

**SFTP（SSH 檔案傳輸協定）** 提供類似 FTP 的互動式介面，功能比 SCP 更豐富，包括可恢復的傳輸和目錄列表。

### Git 驗證

當您將 SSH 金鑰加入 GitHub、GitLab 或 Bitbucket 後，所有透過 SSH 進行的 `git clone`、`git push` 和 `git pull` 操作都會使用您的私密金鑰進行驗證。這樣就不需要在每次 Git 操作時輸入使用者名稱和密碼，是開發者推薦的驗證方式。

```bash
# 使用 SSH 複製儲存庫
git clone git@github.com:username/repository.git

# 驗證您的 GitHub SSH 連線
ssh -T git@github.com
```

### 自動化部署與 CI/CD

GitHub Actions、GitLab CI 和 Jenkins 等部署腳本和 CI/CD 系統使用 SSH 金鑰將程式碼部署到生產伺服器，無需人工干預。私密金鑰以機密形式儲存在 CI 系統中，對應的公開金鑰則被加入伺服器的 `authorized_keys` 檔案。

### 資料庫與應用程式連接埠轉送

SSH 通道可以將遠端服務安全地暴露給您的本機。例如，要存取未暴露在網際網路上的遠端 MySQL 資料庫：

```bash
ssh -L 3307:localhost:3306 user@dbserver.example.com -N
```

這會建立一個本地通道：連線到本機的 3307 埠會透明地將流量轉送到遠端伺服器的 3306 埠，並在 SSH 通道中加密。

## SSH 連接埠與設定選項

### 預設連接埠 22

SSH 預設監聽 **TCP 22 埠**。這是 IANA 指定的知名連接埠，自協定建立以來就為 SSH 保留。當您執行 `ssh user@host` 而不指定連接埠時，SSH 用戶端會連線到 22 埠。

部分管理員會更改 SSH 連接埠（例如改為 2222 或 8022），作為減少日誌中自動掃描雜訊的小型安全措施。這稱為「透過晦澀性實現安全」，不能替代正確的基於金鑰的驗證，但可以減少日誌雜訊。

```bash
# 連線到非標準 SSH 連接埠
ssh -p 2222 user@example.com
```

### SSH 用戶端設定檔

與其每次都輸入冗長的命令列選項，您可以在 `~/.ssh/config` 中設定 SSH 連線。此檔案讓您定義別名、設定每個主機的選項，並大幅簡化工作流程。

```
Host myserver
    HostName 203.0.113.10
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_prod

Host bastion
    HostName jump.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host internal
    HostName 10.0.1.50
    User ubuntu
    ProxyJump bastion
```

有了這個設定，`ssh myserver` 就會使用指定的金鑰在自訂連接埠上連線到完整主機名稱——無需任何旗標。

如需 SSH 命令選項和旗標的完整參考，請參閱我們的 <a href="/zh-Hant/ssh-command/">SSH 命令指南</a>。

### 伺服器端設定（sshd_config）

SSH 伺服器常駐程式（sshd）透過 `/etc/ssh/sshd_config` 進行設定。關鍵設定包括：

```
# 停用密碼驗證——僅使用金鑰
 PasswordAuthentication no

# 停用 root 登入
 PermitRootLogin no

# 更改監聽連接埠
Port 2222

# 限制特定使用者
AllowUsers deploy admin

# 啟用公開金鑰驗證
PubkeyAuthentication yes
```

編輯後，重新載入常駐程式：`sudo systemctl reload sshd`

## SSH 通道

SSH 通道（也稱為 SSH 連接埠轉送）允許您透過加密的 SSH 連線路由任意 TCP 流量。這是 SSH 最強大但也是最少被利用的功能之一。

### 本地連接埠轉送

本地轉送使遠端服務可在您的本機上存取。語法為 `-L local_port:remote_host:remote_port`。

```bash
# 在本地存取遠端 PostgreSQL 資料庫
ssh -L 5433:localhost:5432 user@dbserver -N

# 透過跳板伺服器存取內部 Web 應用程式
ssh -L 8080:internal-app.company.com:80 user@jump.company.com -N
```

`-N` 旗標告訴 SSH 不要執行遠端命令——當您只需要通道時很有用。

### 遠端連接埠轉送

遠端轉送在遠端伺服器上暴露本地服務。語法為 `-R remote_port:local_host:local_port`。

```bash
# 將您的本地 Web 伺服器暴露給遠端機器
ssh -R 9090:localhost:3000 user@server -N
```

這對於臨時與有權存取遠端主機的同事分享本地開發伺服器，或讓客戶臨時存取您正在進行的工作非常有用。

### 動態連接埠轉送（SOCKS 代理）

`-D` 旗標在您的本機建立一個 SOCKS 代理，將所有相容 SOCKS 的流量路由透過 SSH 連線。這實際上將遠端伺服器用作 VPN 出口節點。

```bash
# 在本地 1080 埠建立 SOCKS5 代理
ssh -D 1080 user@remote-server -N -f
```

設定您的瀏覽器或系統使用 `SOCKS5 代理：127.0.0.1:1080`，所有網路流量將透過遠端伺服器路由。這通常用於存取受地理限制的內容，或在不受信任的網路中安全瀏覽。

## 常見問題

### SSH 是什麼的縮寫？

SSH 是 **Secure Shell** 的縮寫。它是一種加密網路協定，用於在不安全的網路上安全地操作網路服務。這個名稱指的是其主要用途：在遠端電腦上提供安全的互動式外殼會話。

### SSH 和 SSL 是一樣的嗎？

不一樣。SSH 和 SSL/TLS 都是加密協定，但用途不同。**SSL/TLS**（安全通訊層/傳輸層安全性）用於保護網路流量（HTTPS）、電子郵件和其他應用程式協定。**SSH** 專門設計用於安全的遠端外殼存取、檔案傳輸和通道。它們使用類似的加密基元（公開金鑰加密、對稱加密），但握手機制和使用情境不同。SSH 內部不使用 SSL/TLS。

### SSH 使用哪個連接埠？

SSH 預設使用 **TCP 22 埠**。這是 IANA 指定的知名連接埠。您可以使用 `/etc/ssh/sshd_config` 中的 `Port` 指令將 SSH 伺服器設定為監聽不同的連接埠，並使用 `ssh -p PORT user@host` 連線到非標準連接埠。

### SSH 能被入侵嗎？

正確設定了基於金鑰驗證的 SSH 伺服器極難被入侵。加密機制——Ed25519 金鑰、AES-256 加密、SHA-2 MAC——被認為對金鑰驗證具有量子抗性，且在計算上無法被暴力破解。現實中的 SSH 入侵通常涉及：私密金鑰檔案被竊取、密碼短語薄弱、伺服器設定錯誤允許弱密碼進行密碼驗證、含有未修補 CVE 的易受攻擊的軟體版本，或者是社交工程攻擊，而非加密攻擊。請保持伺服器更新、停用密碼驗證，並保護您的私密金鑰檔案。

### SSH 和 HTTPS 有什麼區別？

SSH 和 HTTPS 都加密網路流量，但設計用途不同。**HTTPS** 保護網路瀏覽——它使用 TLS 加密瀏覽器和網路伺服器之間的 HTTP 請求和回應。**SSH** 保護遠端管理存取、檔案傳輸和通道。當您向 GitHub 推送程式碼時，可以使用 SSH（22 埠，基於金鑰的驗證）或 HTTPS（443 埠，基於令牌的驗證）。開發者通常偏好 SSH，因為它不需要重新輸入憑證。HTTPS 則可以通過可能封鎖 22 埠的防火牆。
