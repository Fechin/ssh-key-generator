# 什麼是 SSH 金鑰？公開金鑰、私密金鑰與金鑰對詳解

**SSH 金鑰**是一種加密憑證，用於在無需密碼的情況下向 SSH 伺服器進行身份驗證。SSH 金鑰成對出現——**公開金鑰**和**私密金鑰**——兩者合在一起，使遠端存取比傳統密碼驗證更加安全且便利。本指南將說明什麼是 SSH 金鑰、金鑰對系統的運作原理、如何選擇金鑰類型，以及如何在 GitHub 和 GitLab 等服務中使用 SSH 金鑰。

## SSH 金鑰的運作原理

SSH 金鑰基於**非對稱加密**，也稱為公開金鑰加密。與對稱加密（使用單一金鑰同時加密和解密資料）不同，非對稱系統使用數學上關聯的金鑰對，其中用一個金鑰加密的資料只能用另一個金鑰解密。

以下是使用 SSH 金鑰登入伺服器時的驗證流程：

1. 您執行 `ssh user@server.example.com`。
2. 伺服器在 `~/.ssh/authorized_keys` 中查找您的公開金鑰。
3. 伺服器產生一個隨機挑戰，並用您的公開金鑰加密。
4. 您的 SSH 用戶端使用您的私密金鑰解密該挑戰。
5. 用戶端將加密証明（由解密後的挑戰和會話 ID 推導而來）傳回伺服器。
6. 伺服器驗證該証明。只有持有私密金鑰的人才能產生它。
7. 驗證成功。您的機器上從未傳送過任何密碼。

核心要點：**您的私密金鑰永遠不會透過網路傳輸**。伺服器只能看到您的公開金鑰以及您產生的加密証明。即使有人攔截了您 SSH 會話中的每一個封包，他們也無法冒充您。

如需了解 SSH 協定的背景知識，請參閱我們的指南：<a href="/zh-Hant/what-is-ssh/">什麼是 SSH</a>。

## 公開金鑰與私密金鑰：有什麼區別？

理解兩個金鑰之間的區別對於正確使用 SSH 至關重要。

### 公開金鑰——您的鎖

**公開金鑰**設計為可以自由分享。把它想像成一把掛鎖：您將副本分發給您想要存取其系統的所有人。當伺服器在其 `authorized_keys` 檔案中有您的公開金鑰時，它可以驗證您的身份，而無需知道您的私密金鑰。

公開金鑰：
- 可以安全分享——無法用來冒充您
- 放置在伺服器的 `~/.ssh/authorized_keys` 中
- 上傳到 GitHub、GitLab 和其他服務
- 通常具有 `.pub` 副檔名（例如 `id_ed25519.pub`）
- 格式如：`ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... user@hostname`

### 私密金鑰——您的鑰匙

**私密金鑰**是金鑰對的秘密部分。它絕不能離開您的機器，也不應與任何人分享——包括管理員、雲端服務提供商或支援人員。如果您的私密金鑰遭到洩露，任何獲得它的人都可以在您安裝了公開金鑰的每台伺服器上冒充您。

私密金鑰：
- 必須保密且安全保管
- 以受限權限（600）儲存在本地的 `~/.ssh/` 目錄中
- 可選擇使用密碼短語加密
- 沒有副檔名或有特定語言的副檔名（例如 `id_ed25519`）

### `authorized_keys` 檔案

在伺服器端，每個使用者的公開金鑰儲存在 `~/.ssh/authorized_keys` 中。這是一個純文字檔案，每一行包含一個公開金鑰。當 SSH 用戶端嘗試驗證時，伺服器會檢查用戶端是否能證明擁有與此檔案中任何條目對應的私密金鑰。

```
# ~/.ssh/authorized_keys 格式
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGq... alice@laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHr... alice@workstation
ssh-rsa AAAAB3NzaC1yc2EAAAA... backup-deploy-key
```

您可以在此檔案中保存任意數量的公開金鑰，允許多個裝置或自動化系統向同一伺服器進行驗證。

## SSH 金鑰演算法：Ed25519 與 RSA

產生 SSH 金鑰對時，您需要選擇演算法。目前最常見的兩個選項是 **Ed25519** 和 **RSA**。還有 ECDSA 和 DSA，但 DSA 已被棄用，ECDSA 存在實作陷阱，因此我們將重點介紹這兩個推薦選項。

### Ed25519——現代選擇

Ed25519 於 2014 年在 OpenSSH 6.5 版本中引入，現在是推薦的預設選項。它使用由 Daniel J. Bernstein 設計的 Twisted Edwards 曲線（Curve25519）上的橢圓曲線加密。

**Ed25519 的優點：**
- **緊湊** — 金鑰僅 68 個字元（公開金鑰），檔案很小
- **快速** — 簽章產生和驗證極為快速
- **設計安全** — 不易受到影響某些 RSA 實作的旁道攻擊
- **固定金鑰大小** — 256 位元安全等級，無需選擇金鑰大小
- **確定性** — 簽章是確定性的，消除了一類隨機數產生器失敗的問題

```bash
# 產生 Ed25519 金鑰對
ssh-keygen -t ed25519 -C "your@email.com"
```

### RSA——通用選項

RSA 是較老的演算法（1977 年發明），雖然在 4096 位元下仍然安全，但它比 Ed25519 更慢且產生更大的金鑰。然而，RSA 具有更廣泛的相容性——它可以與非常老舊的 SSH 伺服器、舊系統和某些硬體安全金鑰配合使用。

**何時使用 RSA：**
- 您需要連線到執行舊版 OpenSSH（6.5 之前）的舊系統
- 您使用的智慧卡或 HSM 不支援 Ed25519
- 需要與舊工具相容

```bash
# 產生 RSA 4096 位元金鑰對
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

**切勿使用少於 4096 位元的 RSA。** 隨著計算能力的增長，RSA-1024 和 RSA-2048 被認為越來越不足。Ed25519 的 256 位元提供了等效或更好的安全性，而金鑰大小卻只有其一小部分。

### 摘要比較

| 功能 | Ed25519 | RSA 4096 |
|---|---|---|
| 金鑰大小（公開） | 約 68 個字元 | 約 725 個字元 |
| 安全等級 | 128 位元（具有後量子抗性結構） | 約 140 位元傳統安全 |
| 速度 | 非常快 | 較慢 |
| 相容性 | 現代 SSH（OpenSSH 6.5+） | 通用 |
| 推薦用途 | 所有新金鑰 | 僅限舊系統 |

**結論：對於任何現代使用情境，請產生 Ed25519 金鑰。** 僅在相容性要求所迫時才使用 RSA。

## 如何將 SSH 金鑰加入 GitHub

將您的 SSH 公開金鑰加入 GitHub，讓您可以透過 SSH 推送和拉取，而無需在每次操作時輸入使用者名稱和密碼。以下是逐步流程。

### 第 1 步：產生金鑰對

如果您還沒有 SSH 金鑰對，請先產生一個。您可以使用我們的<a href="/">免費線上 SSH 金鑰產生器</a>或執行：

```bash
ssh-keygen -t ed25519 -C "your-github-email@example.com"
```

接受預設的檔案位置（`~/.ssh/id_ed25519`），並可選擇設定密碼短語。

### 第 2 步：複製您的公開金鑰

顯示您的公開金鑰並複製到剪貼簿：

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows（PowerShell）
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

或直接顯示並手動複製：

```bash
cat ~/.ssh/id_ed25519.pub
```

### 第 3 步：將金鑰加入 GitHub

1. 登入 **GitHub.com**
2. 點擊您的個人頭像 → **Settings（設定）**
3. 在左側邊欄中，點擊 **SSH and GPG keys（SSH 和 GPG 金鑰）**
4. 點擊 **New SSH key（新增 SSH 金鑰）**
5. 給它一個描述性標題（例如「MacBook Pro 2024」或「Work Laptop」）
6. 將您的公開金鑰貼入 **Key** 欄位
7. 點擊 **Add SSH key（新增 SSH 金鑰）**

### 第 4 步：測試連線

```bash
ssh -T git@github.com
```

您應該看到：`Hi username! You've successfully authenticated, but GitHub does not provide shell access.`

### 將現有儲存庫切換到 SSH

如果您使用 HTTPS 複製了儲存庫並想切換到 SSH：

```bash
# 檢查目前的遠端 URL
git remote -v

# 更新為 SSH
git remote set-url origin git@github.com:username/repository.git
```

對於 GitLab，流程完全相同——前往 **Preferences → SSH Keys** 並按照相同步驟操作。

## SSH 金鑰儲存在哪裡？

SSH 金鑰儲存在您使用者家目錄下名為 `.ssh` 的目錄中。位置因作業系統而異。

### Linux 和 macOS

```
~/.ssh/
├── id_ed25519          # 您的私密金鑰（權限：600）
├── id_ed25519.pub      # 您的公開金鑰（權限：644）
├── id_rsa              # RSA 私密金鑰（如果有）
├── id_rsa.pub          # RSA 公開金鑰
├── authorized_keys     # 允許登入此機器的公開金鑰
├── known_hosts         # 您已連線過的伺服器指紋
└── config              # 用戶端設定檔
```

`~/.ssh/` 目錄本身必須具有 **700** 權限（僅擁有者可讀寫執行）。私密金鑰檔案必須具有 **600** 權限（僅擁有者可讀寫）。SSH 會拒絕使用權限過於寬鬆的金鑰。

```bash
# 如需要，修正權限
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

### Windows

在安裝了 OpenSSH 的 Windows 10 和 11 上，SSH 金鑰儲存在：

```
C:\Users\YourUsername\.ssh\
├── id_ed25519
├── id_ed25519.pub
├── authorized_keys
├── known_hosts
└── config
```

在 Windows 上，您可以透過檔案總管（可能需要啟用「顯示隱藏的項目」）或使用 PowerShell 中的 `~\.ssh\` 來存取 `.ssh` 目錄。

### 多個金鑰檔案

如果您有不同用途的金鑰（個人 GitHub、工作 GitLab、生產伺服器），請使用 `~/.ssh/config` 檔案將每個主機對應到其金鑰：

```
Host github.com
    IdentityFile ~/.ssh/id_ed25519_github

Host gitlab.work.com
    IdentityFile ~/.ssh/id_ed25519_work

Host prod-server
    HostName 203.0.113.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_production
```

## SSH 金鑰密碼短語

產生 SSH 金鑰對時，`ssh-keygen` 會詢問您是否要設定**密碼短語**。這是一個可選密碼，用於加密磁碟上的私密金鑰檔案。

### 為何要使用密碼短語？

如果您的私密金鑰檔案受到強密碼短語保護，那麼僅竊取該檔案還不足以冒充您——攻擊者還需要破解密碼短語。這提供了關鍵的第二層安全防護，尤其是對於可能遭到實體竊盜的筆記型電腦。

沒有密碼短語的情況下，您的私密金鑰以明文儲存在磁碟上。任何能讀取該檔案的人（透過惡意軟體、實體存取或未妥善保管的備份）都可以立即使用它。

### 密碼短語如何與 ssh-agent 配合使用

密碼短語的缺點是每次使用金鑰時都需要輸入。**ssh-agent** 透過在會話期間將解密的金鑰保存在記憶體中來解決這個問題。

```bash
# 啟動 ssh-agent 並加入您的金鑰
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
# 輸入一次密碼短語——現在它已在會話中快取
```

在 macOS 上，系統鑰匙圈與 ssh-agent 整合。加入 `-K` 以在重新開機後持久保存：

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

### 更改密碼短語

您可以在不重新產生金鑰對的情況下更改現有金鑰的密碼短語：

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

這會提示您輸入舊密碼短語，然後讓您設定新密碼短語。公開金鑰不會改變，因此您不需要在任何伺服器上更新它。

## SSH 金鑰指紋

每個 SSH 金鑰對都有一個**指紋**——公開金鑰的短雜湊值，可唯一識別它。指紋有幾個用途：

- **驗證主機身份** — 首次連線到伺服器時，SSH 會顯示其主機金鑰指紋。如果您確認它與您期望的相符（例如由您的託管服務提供商提供），您就知道您連接到了正確的伺服器。
- **識別金鑰** — 管理多個金鑰時，指紋幫助您識別哪個實體金鑰對應於 `authorized_keys` 或 GitHub 等服務上的哪個條目。
- **稽核和撤銷** — 稽核日誌通常記錄指紋而非完整金鑰。知道金鑰的指紋有助於您在需要時找到並移除它。

### 查看金鑰的指紋

```bash
# 查看您公開金鑰的指紋
ssh-keygen -lf ~/.ssh/id_ed25519.pub
# 輸出：256 SHA256:abcdef1234567890... your@email.com (ED25519)

# 查看伺服器主機金鑰的指紋
ssh-keyscan server.example.com | ssh-keygen -lf -
```

GitHub 在 **Settings → SSH and GPG keys** 中顯示所有已加入 SSH 金鑰的指紋，讓您可以驗證您在那裡看到的金鑰與您機器上的金鑰是否一致。

### `known_hosts` 檔案

第一次連線到新的 SSH 伺服器時，您會看到如下訊息：

```
The authenticity of host 'example.com (203.0.113.10)' can't be established.
ED25519 key fingerprint is SHA256:abcdef1234567890...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

輸入 `yes`，伺服器的指紋就會被加入 `~/.ssh/known_hosts`。在後續連線中，SSH 會靜默地驗證伺服器的金鑰是否與儲存的指紋相符。如果不符——這可能表示設定錯誤或中間人攻擊——SSH 會大聲警告並拒絕連線。

## 常見問題

### SSH 金鑰有什麼用途？

SSH 金鑰讓您無需密碼即可向遠端伺服器進行身份驗證。最常見的用途包括：登入 Linux/Unix 伺服器進行管理、使用 Git 託管服務（GitHub、GitLab、Bitbucket）進行推送/拉取存取、需要無密碼伺服器存取的自動化部署腳本，以及將程式碼部署到生產環境的 CI/CD 流水線。SSH 金鑰比密碼更受歡迎，因為它們不受暴力破解影響、不可在洩露事件中重複使用，且可以在無人干預的情況下使用。

### 我應該有多少個 SSH 金鑰？

沒有統一的正確答案，但常見的最佳實踐是**每台裝置一個金鑰**（而非每個服務一個）。您的筆記型電腦有一個金鑰對；您的桌機有另一個；您的 CI 伺服器有另一個。這樣，如果某台裝置被竊盜或洩露，您只需撤銷該裝置的金鑰。有些人進一步按情境劃分（個人與工作），每台裝置使用兩個金鑰。避免在多台機器間共享單一金鑰——如果該金鑰洩露，您所有的存取權限都會同時面臨風險。

### 我可以分享我的 SSH 公開金鑰嗎？

可以。公開金鑰設計為可以分享。透過電子郵件傳送、發布在 GitHub 文件中，或者在請求伺服器存取的票據中包含它，都是安全的。僅靠公開金鑰無法用來以您的身份進行驗證——那需要對應的私密金鑰，而只有您擁有。這是非對稱加密的基本特性。切勿與任何人分享您的**私密**金鑰。

### 如果我遺失了 SSH 私密金鑰會怎樣？

如果您遺失了私密金鑰且沒有備份，您就失去了使用它進行驗證的能力。仍然安裝在伺服器和服務上的公開金鑰會成為孤兒。您需要另一種方式存取伺服器（如果啟用了密碼驗證、控制台存取或您的託管服務提供商的緊急存取機制），以便從 `authorized_keys` 中移除舊公開金鑰並加入新的。這就是為何備份您的私密金鑰（在密碼管理器或加密備份中）很重要。私密金鑰無法從公開金鑰中重建。

### Ed25519 還是 RSA 更好？

**Ed25519 對於所有新金鑰的產生都更好。** 它更快、產生更小的金鑰、不易受到影響 RSA 的某些實作漏洞的影響，並且 256 位元金鑰提供了等效或更優越的安全性，相比 RSA-4096。今天使用 RSA 的唯一理由是與非常老舊的 SSH 伺服器軟體（2014 年發布的 OpenSSH 6.5 之前的版本）或舊硬體安全模組的相容性。如果您要存取任何現代伺服器或 GitHub 等服務，請使用 Ed25519。
