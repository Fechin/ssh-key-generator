# Как настроить SSH: полное пошаговое руководство для Linux, Mac и Windows

Правильная настройка SSH занимает около десяти минут и окупается годами. После настройки вы сможете входить на серверы без ввода паролей, отправлять код в GitHub без ввода учётных данных при каждой операции и автоматизировать развёртывание с уверенностью. Это руководство проведёт вас через весь процесс — от генерации пары ключей до устранения распространённых ошибок — для Linux, macOS и Windows.

Если вы сначала хотите разобраться в лежащих в основе технологиях, прочитайте наши руководства о том, <a href="/ru/what-is-ssh">что такое SSH</a> и <a href="/ru/what-is-an-ssh-key">что такое SSH-ключи</a>.

## Предварительные требования

Перед началом убедитесь, что у вас есть:

**Для доступа к серверу:**
- Терминальное приложение (Terminal в macOS/Linux, Windows Terminal или PowerShell в Windows)
- Удалённый сервер на Linux/Unix с установленным программным обеспечением SSH-сервера (sshd)
- Учётная запись пользователя на этом сервере (или root-доступ для её создания)
- IP-адрес или имя хоста сервера

**Для доступа к GitHub/GitLab:**
- Аккаунт на GitHub или GitLab
- Терминал с доступной командой `ssh`
- Опционально: git, установленный локально

**Требования к операционной системе:**
- **Linux**: SSH-клиент предустановлен практически во всех дистрибутивах
- **macOS**: SSH-клиент встроен, доступен из Terminal
- **Windows 10 версии 1809+ / Windows 11**: OpenSSH-клиент доступен как дополнительный компонент

Устанавливать ничего не нужно ни в Linux, ни в macOS. В Windows проверьте наличие OpenSSH, открыв PowerShell и выполнив `ssh -V`. Если получите строку версии — готово. Если нет — см. Шаг 4 с инструкцией по установке.

## Шаг 1: Сгенерируйте пару SSH-ключей

Вам нужна пара ключей, прежде чем вы сможете что-либо сделать. Пара ключей состоит из **приватного ключа** (остаётся на вашей машине) и **открытого ключа** (передаётся на серверы или сервисы типа GitHub).

### Вариант А: Используйте онлайн-генератор

Если предпочитаете графический интерфейс, наш <a href="/">бесплатный онлайн-генератор SSH-ключей</a> создаёт пары ключей Ed25519 или RSA в браузере. Приватный ключ генерируется локально с помощью Web Crypto API — ничего не отправляется на наши серверы. Загрузите оба файла и сохраните их в `~/.ssh/` (Linux/macOS) или `C:\Users\YourName\.ssh\` (Windows).

### Вариант Б: Используйте ssh-keygen в командной строке

Команда `ssh-keygen` доступна в Linux, macOS и Windows с OpenSSH:

```bash
# Сгенерировать пару ключей Ed25519 (рекомендуется)
ssh-keygen -t ed25519 -C "your@email.com"
```

**Что означает каждый флаг:**
- `-t ed25519` — тип ключа (Ed25519 современный, быстрый и безопасный)
- `-C "комментарий"` — метка, встроенная в открытый ключ, обычно ваш email

Вам будет предложено:

```
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/user/.ssh/id_ed25519):
```

Нажмите **Enter** для принятия расположения по умолчанию. При наличии нескольких ключей можно дать им описательные имена (например, `/home/user/.ssh/id_ed25519_github`).

```
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
```

Введите надёжную парольную фразу. Она шифрует файл приватного ключа на диске — при краже машины ключ нельзя будет использовать без парольной фразы. Можно оставить пустой для сценариев автоматизации, где участие человека невозможно, но для персональных ключей всегда используйте парольную фразу.

**Вывод:**

```
Your identification has been saved in /home/user/.ssh/id_ed25519
Your public key has been saved in /home/user/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:abc123defghijk... your@email.com
The key's randomart image is:
+--[ED25519 256]--+
...
```

Создаются два файла:
- `~/.ssh/id_ed25519` — ваш **приватный ключ** (никогда не передавайте)
- `~/.ssh/id_ed25519.pub` — ваш **открытый ключ** (можно передавать)

### Для RSA (совместимость с устаревшими системами)

Если нужно подключиться к очень старым системам без поддержки Ed25519:

```bash
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

Используйте не менее 4096 бит. RSA-ключи меньше 4096 бит больше не считаются достаточно защищёнными.

## Шаг 2: Настройка SSH в Linux

В этом разделе описано копирование открытого ключа на Linux-сервер для аутентификации без пароля.

### Метод 1: ssh-copy-id (самый простой)

`ssh-copy-id` — простейший способ установить открытый ключ на удалённый сервер. Он подключается с использованием аутентификации по паролю, добавляет открытый ключ в файл `~/.ssh/authorized_keys` сервера и устанавливает правильные права доступа.

```bash
ssh-copy-id user@server.example.com

# Явно указать файл ключа
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server.example.com

# Подключиться на нестандартном порту
ssh-copy-id -p 2222 user@server.example.com
```

Вас попросят ввести пароль сервера последний раз. После этого вы сможете подключаться с помощью `ssh user@server.example.com` используя ключ.

### Метод 2: Ручное копирование

Если `ssh-copy-id` недоступен (распространено в некоторых BSD-системах или минималистичных установках Linux):

```bash
# Отобразить открытый ключ и скопировать вывод
cat ~/.ssh/id_ed25519.pub

# Затем подключиться к серверу (с паролем) и выполнить:
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "ВСТАВЬТЕ_ОТКРЫТЫЙ_КЛЮЧ_ЗДЕСЬ" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

Или всё в одном конвейере:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@server.example.com \
  "mkdir -p ~/.ssh && chmod 700 ~/.ssh && \
   cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

### Проверка установки ключа

```bash
ssh user@server.example.com
# Должно подключиться без запроса пароля (только парольная фраза, если вы её задали)
```

### Настройка SSH-сервера (sshd)

На удалённом сервере конфигурация SSH-демона находится в `/etc/ssh/sshd_config`. Для безопасной настройки:

```bash
sudo nano /etc/ssh/sshd_config
```

Рекомендуемые настройки:

```
# Включить аутентификацию по открытому ключу
PubkeyAuthentication yes

# Отключить аутентификацию по паролю (после подтверждения работы ключей!)
 PasswordAuthentication no

# Отключить вход root по SSH
 PermitRootLogin no

# Разрешить только определённых пользователей
AllowUsers alice bob deploy

# Поддерживать сеансы активными
ClientAliveInterval 60
ClientAliveCountMax 3
```

После редактирования перезагрузите SSH-демон:

```bash
sudo systemctl reload sshd
# Или в старых системах:
sudo service ssh reload
```

**Важно**: Не отключайте `PasswordAuthentication`, пока не убедитесь в работе аутентификации по ключу. Если вы заблокируете себя, потребуется консольный доступ (или экстренный доступ хостинг-провайдера) для восстановления.

### Настройка ssh-agent в Linux

Если ключ защищён парольной фразой, добавьте его в `ssh-agent` в начале сеанса, чтобы не вводить её повторно:

```bash
# Запустить агент (часто автоматически запускается средами рабочего стола)
eval "$(ssh-agent -s)"

# Добавить ключ (введите парольную фразу один раз)
ssh-add ~/.ssh/id_ed25519

# Проверить загруженные ключи
ssh-add -l
```

Для автоматического запуска `ssh-agent` и добавления ключа при входе добавьте в `~/.bashrc` или `~/.zshrc`:

```bash
# Автозапуск ssh-agent
if [ -z "$SSH_AUTH_SOCK" ]; then
  eval "$(ssh-agent -s)"
  ssh-add ~/.ssh/id_ed25519
fi
```

## Шаг 3: Настройка SSH в macOS

macOS включает полный OpenSSH-клиент. Процесс настройки аутентификации по ключу на удалённом сервере идентичен Linux (см. Шаг 2 выше). Специфика macOS касается интеграции с Keychain.

### Интеграция с Keychain

macOS может сохранить вашу парольную фразу в системной цепочке ключей, чтобы после первого ввода больше не запрашивать её:

```bash
# Добавить ключ в ssh-agent с интеграцией Keychain
ssh-add --apple-use-keychain ~/.ssh/id_ed25519

# Для старых версий macOS (до Monterey)
ssh-add -K ~/.ssh/id_ed25519
```

Чтобы macOS всегда загружала ключи из Keychain, добавьте в `~/.ssh/config`:

```
Host *
    AddKeysToAgent yes
    UseKeychain yes
    IdentityFile ~/.ssh/id_ed25519
```

После этого парольная фраза сохраняется в Keychain и загружается автоматически — при последующих загрузках запрашиваться не будет.

### Проверить версию SSH-клиента

```bash
ssh -V
# OpenSSH_9.x, LibreSSL x.x.x
```

macOS поставляется с достаточно свежей версией OpenSSH, но не всегда с самой последней. Если нужны новые функции, установите через Homebrew: `brew install openssh`.

## Шаг 4: Настройка SSH в Windows

Windows 10 (1809+) и Windows 11 включают OpenSSH. Вот как заставить его работать.

### Установить OpenSSH-клиент

**Проверить, установлен ли:**

```powershell
ssh -V
```

**Установить через Параметры:**
1. Откройте **Параметры → Приложения → Дополнительные возможности**
2. Найдите **Клиент OpenSSH**
3. Нажмите **Установить**

**Установить через PowerShell (от администратора):**

```powershell
Add-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0
```

### Сгенерировать ключи в Windows

Откройте PowerShell и выполните:

```powershell
ssh-keygen -t ed25519 -C "your@email.com"
```

Ключи сохраняются в `C:\Users\YourUsername\.ssh\`.

### Скопировать открытый ключ на Linux-сервер

Windows не включает `ssh-copy-id`, но это можно сделать вручную из PowerShell:

```powershell
# Отобразить открытый ключ
Get-Content ~/.ssh/id_ed25519.pub

# Скопировать на удалённый сервер (будет запрошен пароль сервера)
$pubkey = Get-Content ~/.ssh/id_ed25519.pub -Raw
ssh user@server.example.com "mkdir -p ~/.ssh && echo '$pubkey' >> ~/.ssh/authorized_keys && chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys"
```

Или с помощью `Add-Content` для передачи через конвейер:

```powershell
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh user@server.example.com "cat >> ~/.ssh/authorized_keys"
```

### Настройка ssh-agent в Windows

Чтобы не вводить парольную фразу повторно, запустите службу OpenSSH Authentication Agent:

```powershell
# Запустить службу
Start-Service ssh-agent

# Настроить автозапуск
Set-Service -Name ssh-agent -StartupType Automatic

# Добавить ключ
ssh-add ~/.ssh/id_ed25519
```

### PuTTY как альтернатива

Если предпочитаете графический SSH-клиент, PuTTY — бесплатный вариант для Windows. Используйте **PuTTYgen** для генерации ключей (сохраняйте в формате `.ppk` для PuTTY и экспортируйте в формат OpenSSH). Однако для новых установок встроенный OpenSSH-клиент проще и совместим с остальной экосистемой.

## Как настроить SSH-ключи для GitHub и GitLab

Аутентификация по SSH-ключу в GitHub и GitLab работает одинаково — вы загружаете открытый ключ в сервис и используете соответствующий приватный ключ при Git-операциях.

### Настройка SSH для GitHub

**1. Скопируйте открытый ключ:**

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows PowerShell
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

Или просто отобразите и скопируйте вручную:

```bash
cat ~/.ssh/id_ed25519.pub
```

**2. Добавьте в GitHub:**

1. Перейдите на **github.com** и войдите
2. Нажмите аватар → **Settings**
3. В левой панели: **SSH and GPG keys**
4. Нажмите **New SSH key**
5. Дайте понятное **Название** (например, «MacBook Pro» или «Рабочий ПК»)
6. Установите **Key type** в **Authentication Key**
7. Вставьте открытый ключ
8. Нажмите **Add SSH key**

**3. Проверьте соединение:**

```bash
ssh -T git@github.com
# Hi username! You've successfully authenticated, but GitHub does not provide shell access.
```

**4. Используйте SSH для Git-операций:**

```bash
# Клонировать по SSH вместо HTTPS
git clone git@github.com:username/repository.git

# Если существующий репозиторий использует HTTPS, переключите на SSH
git remote set-url origin git@github.com:username/repository.git
```

### Настройка SSH для GitLab

Процесс идентичен:

1. Перейдите на **gitlab.com** → Нажмите аватар → **Preferences**
2. В левой панели: **SSH Keys**
3. Вставьте открытый ключ, дайте название и при желании установите дату истечения
4. Нажмите **Add key**

Проверьте:

```bash
ssh -T git@gitlab.com
# Welcome to GitLab, @username!
```

Для self-hosted GitLab замените `gitlab.com` именем хоста вашего экземпляра.

### Использование отдельного ключа для GitHub и других сервисов

Если вам нужен отдельный ключ для GitHub (рекомендуется для гигиены ключей), сгенерируйте именованный ключ:

```bash
ssh-keygen -t ed25519 -C "github" -f ~/.ssh/id_ed25519_github
```

Затем добавьте блок `Host` в `~/.ssh/config`:

```
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/id_ed25519_github
    AddKeysToAgent yes
```

Теперь `git clone git@github.com:...` автоматически использует ключ, специфичный для GitHub.

## Тестирование и устранение неполадок SSH

### Базовая проверка подключения

```bash
ssh -v user@server.example.com
```

Флаг `-v` (подробный режим) показывает, что именно происходит на каждом шаге. Ищите строки вида:
- `Offering public key: ~/.ssh/id_ed25519` — SSH пробует ваш ключ
- `Server accepts key` — ключ есть в `authorized_keys`
- `Authenticated to server.example.com` — успех

### Распространённые ошибки и способы их устранения

**«Permission denied (publickey)»**

Самая распространённая ошибка. Причины и решения:

```bash
# Проверить, загружен ли ключ в ssh-agent
ssh-add -l

# Если ключи не перечислены, добавить ключ
ssh-add ~/.ssh/id_ed25519

# На удалённом сервере проверить authorized_keys
cat ~/.ssh/authorized_keys
# Убедиться, что ваш открытый ключ там есть

# Проверить права на удалённом сервере
ls -la ~/.ssh/
# ~/.ssh должна иметь права 700
# authorized_keys должна иметь права 600
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

# Проверить, разрешает ли сервер аутентификацию по открытому ключу
sudo grep PubkeyAuthentication /etc/ssh/sshd_config
# Должно быть: PubkeyAuthentication yes
```

**«WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED»**

Хост-ключ сервера изменился — SSH расценивает это как потенциальную угрозу безопасности. Если вы знаете, что это легитимно (сервер перестроен, IP переназначен), удалите старую запись:

```bash
# Удалить конкретную запись хоста
ssh-keygen -R server.example.com

# Или отредактировать known_hosts вручную и удалить затронутую строку
nano ~/.ssh/known_hosts
```

**«Bad permissions» или «Permissions too open»**

SSH строг в отношении прав доступа к файлам ключей:

```bash
# Исправить права приватного ключа
chmod 600 ~/.ssh/id_ed25519

# Исправить права открытого ключа
chmod 644 ~/.ssh/id_ed25519.pub

# Исправить права директории .ssh
chmod 700 ~/.ssh
```

В Windows права приватных ключей управляются иначе. Щёлкните правой кнопкой мыши по файлу ключа → Свойства → Безопасность → Дополнительно → Отключить наследование → Удалить все, кроме вашей учётной записи, предоставив полный контроль только себе.

### Настройка и использование ssh-agent

Если вас постоянно запрашивают парольную фразу, `ssh-agent` не запущен или ключ не добавлен:

```bash
# Запустить агент
eval "$(ssh-agent -s)"
# Agent pid 12345

# Добавить ключ
ssh-add ~/.ssh/id_ed25519
# Enter passphrase for /home/user/.ssh/id_ed25519:
# Identity added: /home/user/.ssh/id_ed25519

# Проверить
ssh-add -l
# 256 SHA256:abc123... your@email.com (ED25519)
```

### Предварительное заполнение known_hosts в автоматизации

В CI/CD-пайплайнах используйте `ssh-keyscan` для добавления отпечатка удалённого хоста перед подключением:

```bash
ssh-keyscan -H server.example.com >> ~/.ssh/known_hosts
ssh-keyscan -H github.com >> ~/.ssh/known_hosts
```

Это предотвращает интерактивные запросы отпечатка в автоматизированных скриптах.

## Часто задаваемые вопросы

### Как исправить «Permission denied (publickey)»?

Пройдите по этому чек-листу: (1) Убедитесь, что ключ есть в SSH-агенте: `ssh-add -l`. Если нет — выполните `ssh-add ~/.ssh/id_ed25519`. (2) Проверьте, что ваш открытый ключ есть в файле `~/.ssh/authorized_keys` на сервере. (3) Проверьте права: `~/.ssh` должна иметь права `700`, `authorized_keys` — `600`. (4) Убедитесь, что на сервере `PubkeyAuthentication yes` в `/etc/ssh/sshd_config`. (5) Используйте `ssh -vvv user@server`, чтобы увидеть, где именно происходит сбой аутентификации. Подробный вывод покажет, какие ключи перебираются и принял ли их сервер.

### Как добавить SSH-ключ в ssh-agent?

Выполните `ssh-add ~/.ssh/id_ed25519` (замените именем вашего файла ключа). Если агент не запущен, сначала запустите его с помощью `eval "$(ssh-agent -s)"`. В macOS используйте `ssh-add --apple-use-keychain ~/.ssh/id_ed25519` для сохранения парольной фразы в Keychain. В Windows включите службу `ssh-agent` с помощью `Start-Service ssh-agent`, затем выполните `ssh-add`. Запускать `ssh-add` нужно один раз за сеанс (или один раз навсегда с Keychain в macOS).

### Можно ли иметь несколько SSH-ключей?

Да, и это хорошая практика. Генерируйте отдельные ключи для разных целей — например, один для личного GitHub, один для рабочего GitLab, один для продакшн-серверов. Используйте `~/.ssh/config` для сопоставления каждого хоста с конкретным ключом через `IdentityFile`. Наличие отдельных ключей означает, что отзыв одного (удалением из сервиса или сервера) не влияет на остальные, ограничивая последствия компрометации ключа.

### Как сменить парольную фразу SSH-ключа?

Используйте `ssh-keygen -p`:

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

Вас попросят ввести текущую парольную фразу, затем задать и подтвердить новую. Сам ключ (и, соответственно, открытый ключ) не изменится — обновлять его нигде не нужно. Это полезно, если вы подозреваете компрометацию парольной фразы или хотите сменить её в рамках плановой ротации безопасности.

### Как удалить старый SSH-ключ с сервера?

Откройте `~/.ssh/authorized_keys` на сервере и удалите строку со старым открытым ключом:

```bash
# Отредактировать authorized_keys на удалённом сервере
ssh user@server.example.com
nano ~/.ssh/authorized_keys
# Удалить строку с ключом, который нужно убрать
# Сохранить и выйти
```

Каждая строка в `authorized_keys` — один открытый ключ. Комментарий ключа (последнее поле, обычно `user@hostname` или email) помогает определить, какой ключ есть какой. Если не знаете, какую строку удалить, сравните отпечатки:

```bash
# На локальной машине получить отпечаток
ssh-keygen -lf ~/.ssh/id_ed25519.pub

# На сервере проверить отпечатки всех авторизованных ключей
ssh-keygen -lf ~/.ssh/authorized_keys
```

Сопоставьте отпечатки, чтобы найти и удалить нужный ключ.
