# Что такое SSH? Полное руководство по Secure Shell

SSH, что расшифровывается как **Secure Shell**, — это криптографический сетевой протокол, позволяющий безопасно подключаться к удалённым компьютерам и управлять ими через незащищённую сеть. Входите ли вы на Linux-сервер, отправляете код в GitHub или передаёте файлы между машинами — почти наверняка SSH работает за кулисами. Это руководство объясняет всё, что нужно знать о SSH: что это такое, как он работает и почему стал универсальным стандартом безопасного удалённого доступа.

## Что означает аббревиатура SSH?

**SSH расшифровывается как Secure Shell** — «защищённая оболочка». Название точно описывает назначение протокола: он обеспечивает безопасный зашифрованный сеанс командной оболочки на удалённой машине. До появления SSH администраторы использовали такие инструменты, как Telnet и rsh (remote shell), для управления удалёнными серверами. Эти протоколы передавали всё в открытом тексте, включая имена пользователей и пароли, что делало их тривиально лёгкой целью для перехвата в любой общей сети.

SSH был создан финским учёным-компьютерщиком **Тату Илёненом** в 1995 году — непосредственно под влиянием атаки с перехватом паролей, которую он наблюдал в университетской сети. Он выпустил SSH-1 как свободное программное обеспечение, и оно было быстро принято по всему интернету. Впоследствии протокол был стандартизирован IETF в виде SSH-2 (RFC 4251–4254), который ввёл более стойкую криптографию и устранил ряд уязвимостей в исходной разработке.

Сегодня, когда говорят «SSH», почти всегда имеют в виду SSH-2. Наиболее широко используемая реализация — **OpenSSH**, проект с открытым исходным кодом, поддерживаемый командой OpenBSD и включённый в поставку практически всех дистрибутивов Linux и macOS.

На практике **смысл SSH** выходит далеко за рамки простого инструмента для входа в систему. Это полный набор протоколов, лежащий в основе:
- Безопасного удалённого доступа к терминалу
- Зашифрованной передачи файлов (SFTP и SCP)
- Аутентификации Git (GitHub, GitLab, Bitbucket)
- Безопасного перенаправления портов и туннелирования
- Автоматизированных пайплайнов развёртывания

## Как работает SSH

**Протокол Secure Shell** работает на основе многоуровневой архитектуры, которая раздельно обрабатывает шифрование, аутентификацию и мультиплексирование каналов. Понимание этих уровней помогает разобраться, почему SSH считается надёжным инструментом для критически важных операций.

### Три уровня протокола

**1. Транспортный уровень (RFC 4253)**

При инициации SSH-соединения клиент и сервер сначала выполняют рукопожатие для согласования алгоритмов шифрования и установки общего секрета. Используется алгоритм обмена ключами — современный SSH по умолчанию применяет обмен ключами на основе Curve25519. После рукопожатия всё последующее взаимодействие шифруется. Транспортный уровень также обеспечивает целостность данных с помощью кодов аутентификации сообщений (MAC), гарантируя, что данные не могут быть подделаны при передаче.

**2. Уровень аутентификации пользователя (RFC 4252)**

После установки зашифрованного канала сервер запрашивает у клиента аутентификацию. SSH поддерживает несколько методов:

- **Аутентификация по открытому ключу** — клиент доказывает владение приватным ключом, никогда не передавая его по сети. Это наиболее безопасный и распространённый метод.
- **Аутентификация по паролю** — пароль пользователя передаётся по зашифрованному каналу. Удобнее, но уязвим к атакам методом перебора.
- **Клавиатурно-интерактивный метод** — механизм «вызов-ответ», часто используемый для многофакторной аутентификации.
- **GSSAPI/Kerberos** — применяется в корпоративных средах с централизованным управлением идентификацией.

**3. Уровень соединения (RFC 4254)**

После аутентификации уровень соединения мультиплексирует одно зашифрованное соединение в несколько логических каналов. Именно это позволяет одновременно поддерживать интерактивную оболочку, выполнять удалённые команды, перенаправлять порты и передавать файлы — всё через одно TCP-соединение.

### Обмен ключами в деталях

Когда вы вводите `ssh user@example.com`, вот что происходит в первые несколько миллисекунд:

1. Клиент подключается к TCP-порту 22 на сервере.
2. Обе стороны обмениваются строками версии протокола.
3. Обе стороны отправляют списки поддерживаемых алгоритмов (обмен ключами, шифры, MAC, сжатие).
4. Выполняется обмен ключами по Диффи-Хеллману или ECDH для получения общего сессионного ключа — именно поэтому, даже если кто-то записывает зашифрованный трафик, он не сможет расшифровать его позднее.
5. Сервер отправляет свой **хост-ключ** (открытый ключ), который клиент проверяет по файлу `~/.ssh/known_hosts`.
6. Проводится аутентификация, как описано выше.
7. Сеанс начинается.

## SSH и Telnet: почему победил SSH

До SSH протокол **Telnet** был стандартом для удалённого доступа к терминалу. Telnet прост и до сих пор работает, но у него есть один фатальный недостаток: всё передаётся в открытом тексте. Ваше имя пользователя, пароль, каждая введённая команда и каждый фрагмент вывода — всё это видно любому, кто может перехватить сетевой трафик.

В 1990-е годы, когда интернет-магистраль была доверенными академическими и исследовательскими сетями, это не считалось серьёзной проблемой. По мере коммерциализации интернета такие инструменты, как Dsniff и Ettercap, сделали тривиально простым перехват учётных данных Telnet в локальных сетях.

SSH решил эту проблему, зашифровав всё. Вот прямое сравнение:

| Характеристика | Telnet | SSH |
|---|---|---|
| Шифрование | Нет | Стойкое (AES, ChaCha20) |
| Аутентификация | Пароль в открытом виде | Открытый ключ + зашифрованный пароль |
| Целостность данных | Нет | Проверка HMAC |
| Порт | 23 | 22 |
| Рекомендуется сегодня | Нет | Да |

rsh (remote shell) и rlogin имели аналогичные проблемы. Они также полагались на доверие по IP-адресу, которое тривиально подделать. SSH заменил все эти протоколы и остаётся непревзойдённым стандартом уже более 25 лет.

## Что такое SSH-ключ?

SSH поддерживает два основных метода аутентификации: пароли и **SSH-ключи**. SSH-ключи — это криптографические пары ключей: **открытый ключ** и **приватный ключ**, — основанные на асимметричной криптографии.

**Приватный ключ** хранится на вашей локальной машине, защищённый опциональной парольной фразой. **Открытый ключ** размещается на любом сервере, к которому вы хотите получить доступ, в файле `~/.ssh/authorized_keys`. При подключении сервер выдаёт криптографическую задачу, на которую может ответить только владелец соответствующего приватного ключа. По сети не передаётся никакого пароля.

Такой подход имеет ряд преимуществ:
- **Защита от брутфорса** — угадывать нечего. 256-битный ключ Ed25519 имеет 2²⁵⁶ возможных значений.
- **Нет пароля для кражи** — даже если кто-то перехватит рукопожатие, он не получит ничего полезного.
- **Автоматизируемость** — скрипты и CI/CD-пайплайны могут проходить аутентификацию без участия человека.
- **Отзываемость** — удаление открытого ключа из `authorized_keys` немедленно отзывает доступ.

Наиболее современный и рекомендуемый тип ключей — **Ed25519**, использующий эллиптическую криптографию на кривой Эдвардса. RSA-ключи также широко поддерживаются, однако следует использовать не менее 4096 бит.

Подробное погружение в тему пар ключей — в нашем руководстве <a href="/ru/what-is-an-ssh-key/">что такое SSH-ключ</a>.

## Типичные сценарии использования SSH

SSH применяется в удивительно широком спектре задач. Вот наиболее важные из них.

### Удалённое администрирование серверов

Самый фундаментальный сценарий использования. Системные администраторы входят на серверы Linux и Unix для установки программного обеспечения, редактирования конфигурационных файлов, управления службами и диагностики проблем — всё через SSH-сеанс терминала.

```bash
ssh user@192.168.1.100
ssh admin@myserver.example.com
ssh -p 2222 deploy@staging.example.com
```

### Передача файлов через SFTP и SCP

SSH предоставляет два протокола передачи файлов, построенных на основе одного зашифрованного канала:

**SCP (Secure Copy Protocol)** — простой инструмент командной строки для копирования файлов:

```bash
# Копировать локальный файл на удалённый сервер
scp report.pdf user@server:/home/user/documents/

# Копировать удалённый файл на локальную машину
scp user@server:/var/log/app.log ./local-copy.log

# Копировать директорию рекурсивно
scp -r ./project/ user@server:/var/www/
```

**SFTP (SSH File Transfer Protocol)** предоставляет интерактивный интерфейс в стиле FTP с более широким набором возможностей, чем SCP, включая возобновляемые передачи и листинг директорий.

### Аутентификация Git

Когда вы добавляете SSH-ключ в GitHub, GitLab или Bitbucket, каждая операция `git clone`, `git push` и `git pull` по SSH аутентифицируется с помощью вашего приватного ключа. Это избавляет от необходимости вводить имя пользователя и пароль при каждой Git-операции и является рекомендуемым методом аутентификации для разработчиков.

```bash
# Клонировать репозиторий по SSH
git clone git@github.com:username/repository.git

# Проверить SSH-соединение с GitHub
ssh -T git@github.com
```

### Автоматизированное развёртывание и CI/CD

Скрипты развёртывания и системы CI/CD — GitHub Actions, GitLab CI, Jenkins — используют SSH-ключи для деплоя кода на продакшн-серверы без участия человека. Приватный ключ хранится как секрет в системе CI, а соответствующий открытый ключ добавляется в файл `authorized_keys` на сервере.

### Перенаправление портов для баз данных и приложений

SSH-туннелирование позволяет безопасно открыть доступ к удалённым сервисам на вашей локальной машине. Например, для доступа к удалённой базе данных MySQL, не открытой в интернет:

```bash
ssh -L 3307:localhost:3306 user@dbserver.example.com -N
```

Это создаёт локальный туннель: подключение к порту 3307 на вашей машине прозрачно перенаправляет трафик на порт 3306 удалённого сервера, зашифровав его в SSH-туннеле.

## Порт SSH и параметры конфигурации

### Стандартный порт 22

SSH по умолчанию прослушивает **TCP-порт 22**. Это хорошо известный порт, назначенный IANA и зарезервированный для SSH с момента создания протокола. Когда вы запускаете `ssh user@host`, не указывая порт, SSH-клиент подключается к порту 22.

Некоторые администраторы меняют порт SSH (например, на 2222 или 8022) как незначительную меру безопасности для снижения шума от автоматического сканирования в логах. Это называется «безопасностью через неизвестность» и не заменяет надлежащую аутентификацию по ключу, но может уменьшить засорение логов.

```bash
# Подключиться к нестандартному SSH-порту
ssh -p 2222 user@example.com
```

### Файл конфигурации SSH-клиента

Вместо того чтобы каждый раз вводить длинные параметры командной строки, вы можете настроить SSH-соединения в файле `~/.ssh/config`. Этот файл позволяет определять псевдонимы, устанавливать параметры для каждого хоста и значительно упрощать рабочий процесс.

```
Host myserver
    HostName 203.0.113.10
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_prod

Host bastion
    HostName jump.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host internal
    HostName 10.0.1.50
    User ubuntu
    ProxyJump bastion
```

При такой конфигурации `ssh myserver` подключается к полному имени хоста на нестандартном порту с использованием указанного ключа — без каких-либо флагов.

Полное справочное руководство по параметрам и флагам команды SSH — в нашем <a href="/ru/ssh-command/">руководстве по команде SSH</a>.

### Конфигурация на стороне сервера (sshd_config)

Демон SSH-сервера (sshd) настраивается через `/etc/ssh/sshd_config`. Ключевые настройки:

```
# Отключить аутентификацию по паролю — только ключи
 PasswordAuthentication no

# Отключить вход под root
 PermitRootLogin no

# Изменить порт прослушивания
Port 2222

# Ограничить доступ определёнными пользователями
AllowUsers deploy admin

# Включить аутентификацию по открытому ключу
PubkeyAuthentication yes
```

После редактирования перезагрузите демон: `sudo systemctl reload sshd`

## SSH-туннелирование

SSH-туннелирование (также называемое перенаправлением портов SSH) позволяет маршрутизировать произвольный TCP-трафик через зашифрованное SSH-соединение. Это одна из наиболее мощных и недооценённых возможностей SSH.

### Локальное перенаправление портов

Локальное перенаправление делает удалённый сервис доступным на вашей локальной машине. Синтаксис: `-L локальный_порт:удалённый_хост:удалённый_порт`.

```bash
# Получить локальный доступ к удалённой базе данных PostgreSQL
ssh -L 5433:localhost:5432 user@dbserver -N

# Получить доступ к внутреннему веб-приложению через промежуточный сервер
ssh -L 8080:internal-app.company.com:80 user@jump.company.com -N
```

Флаг `-N` указывает SSH не выполнять удалённую команду — полезно, когда нужен только туннель.

### Удалённое перенаправление портов

Удалённое перенаправление открывает локальный сервис на удалённом сервере. Синтаксис: `-R удалённый_порт:локальный_хост:локальный_порт`.

```bash
# Открыть доступ к локальному веб-серверу на удалённой машине
ssh -R 9090:localhost:3000 user@server -N
```

Это полезно для временного предоставления коллеге, имеющему доступ к удалённому хосту, возможности просмотреть ваш локальный сервер разработки или для демонстрации клиенту работы в процессе.

### Динамическое перенаправление портов (SOCKS-прокси)

Флаг `-D` создаёт SOCKS-прокси на вашей локальной машине, маршрутизируя весь совместимый с SOCKS трафик через SSH-соединение. Фактически это превращает удалённый сервер в выходной узел VPN.

```bash
# Создать SOCKS5-прокси на локальном порту 1080
ssh -D 1080 user@remote-server -N -f
```

Настройте браузер или систему на использование `SOCKS5-прокси: 127.0.0.1:1080`, и весь веб-трафик будет маршрутизироваться через удалённый сервер. Часто используется для доступа к геоблокированному контенту или безопасного сёрфинга из ненадёжной сети.

## Часто задаваемые вопросы

### Что означает SSH?

SSH расшифровывается как **Secure Shell** («защищённая оболочка»). Это криптографический сетевой протокол для безопасного выполнения сетевых служб через незащищённую сеть. Название отражает его основное назначение: обеспечение безопасного интерактивного сеанса оболочки на удалённом компьютере.

### SSH и SSL — это одно и то же?

Нет. SSH и SSL/TLS — оба криптографические протоколы, но они служат разным целям. **SSL/TLS** (Secure Sockets Layer / Transport Layer Security) используется для защиты веб-трафика (HTTPS), электронной почты и других прикладных протоколов. **SSH** специально разработан для безопасного удалённого доступа к оболочке, передачи файлов и туннелирования. Они используют схожие криптографические примитивы (криптография с открытым ключом, симметричное шифрование), но имеют разные механизмы рукопожатия и области применения. SSH не использует SSL/TLS внутри.

### Какой порт использует SSH?

SSH по умолчанию использует **TCP-порт 22**. Это хорошо известный порт, назначенный IANA. Вы можете настроить SSH-сервер на прослушивание другого порта с помощью директивы `Port` в `/etc/ssh/sshd_config` и подключаться к нестандартным портам с помощью `ssh -p ПОРТ user@host`.

### Можно ли взломать SSH?

Правильно настроенный SSH-сервер с аутентификацией по ключу крайне сложно скомпрометировать. Криптография — ключи Ed25519, шифрование AES-256, MAC SHA-2 — считается квантово-устойчивой для аутентификации по ключу и вычислительно невозможной для брутфорса. Реальные компрометации SSH обычно связаны с: кражей файлов приватных ключей, слабыми парольными фразами, неправильно настроенными серверами, разрешающими аутентификацию по паролю со слабыми паролями, уязвимыми версиями программного обеспечения с неустановленными патчами CVE, или социальной инженерией, а не криптографическими атаками. Обновляйте сервер, отключайте аутентификацию по паролю и защищайте файл приватного ключа.

### В чём разница между SSH и HTTPS?

Оба протокола шифруют сетевой трафик, но предназначены для разных целей. **HTTPS** защищает веб-браузинг — использует TLS для шифрования HTTP-запросов и ответов между браузерами и веб-серверами. **SSH** защищает удалённый административный доступ, передачу файлов и туннелирование. При отправке кода в GitHub можно использовать как SSH (порт 22, аутентификация по ключу), так и HTTPS (порт 443, аутентификация по токену). Разработчики, как правило, предпочитают SSH, поскольку он не требует повторного ввода учётных данных. HTTPS универсально поддерживается через межсетевые экраны, которые могут блокировать порт 22.
