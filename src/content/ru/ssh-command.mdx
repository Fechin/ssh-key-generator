# Команда SSH: полный справочник с примерами для Linux, Mac и Windows

Команда `ssh` — основной инструмент для установки SSH-соединений из командной строки. Входите ли вы на удалённый Linux-сервер, перенаправляете порт для доступа к базе данных или прыгаете через бастионный хост во внутреннюю сеть — команда `ssh` является вашей точкой входа. В этом руководстве рассматривается всё: от базового синтаксиса до продвинутых флагов, туннелирования и конфигурации ProxyJump — с реальными примерами для Linux, macOS и Windows.

Перед выполнением SSH-команд вам потребуются SSH-ключи. См. наше руководство <a href="/ru/what-is-an-ssh-key/">что такое SSH-ключ</a> или воспользуйтесь <a href="/">бесплатным онлайн-генератором</a> для создания пары ключей.

## Синтаксис команды SSH

Базовый синтаксис команды `ssh`:

```
ssh [параметры] [пользователь@]хост [команда]
```

Разберём по частям:

- **`[параметры]`** — ноль или более флагов, изменяющих поведение (рассматриваются по всему руководству)
- **`[пользователь@]`** — необязательное имя пользователя. Если опущено, SSH использует текущее локальное имя пользователя
- **`хост`** — удалённый хост для подключения. Может быть IP-адресом, доменным именем или псевдонимом, определённым в `~/.ssh/config`
- **`[команда]`** — необязательная команда для выполнения на удалённом хосте. Если опущена, SSH запускает интерактивный сеанс оболочки

**Базовые примеры:**

```bash
# Подключиться с текущим именем пользователя
ssh server.example.com

# Подключиться как конкретный пользователь
ssh alice@server.example.com

# Подключиться по IP-адресу
ssh ubuntu@192.168.1.100

# Выполнить одну команду и немедленно отключиться
ssh user@server.example.com "df -h"

# Подключиться и оставаться в соединении
ssh user@server.example.com
```

## SSH в Linux и macOS

Дистрибутивы Linux и macOS поставляются с предустановленным **OpenSSH**. В Linux он входит в пакет `openssh-client` (Debian/Ubuntu) или `openssh-clients` (RHEL/CentOS/Fedora). В macOS он включён в систему и доступен сразу из Terminal или iTerm2.

### Проверить наличие SSH

```bash
ssh -V
# Вывод: OpenSSH_9.6p1, LibreSSL 3.3.6
```

### Базовое подключение

```bash
# Подключиться к серверу
ssh user@myserver.example.com

# SSH проверяет ~/.ssh/known_hosts при первом подключении и спрашивает:
# The authenticity of host 'myserver.example.com' can't be established.
# ED25519 key fingerprint is SHA256:abc123...
# Are you sure you want to continue connecting (yes/no)? yes

# Отпечаток хоста сохраняется в ~/.ssh/known_hosts
```

### Использование конфигурационного файла SSH для удобства

Вместо того чтобы каждый раз вводить полные данные подключения, создайте `~/.ssh/config`:

```
Host myserver
    HostName myserver.example.com
    User alice
    Port 22
    IdentityFile ~/.ssh/id_ed25519

Host dev
    HostName dev.example.com
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_work
```

Теперь `ssh myserver` разворачивается в полное соединение — никаких флагов не нужно.

## SSH в Windows (PowerShell и OpenSSH)

Windows 10 (версия 1809) и Windows 11 включают **OpenSSH-клиент** как дополнительный компонент, делая команду `ssh` доступной в PowerShell, Командной строке и Windows Terminal.

### Проверить, установлен ли OpenSSH

```powershell
# В PowerShell
ssh -V
```

Если команда не найдена, установите её:

**Через Параметры:**
1. Откройте **Параметры → Приложения → Дополнительные возможности**
2. Нажмите **Добавить компонент**
3. Найдите **Клиент OpenSSH** и установите

**Через PowerShell (от администратора):**

```powershell
Add-WindowsCapability -Online -Name OpenSSH.Client~~~~0.0.1.0
```

### Использование SSH в PowerShell

После установки синтаксис идентичен Linux и macOS:

```powershell
# Подключиться к серверу
ssh user@server.example.com

# Подключиться на нестандартном порту
ssh -p 2222 user@server.example.com

# Использовать конкретный файл ключа
ssh -i C:\Users\YourName\.ssh\id_ed25519 user@server.example.com
```

### Расположение SSH-ключей в Windows

SSH-ключи в Windows хранятся в `C:\Users\YourUsername\.ssh\`. PowerShell распознаёт `~` как сокращение:

```powershell
# Эти команды эквивалентны
ssh -i ~/.ssh/id_ed25519 user@server
ssh -i C:\Users\YourUsername\.ssh\id_ed25519 user@server
```

### PuTTY и OpenSSH в Windows

До появления OpenSSH в Windows стандартным SSH-клиентом был **PuTTY**. PuTTY использует собственный формат ключей (файлы `.ppk`) и собственный генератор ключей (PuTTYgen). Если у вас есть ключ `.ppk` и нужно конвертировать его в стандартный формат OpenSSH:

```bash
# Используя инструменты PuTTY
puttygen mykey.ppk -O private-openssh -o id_ed25519
```

Для новых установок отдавайте предпочтение встроенному OpenSSH-клиенту в Windows — он поддерживается проектом OpenSSH и использует тот же формат ключей и конфигурационный файл, что и Linux и macOS.

## SSH с аутентификацией по ключу: флаг -i

По умолчанию SSH ищет файлы ключей в `~/.ssh/` со стандартными именами (`id_ed25519`, `id_rsa`, `id_ecdsa`, `id_dsa`). **Флаг `-i`** (файл идентификатора) позволяет явно указать другой файл ключа.

```bash
# Использовать конкретный файл ключа
ssh -i ~/.ssh/id_ed25519_production user@prod.example.com

# Использовать ключ, хранящийся в нестандартном расположении
ssh -i /path/to/my/custom_key deploy@server.example.com

# Комбинировать с другими параметрами
ssh -i ~/.ssh/deploy_key -p 2222 -l deploy server.example.com
```

Флаг `-i` необходим в нескольких ситуациях:
- У вас несколько пар ключей для разных серверов или целей
- Ключ не находится в директории `~/.ssh/` по умолчанию
- Вы указываете ключ в среде CI/CD, где путь задан явно
- Вы загрузили файл `.pem` из AWS или другого облачного провайдера

### Пример AWS EC2

AWS предоставляет файл ключа `.pem` при запуске экземпляра EC2:

```bash
# Подключиться к экземпляру AWS EC2 с ключом .pem
ssh -i ~/Downloads/my-aws-key.pem ec2-user@54.123.456.789

# Исправить права при необходимости (обязательно — SSH отклоняет ключи с открытыми правами)
chmod 400 ~/Downloads/my-aws-key.pem
```

## Основные параметры и флаги команды SSH

Команда `ssh` имеет десятки параметров. Вот те, которые вам действительно понадобятся.

| Флаг | Описание | Пример |
|---|---|---|
| `-p ПОРТ` | Подключиться на нестандартном порту | `ssh -p 2222 user@host` |
| `-i ФАЙЛ` | Использовать конкретный файл идентификатора (ключа) | `ssh -i ~/.ssh/mykey user@host` |
| `-l ПОЛЬЗОВАТЕЛЬ` | Указать имя пользователя для входа (альтернатива `user@`) | `ssh -l alice host` |
| `-L локальный:хост:удалённый` | Локальное перенаправление портов | `ssh -L 8080:localhost:80 host` |
| `-R удалённый:хост:локальный` | Удалённое перенаправление портов | `ssh -R 9090:localhost:3000 host` |
| `-D порт` | Динамический (SOCKS) прокси | `ssh -D 1080 host` |
| `-N` | Не выполнять удалённую команду (только туннелирование) | `ssh -N -L 3307:db:3306 host` |
| `-f` | Перейти в фоновый режим после аутентификации | `ssh -f -N -D 1080 host` |
| `-v` | Подробный (отладочный) вывод | `ssh -v user@host` |
| `-vv` или `-vvv` | Ещё более подробный вывод | `ssh -vvv user@host` |
| `-A` | Включить перенаправление SSH-агента | `ssh -A user@jump-host` |
| `-X` | Включить перенаправление X11 | `ssh -X user@host xclock` |
| `-C` | Включить сжатие | `ssh -C user@host` |
| `-T` | Отключить выделение псевдотерминала | `ssh -T git@github.com` |
| `-q` | Тихий режим (подавить предупреждения) | `ssh -q user@host` |
| `-o ПАРАМЕТР=ЗНАЧЕНИЕ` | Задать параметр конфигурации inline | `ssh -o StrictHostKeyChecking=no host` |
| `-J ПРЫЖОК` | ProxyJump через бастионный хост | `ssh -J jump.host user@internal` |
| `-W хост:порт` | Перенаправить stdio на хост:порт (для ProxyJump) | Используется внутри `-J` |

### Подробный режим для отладки

Когда SSH не работает, `-v` (или `-vv` / `-vvv`) показывает, что именно происходит:

```bash
ssh -v user@server.example.com
# Показывает: какие ключи перебираются, какие методы аутентификации принимает сервер,
# задействован ли агент и где происходит сбой подключения

ssh -vvv user@server.example.com
# Максимальная детализация — показывает подробное криптографическое согласование
```

### Отключение проверки хост-ключа (используйте с осторожностью)

В средах автоматизации и тестирования, где проверка хост-ключа неудобна:

```bash
ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null user@host
```

**Не используйте это в продакшне.** Это отключает защиту от атак «человек посередине». Вместо этого предварительно заполняйте `known_hosts` с помощью `ssh-keyscan` в скриптах подготовки.

### Перенаправление агента

```bash
# Перенаправить локальный SSH-агент на удалённый сервер
ssh -A user@jump-server

# С jump-server теперь можно SSH-подключаться к внутренним хостам с локальными ключами
# Не нужно копировать приватные ключи на jump-сервер
ssh internal-server.private
```

Перенаправление агента полезно, но имеет последствия для безопасности: любой, имеющий root-доступ на промежуточном сервере, может использовать перенаправленный агент. Используйте его только с доверенными серверами и предпочтительно применяйте **ProxyJump** (он не раскрывает агент промежуточным хостам).

## Команды SSH-туннелирования

SSH-туннелирование маршрутизирует TCP-соединения через зашифрованный SSH-канал. Общие сведения о туннелировании — в <a href="/ru/what-is-ssh/">разделе о туннелировании в нашем SSH-руководстве</a>.

### Локальное перенаправление портов

Делает удалённый сервис доступным на вашей локальной машине.

```bash
# Получить доступ к удалённой MySQL-базе данных на локальном порту 3307
ssh -L 3307:localhost:3306 user@db-server.example.com -N

# Подключиться к удалённому веб-серверу, доступному только с SSH-хоста
ssh -L 8080:intranet.company.com:80 user@vpn.company.com -N

# Получить доступ к экземпляру Redis в частной сети
ssh -L 6380:10.0.1.50:6379 user@bastion.example.com -N
```

После выполнения любой из приведённых команд подключайте локальное приложение к `127.0.0.1:ЛОКАЛЬНЫЙ_ПОРТ`.

### Удалённое перенаправление портов

Открывает локальный сервис на удалённом сервере.

```bash
# Открыть локальный сервер разработки (порт 3000) на порту 9090 удалённого сервера
ssh -R 9090:localhost:3000 user@remote.example.com -N

# Разрешить любому интерфейсу на удалённом сервере подключаться (требует GatewayPorts yes в sshd_config)
ssh -R 0.0.0.0:9090:localhost:3000 user@remote.example.com -N
```

### Динамическое перенаправление портов (SOCKS-прокси)

```bash
# Создать SOCKS5-прокси на локальном порту 1080, работающий в фоне
ssh -D 1080 -N -f user@remote.example.com

# Теперь настройте браузер или систему на использование SOCKS5-прокси: 127.0.0.1:1080
# Весь трафик маршрутизируется через удалённый сервер
```

Для использования с curl:

```bash
curl --socks5 127.0.0.1:1080 https://api.example.com/
```

### Запуск туннеля в фоновом режиме

```bash
# -f отправляет SSH в фоновый режим, -N означает без удалённой команды
ssh -f -N -L 3307:localhost:3306 user@db-server

# Найти фоновый SSH-процесс
ps aux | grep ssh

# Завершить его
kill PID
```

## SSH ProxyJump и бастионные хосты

Во многих организациях продакшн-серверы находятся в частных сетях, напрямую недоступных из интернета. **Бастионный хост** (или jump-хост) — это защищённый сервер на периметре сети, в который вы сначала заходите по SSH, а затем перепрыгиваете на внутренние серверы. Флаг `ssh -J` (ProxyJump, добавленный в OpenSSH 7.3) делает это бесшовным.

### Базовый ProxyJump

```bash
# Прыгнуть через один бастионный хост для доступа к внутреннему серверу
ssh -J ec2-user@bastion.example.com ubuntu@10.0.1.50

# Краткая форма
ssh -J user@jump-host user@internal-server
```

Это создаёт два SSH-соединения: сначала к бастиону, затем с бастиона к внутреннему серверу — всё в одной команде, со сквозным шифрованием. Ваш локальный SSH-агент обрабатывает аутентификацию на обоих хостах.

### Цепочка из нескольких прыжков

```bash
# Прыгнуть через два промежуточных хоста
ssh -J user@jump1.example.com,user@jump2.example.com user@final-destination
```

### ProxyJump в файле конфигурации SSH

Для часто используемых серверов внесите конфигурацию ProxyJump в `~/.ssh/config`:

```
Host bastion
    HostName bastion.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host prod-web
    HostName 10.0.1.100
    User ubuntu
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519_aws

Host prod-db
    HostName 10.0.2.50
    User postgres
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519_aws
```

Теперь `ssh prod-web` автоматически подключается через бастион. Ручные флаги `-J` не нужны.

### ProxyJump против перенаправления агента

ProxyJump предпочтительнее перенаправления агента (`-A`), потому что:
- Приватный ключ никогда не передаётся промежуточному хосту
- Бастионный хост не может использовать ваши учётные данные для аутентификации где-либо ещё
- Соединение аутентифицировано сквозным образом — сервер назначения напрямую проверяет вас

Подробности настройки — в нашем полном руководстве <a href="/ru/how-to-set-up-ssh/">как настроить SSH</a>.

## Часто задаваемые вопросы

### Как выполнить SSH-команду без пароля?

Настройте аутентификацию по SSH-ключу. Сгенерируйте пару ключей (`ssh-keygen -t ed25519`), скопируйте открытый ключ в файл `~/.ssh/authorized_keys` на сервере с помощью `ssh-copy-id user@host`, и последующие подключения будут использовать ключ без запроса пароля. Если ключ защищён парольной фразой, используйте `ssh-agent` и `ssh-add` для её кэширования — вводить придётся только один раз за сеанс.

### Как завершить SSH-сеанс?

Введите `exit` или нажмите **Ctrl+D**, чтобы закрыть удалённую оболочку и отключиться. Если сеанс завис (например, сеть прервалась), используйте escape-последовательность SSH: нажмите **Enter**, затем введите `~.` (тильда и точка). SSH немедленно завершит соединение. Другие escape-последовательности: `~^Z` переводит SSH в фоновый режим, `~?` показывает все escape-последовательности.

### Как выполнить команду на удалённом сервере через SSH?

Добавьте команду после имени хоста:

```bash
ssh user@server "command to run"
ssh user@server "ls -la /var/log/"
ssh user@server "sudo systemctl status nginx"
```

Для многострочных команд или скриптов используйте heredoc:

```bash
ssh user@server << 'EOF'
cd /var/www/html
git pull origin main
systemctl reload nginx
EOF
```

Для интерактивных команд (требующих терминала) добавьте флаг `-t` для выделения псевдотерминала: `ssh -t user@server sudo apt upgrade`.

### В чём разница между ssh и scp?

`ssh` открывает сеанс терминала (или выполняет одну команду) на удалённой машине. `scp` (Secure Copy Protocol) копирует файлы между вашей локальной машиной и удалённой, или между двумя удалёнными машинами, используя SSH для шифрования и аутентификации. Оба используют одинаковый протокол SSH и инфраструктуру ключей. Для передачи файлов `scp` — специализированный инструмент:

```bash
# scp с локального на удалённый
scp file.txt user@server:/destination/path/

# scp с удалённого на локальный
scp user@server:/remote/file.txt ./local-copy.txt
```

Для более продвинутой передачи файлов (рекурсивной, возобновляемой, с навигацией по директориям) используйте `sftp`. Для крупных передач `rsync -avz -e ssh` ещё эффективнее.

### Как использовать SSH с файлом конфигурации?

Создайте `~/.ssh/config` и добавьте записи хостов. Каждый блок `Host` определяет псевдоним подключения с полными настройками:

```
Host alias-name
    HostName actual.server.com
    User username
    Port 22
    IdentityFile ~/.ssh/specific_key
```

Затем `ssh alias-name` автоматически использует все заданные настройки. Файл конфигурации поддерживает wildcards: `Host *.example.com` применяет настройки ко всем совпадающим хостам. `Host *` в конце применяет настройки по умолчанию ко всем соединениям. Параметры в конкретном блоке `Host` имеют приоритет над глобальными значениями по умолчанию.
