# Что такое SSH-ключ? Открытые ключи, приватные ключи и пары ключей

**SSH-ключ** — это криптографическое удостоверение для аутентификации на SSH-серверах без пароля. SSH-ключи поставляются парами: **открытый ключ** и **приватный ключ**, — и вместе они делают удалённый доступ более безопасным и удобным, чем традиционная аутентификация по паролю. В этом руководстве объясняется, что такое SSH-ключ, как работает система пар ключей, какие типы ключей выбирать и как использовать SSH-ключи с такими сервисами, как GitHub и GitLab.

## Как работают SSH-ключи

SSH-ключи основаны на **асимметричной криптографии**, также называемой криптографией с открытым ключом. В отличие от симметричной криптографии (где один секретный ключ и шифрует, и дешифрует данные), асимметричные системы используют математически связанные пары ключей, где данные, зашифрованные одним ключом, можно расшифровать только другим.

Вот как выглядит процесс аутентификации при входе на сервер с помощью SSH-ключа:

1. Вы выполняете `ssh user@server.example.com`.
2. Сервер ищет ваш открытый ключ в `~/.ssh/authorized_keys`.
3. Сервер генерирует случайную задачу и шифрует её вашим открытым ключом.
4. Ваш SSH-клиент расшифровывает задачу с помощью приватного ключа.
5. Клиент отправляет серверу криптографическое доказательство (полученное из расшифрованной задачи и идентификатора сессии).
6. Сервер проверяет доказательство. Только владелец приватного ключа мог его создать.
7. Аутентификация прошла успешно. Ни один пароль не покидает вашу машину.

Ключевой момент: **приватный ключ никогда не передаётся по сети**. Сервер видит только ваш открытый ключ и криптографические доказательства, которые вы создаёте. Даже если кто-то перехватит каждый пакет вашего SSH-сеанса, он не получит ничего, что поможет ему выдать себя за вас.

Общие сведения о протоколе SSH — в нашем руководстве <a href="/ru/what-is-ssh/">что такое SSH</a>.

## Открытый ключ и приватный ключ: в чём разница?

Понимание различия между двумя ключами необходимо для правильного использования SSH.

### Открытый ключ — ваш замок

**Открытый ключ** предназначен для свободного распространения. Думайте о нём как о навесном замке: вы раздаёте копии всем, чьи системы хотите использовать для входа. Когда сервер хранит ваш открытый ключ в файле `authorized_keys`, он может проверить вашу личность, не зная приватного ключа.

Открытые ключи:
- Безопасно передавать — их нельзя использовать для вашей имперсонации
- Размещаются на серверах в `~/.ssh/authorized_keys`
- Загружаются в GitHub, GitLab и другие сервисы
- Обычно имеют расширение `.pub` (например, `id_ed25519.pub`)
- Выглядят так: `ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... user@hostname`

### Приватный ключ — ваш ключ

**Приватный ключ** — секретная половина пары. Он никогда не должен покидать вашу машину и не должен передаваться кому-либо — ни администраторам, ни облачным провайдерам, ни сотрудникам службы поддержки. Если ваш приватный ключ скомпрометирован, любой, кто его получит, может выдать себя за вас на каждом сервере, где установлен ваш открытый ключ.

Приватные ключи:
- Должны храниться в тайне и безопасности
- Хранятся локально в `~/.ssh/` с ограниченными правами доступа (600)
- Опционально шифруются парольной фразой
- Не имеют расширения или имеют специфическое расширение (например, `id_ed25519`)

### Файл `authorized_keys`

На стороне сервера открытые ключи каждого пользователя хранятся в `~/.ssh/authorized_keys`. Это текстовый файл, где каждая строка содержит один открытый ключ. При попытке аутентификации SSH-клиента сервер проверяет, может ли клиент доказать владение приватным ключом, соответствующим любой записи в этом файле.

```
# Формат ~/.ssh/authorized_keys
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGq... alice@laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHr... alice@workstation
ssh-rsa AAAAB3NzaC1yc2EAAAA... backup-deploy-key
```

В этом файле можно хранить столько открытых ключей, сколько нужно, позволяя нескольким устройствам или системам автоматизации аутентифицироваться на одном сервере.

## Алгоритмы SSH-ключей: Ed25519 против RSA

При генерации пары SSH-ключей вы выбираете алгоритм. Два наиболее распространённых варианта сегодня — **Ed25519** и **RSA**. Существуют также ECDSA и DSA, но DSA устарел, а ECDSA имеет подводные камни в реализации, поэтому сосредоточимся на двух рекомендуемых вариантах.

### Ed25519 — современный выбор

Ed25519 был введён в OpenSSH в версии 6.5 (2014) и сейчас является рекомендуемым стандартом. Он использует эллиптическую криптографию на скрученной кривой Эдвардса (Curve25519), разработанной Дэниелом Дж. Бернштейном.

**Преимущества Ed25519:**
- **Компактность** — ключи занимают всего 68 символов (открытый ключ), файлы крошечного размера
- **Скорость** — генерация и проверка подписи чрезвычайно быстры
- **Безопасная конструкция** — не уязвим к атакам по сторонним каналам, влияющим на некоторые реализации RSA
- **Фиксированный размер ключа** — уровень безопасности 256 бит, не нужно выбирать размер
- **Детерминированность** — подписи детерминированы, что исключает целый класс сбоев генератора случайных чисел

```bash
# Сгенерировать пару ключей Ed25519
ssh-keygen -t ed25519 -C "your@email.com"
```

### RSA — универсальный вариант

RSA — более старый алгоритм (изобретён в 1977 году), и хотя он всё ещё безопасен при 4096 битах, он медленнее и производит более крупные ключи, чем Ed25519. Однако RSA имеет значительно более широкую совместимость — работает со старыми SSH-серверами, устаревшими системами и некоторыми аппаратными ключами безопасности.

**Когда использовать RSA:**
- Нужно подключиться к устаревшим системам со старыми версиями OpenSSH (до 6.5)
- Используется смарт-карта или HSM без поддержки Ed25519
- Требуется совместимость со старыми инструментами

```bash
# Сгенерировать пару ключей RSA 4096 бит
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

**Никогда не используйте RSA менее чем с 4096 битами.** RSA-1024 и RSA-2048 всё больше считаются недостаточными по мере роста вычислительных мощностей. Ed25519 с 256 битами обеспечивает эквивалентную или лучшую безопасность при доле размера ключа.

### Сравнительная таблица

| Характеристика | Ed25519 | RSA 4096 |
|---|---|---|
| Размер ключа (открытый) | ~68 символов | ~725 символов |
| Уровень безопасности | 128-бит (постквантово-устойчивая структура) | ~140-бит классический |
| Скорость | Очень высокая | Медленнее |
| Совместимость | Современный SSH (OpenSSH 6.5+) | Универсальная |
| Рекомендуется для | Всех новых ключей | Только устаревших систем |

**Вывод: генерируйте ключи Ed25519 для любого современного применения.** RSA используйте только при вынужденных требованиях совместимости.

## Как добавить SSH-ключ в GitHub

Добавление вашего открытого SSH-ключа в GitHub позволяет выполнять push и pull по SSH без ввода имени пользователя и пароля при каждой операции. Вот пошаговый процесс.

### Шаг 1: Сгенерируйте пару ключей

Если у вас ещё нет пары SSH-ключей, сгенерируйте её. Вы можете использовать наш <a href="/">бесплатный онлайн-генератор SSH-ключей</a> или выполнить:

```bash
ssh-keygen -t ed25519 -C "your-github-email@example.com"
```

Примите расположение файла по умолчанию (`~/.ssh/id_ed25519`) и при желании задайте парольную фразу.

### Шаг 2: Скопируйте открытый ключ

Отобразите открытый ключ и скопируйте его в буфер обмена:

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows (PowerShell)
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

Или просто отобразите и скопируйте вручную:

```bash
cat ~/.ssh/id_ed25519.pub
```

### Шаг 3: Добавьте ключ в GitHub

1. Войдите в **GitHub.com**
2. Нажмите на аватар → **Settings**
3. В левой панели нажмите **SSH and GPG keys**
4. Нажмите **New SSH key**
5. Дайте ему понятное название (например, «MacBook Pro 2024» или «Рабочий ноутбук»)
6. Вставьте открытый ключ в поле **Key**
7. Нажмите **Add SSH key**

### Шаг 4: Проверьте соединение

```bash
ssh -T git@github.com
```

Вы должны увидеть: `Hi username! You've successfully authenticated, but GitHub does not provide shell access.`

### Переключение существующего репозитория на SSH

Если вы клонировали репозиторий по HTTPS и хотите перейти на SSH:

```bash
# Проверить текущий URL удалённого репозитория
git remote -v

# Обновить на SSH
git remote set-url origin git@github.com:username/repository.git
```

Для GitLab процесс идентичен — перейдите в **Preferences → SSH Keys** и выполните те же шаги.

## Где хранятся SSH-ключи?

SSH-ключи хранятся в директории `.ssh` в домашней папке пользователя. Расположение зависит от операционной системы.

### Linux и macOS

```
~/.ssh/
├── id_ed25519          # Приватный ключ (права: 600)
├── id_ed25519.pub      # Открытый ключ (права: 644)
├── id_rsa              # RSA приватный ключ (если есть)
├── id_rsa.pub          # RSA открытый ключ
├── authorized_keys     # Открытые ключи для входа на эту машину
├── known_hosts         # Отпечатки серверов, к которым вы подключались
└── config              # Файл конфигурации клиента
```

Сама директория `~/.ssh/` должна иметь права **700** (только владелец: чтение/запись/выполнение). Файлы приватных ключей должны иметь права **600** (только владелец: чтение/запись). SSH откажется использовать ключи с чрезмерно открытыми правами.

```bash
# Исправить права при необходимости
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

### Windows

В Windows 10 и 11 с установленным OpenSSH SSH-ключи хранятся в:

```
C:\Users\YourUsername\.ssh\
├── id_ed25519
├── id_ed25519.pub
├── authorized_keys
├── known_hosts
└── config
```

В Windows директория `.ssh` доступна из Проводника (может потребоваться включить «Показывать скрытые файлы») или из PowerShell с помощью `~\.ssh\`.

### Несколько файлов ключей

Если у вас есть ключи для разных целей (личный GitHub, рабочий GitLab, продакшн-серверы), используйте файл `~/.ssh/config` для сопоставления каждого хоста с его ключом:

```
Host github.com
    IdentityFile ~/.ssh/id_ed25519_github

Host gitlab.work.com
    IdentityFile ~/.ssh/id_ed25519_work

Host prod-server
    HostName 203.0.113.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_production
```

## Парольные фразы для SSH-ключей

При генерации пары SSH-ключей `ssh-keygen` спрашивает, хотите ли вы задать **парольную фразу**. Это опциональный пароль, который шифрует файл приватного ключа на диске.

### Зачем использовать парольную фразу?

Если файл приватного ключа защищён надёжной парольной фразой, кражи файла недостаточно для вашей имперсонации — злоумышленнику также потребуется взломать парольную фразу. Это создаёт критически важный второй уровень защиты, особенно для ноутбуков, которые могут быть физически похищены.

Без парольной фразы приватный ключ хранится в открытом виде на диске. Любой, кто может прочитать файл (через вредоносное ПО, физический доступ или ненадлежащим образом защищённую резервную копию), может немедленно использовать его.

### Взаимодействие парольных фраз с ssh-agent

Недостаток парольной фразы — необходимость вводить её каждый раз при использовании ключа. **ssh-agent** решает эту проблему, храня расшифрованные ключи в памяти на протяжении сеанса.

```bash
# Запустить ssh-agent и добавить ключ
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
# Введите парольную фразу один раз — она кэшируется на время сеанса
```

В macOS системная цепочка ключей интегрирована с ssh-agent. Добавьте `--apple-use-keychain` для сохранения между перезагрузками:

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

### Смена парольной фразы

Вы можете сменить парольную фразу существующего ключа без перегенерации пары:

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

Будет запрошена старая парольная фраза, затем предложено задать новую. Открытый ключ при этом не меняется, поэтому обновлять его на серверах не нужно.

## Отпечатки SSH-ключей

Каждая пара SSH-ключей имеет **отпечаток** — короткий хэш открытого ключа, уникально его идентифицирующий. Отпечатки служат нескольким целям:

- **Проверка подлинности хоста** — при первом подключении к серверу SSH показывает отпечаток его хост-ключа. Если вы подтверждаете совпадение с ожидаемым (например, предоставленным вашим хостинг-провайдером), значит, вы подключаетесь к нужному серверу.
- **Идентификация ключей** — при управлении множеством ключей отпечатки помогают определить, какой физический ключ соответствует какой записи в `authorized_keys` или на таком сервисе, как GitHub.
- **Аудит и отзыв** — в журналах аудита часто записываются отпечатки, а не полные ключи. Знание отпечатка ключа помогает найти и удалить его при необходимости.

### Просмотр отпечатка ключа

```bash
# Просмотреть отпечаток открытого ключа
ssh-keygen -lf ~/.ssh/id_ed25519.pub
# Вывод: 256 SHA256:abcdef1234567890... your@email.com (ED25519)

# Просмотреть отпечаток хост-ключа сервера
ssh-keyscan server.example.com | ssh-keygen -lf -
```

GitHub отображает отпечатки всех добавленных SSH-ключей в **Settings → SSH and GPG keys**, позволяя проверить, что видимый там ключ совпадает с ключом на вашей машине.

### Файл `known_hosts`

При первом подключении к новому SSH-серверу вы видите сообщение:

```
The authenticity of host 'example.com (203.0.113.10)' can't be established.
ED25519 key fingerprint is SHA256:abcdef1234567890...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

Введите `yes`, и отпечаток сервера будет добавлен в `~/.ssh/known_hosts`. При последующих подключениях SSH молча проверяет, совпадает ли ключ сервера с сохранённым отпечатком. Если нет — что может указывать на неправильную конфигурацию или атаку «человек посередине» — SSH громко предупредит вас и откажется подключаться.

## Часто задаваемые вопросы

### Для чего используется SSH-ключ?

SSH-ключи аутентифицируют вас на удалённых серверах без пароля. Наиболее распространённые случаи использования: вход на серверы Linux/Unix для администрирования, аутентификация в сервисах хостинга Git (GitHub, GitLab, Bitbucket) для push/pull-операций, скрипты автоматического развёртывания, требующие беспарольного доступа к серверам, и CI/CD-пайплайны, деплоящие код в продакшн. SSH-ключи предпочтительнее паролей, потому что они защищены от брутфорса, не используются повторно при утечках и могут применяться без участия человека.

### Сколько SSH-ключей мне нужно?

Единственного правильного ответа нет, но распространённая лучшая практика — **один ключ на устройство** (не на сервис). На вашем ноутбуке — одна пара ключей, на десктопе — другая, на CI-сервере — ещё одна. Тогда при краже или компрометации устройства вы отзываете только ключ этого устройства. Некоторые дополнительно разделяют по контексту (личный vs рабочий), получая два ключа на устройство. Избегайте использования одного ключа на нескольких машинах — при его компрометации весь ваш доступ окажется под угрозой одновременно.

### Можно ли передавать открытый SSH-ключ?

Да. Открытый ключ предназначен для передачи. Его можно отправить по электронной почте, опубликовать в документации GitHub или включить в заявку на получение доступа к серверу. Один лишь открытый ключ не может быть использован для аутентификации от вашего имени — для этого требуется соответствующий приватный ключ, который есть только у вас. Это фундаментальное свойство асимметричной криптографии. Никогда никому не передавайте **приватный** ключ.

### Что произойдёт, если я потеряю приватный SSH-ключ?

Если вы потеряете приватный ключ и у вас нет резервной копии, вы утратите возможность аутентифицироваться с его помощью. Ваш открытый ключ, по-прежнему установленный на серверах и сервисах, станет «сиротой». Вам потребуется альтернативный способ доступа к серверу (аутентификация по паролю, если она включена, консольный доступ, экстренный доступ хостинг-провайдера) для удаления старого открытого ключа из `authorized_keys` и добавления нового. Именно поэтому важно делать резервную копию приватного ключа (в менеджере паролей или зашифрованной резервной копии). Приватный ключ не может быть восстановлен из открытого.

### Что лучше: Ed25519 или RSA?

**Ed25519 лучше для всех новых ключей.** Он быстрее, производит меньшие ключи, не уязвим к определённым слабостям реализации, влияющим на RSA, и обеспечивает эквивалентную или превосходящую безопасность с 256-битным ключом по сравнению с RSA-4096. Единственная причина использовать RSA сегодня — совместимость с очень старым SSH-серверным ПО (OpenSSH до версии 6.5, выпущенной в 2014 году) или устаревшими аппаратными модулями безопасности. Если вы обращаетесь к любому современному серверу или сервису типа GitHub — используйте Ed25519.
