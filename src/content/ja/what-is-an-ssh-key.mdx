# SSH鍵とは？公開鍵・秘密鍵・鍵ペアを解説

**SSH鍵**は、パスワードなしでSSHサーバーに認証するための暗号資格情報です。SSH鍵はペアで存在します――**公開鍵**と**秘密鍵**――この組み合わせにより、従来のパスワード認証よりも安全で便利なリモートアクセスが実現します。本ガイドでは、SSH鍵とは何か、鍵ペアシステムの仕組み、使用する鍵の種類の選び方、そしてGitHubやGitLabなどのサービスでSSH鍵を使う方法を解説します。

## SSH鍵の仕組み

SSH鍵は**非対称暗号**（公開鍵暗号とも呼ばれる）に基づいています。対称暗号（単一の秘密鍵でデータを暗号化・復号する方式）とは異なり、非対称システムでは数学的に関連付けられた鍵ペアを使用し、一方の鍵で暗号化したデータはもう一方の鍵でのみ復号できます。

SSH鍵を使用してサーバーにログインする際の認証フローは以下の通りです：

1. `ssh user@server.example.com` を実行します。
2. サーバーは `~/.ssh/authorized_keys` であなたの公開鍵を検索します。
3. サーバーはランダムなチャレンジを生成し、あなたの公開鍵で暗号化します。
4. あなたのSSHクライアントは秘密鍵を使ってチャレンジを復号します。
5. クライアントは暗号証明（復号したチャレンジとセッションIDから導出）をサーバーに送り返します。
6. サーバーは証明を検証します。秘密鍵の所持者のみがこれを生成できます。
7. 認証成功。パスワードはマシンの外に出ることはありません。

重要なポイントは：**秘密鍵はネットワーク越しに送信されることは一切ありません**。サーバーが見るのは公開鍵と、あなたが生成した暗号証明だけです。SSHセッションのすべてのパケットを傍受されても、なりすましに使える情報は何も得られません。

SSHプロトコル全体の背景については、<a href="/ja/what-is-ssh">SSHとは何か</a>のガイドをご覧ください。

## 公開鍵と秘密鍵：違いは何か？

SSHを正しく使用するには、2つの鍵の違いを理解することが不可欠です。

### 公開鍵――あなたの錠前

**公開鍵**は自由に共有できるよう設計されています。南京錠のようなものと考えてください：アクセスしたいシステムを持つ全員にコピーを渡します。サーバーの `authorized_keys` ファイルにあなたの公開鍵があれば、秘密鍵を知らなくてもあなたの身元を確認できます。

公開鍵の特徴：
- 安全に共有できる――なりすましに使用できない
- サーバーの `~/.ssh/authorized_keys` に配置する
- GitHub、GitLabなどのサービスにアップロードする
- 通常 `.pub` 拡張子がある（例：`id_ed25519.pub`）
- 形式：`ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... user@hostname`

### 秘密鍵――あなたの鍵

**秘密鍵**はペアの秘密の半分です。マシンから出してはならず、誰とも共有してはいけません――管理者も、クラウドプロバイダも、サポートスタッフも含めて。秘密鍵が漏洩すると、それを入手した人があなたの公開鍵がインストールされているすべてのサーバーであなたになりすますことができます。

秘密鍵の特徴：
- 秘密を保ち安全に保管しなければならない
- ローカルの `~/.ssh/` に制限されたパーミッション（600）で保存する
- 任意でパスフレーズで暗号化できる
- 拡張子なし、または言語固有の拡張子（例：`id_ed25519`）

### `authorized_keys` ファイル

サーバー側では、各ユーザーの公開鍵が `~/.ssh/authorized_keys` に保存されています。これはプレーンテキストファイルで、各行に1つの公開鍵が含まれます。SSHクライアントが認証を試みると、サーバーはクライアントがこのファイルのいずれかのエントリに対応する秘密鍵の所持を証明できるかどうかを確認します。

```
# ~/.ssh/authorized_keys の形式
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGq... alice@laptop
ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIHr... alice@workstation
ssh-rsa AAAAB3NzaC1yc2EAAAA... backup-deploy-key
```

このファイルには必要な数の公開鍵を持つことができ、複数のデバイスや自動化システムが同じサーバーに認証できます。

## SSH鍵のアルゴリズム：Ed25519対RSA

SSH鍵ペアを生成する際、アルゴリズムを選択します。現在最も一般的な2つのオプションは **Ed25519** と **RSA** です。ECDSAとDSAもありますが、DSAは非推奨でECDSAには実装上の落とし穴があるため、推奨される2つに絞って説明します。

### Ed25519――モダンな選択

Ed25519は2014年にOpenSSH 6.5で導入され、現在推奨されるデフォルトです。Daniel J. Bernsteinが設計したTwisted Edwards曲線（Curve25519）上の楕円曲線暗号を使用します。

**Ed25519の利点：**
- **コンパクト** ― 公開鍵はわずか68文字、ファイルは小さい
- **高速** ― 署名の生成と検証が非常に速い
- **設計上安全** ― 一部のRSA実装に影響するサイドチャネル攻撃に脆弱でない
- **固定鍵サイズ** ― 256ビットセキュリティレベルで、鍵サイズを選ぶ必要がない
- **決定論的** ― 署名は決定論的で、乱数生成器障害の一種を排除する

```bash
# Ed25519鍵ペアを生成
ssh-keygen -t ed25519 -C "your@email.com"
```

### RSA――汎用オプション

RSAは古いアルゴリズム（1977年発明）で、4096ビットでは依然として安全ですが、Ed25519より遅く大きな鍵を生成します。ただし、RSAは非常に古いSSHサーバー、レガシーシステム、一部のハードウェアセキュリティキーとの互換性が高いです。

**RSAを使用する場合：**
- 古いバージョンのOpenSSH（6.5以前）を実行するレガシーシステムへの接続が必要な場合
- Ed25519をサポートしないスマートカードやHSMを使用している場合
- 古いツールとの互換性が必要な場合

```bash
# RSA 4096ビット鍵ペアを生成
ssh-keygen -t rsa -b 4096 -C "your@email.com"
```

**4096ビット未満のRSAは絶対に使用しないでください。** コンピュータの処理能力が増すにつれ、RSA-1024とRSA-2048はますます不十分と見なされています。Ed25519の256ビットは、鍵サイズのごく一部で同等またはそれ以上のセキュリティを提供します。

### 比較まとめ

| 機能 | Ed25519 | RSA 4096 |
|---|---|---|
| 鍵サイズ（公開） | 約68文字 | 約725文字 |
| セキュリティレベル | 128ビット（耐量子構造） | 約140ビット（古典的） |
| 速度 | 非常に速い | 遅い |
| 互換性 | モダンSSH（OpenSSH 6.5以降） | 汎用 |
| 推奨用途 | すべての新しい鍵 | レガシーシステムのみ |

**結論：モダンなユースケースにはEd25519鍵を生成してください。** 互換性の要件に迫られた場合のみRSAを使用してください。

## GitHubにSSH鍵を追加する方法

GitHubにSSH公開鍵を追加することで、すべての操作でユーザー名とパスワードを入力することなくSSH経由でプッシュ・プルできます。手順を説明します。

### ステップ1：鍵ペアを生成する

SSH鍵ペアをまだ持っていない場合は生成します。<a href="/">無料のオンラインSSH鍵ジェネレーター</a>を使うか、以下を実行します：

```bash
ssh-keygen -t ed25519 -C "your-github-email@example.com"
```

デフォルトのファイル場所（`~/.ssh/id_ed25519`）を受け入れ、必要に応じてパスフレーズを設定します。

### ステップ2：公開鍵をコピーする

公開鍵を表示してクリップボードにコピーします：

```bash
# Linux
cat ~/.ssh/id_ed25519.pub | xclip -selection clipboard

# macOS
cat ~/.ssh/id_ed25519.pub | pbcopy

# Windows（PowerShell）
Get-Content ~/.ssh/id_ed25519.pub | Set-Clipboard
```

または単に表示して手動でコピーします：

```bash
cat ~/.ssh/id_ed25519.pub
```

### ステップ3：GitHubに鍵を追加する

1. **GitHub.com** にログイン
2. プロフィール画像をクリック → **Settings（設定）**
3. 左サイドバーで **SSH and GPG keys** をクリック
4. **New SSH key** をクリック
5. わかりやすいタイトルを付ける（例：「MacBook Pro 2024」や「Work Laptop」）
6. **Key** フィールドに公開鍵を貼り付ける
7. **Add SSH key** をクリック

### ステップ4：接続をテストする

```bash
ssh -T git@github.com
```

次のメッセージが表示されるはずです：`Hi username! You've successfully authenticated, but GitHub does not provide shell access.`

### 既存リポジトリをSSHに切り替える

HTTPSでクローンしたリポジトリをSSHに切り替えるには：

```bash
# 現在のリモートURLを確認
git remote -v

# SSHに更新
git remote set-url origin git@github.com:username/repository.git
```

GitLabの場合、手順は同じです――**Preferences → SSH Keys** に移動して同じ手順に従います。

## SSH鍵はどこに保存されるか？

SSH鍵はユーザーのホームフォルダ内の `.ssh` ディレクトリに保存されます。場所はOSによって異なります。

### LinuxとmacOS

```
~/.ssh/
├── id_ed25519          # 秘密鍵（パーミッション：600）
├── id_ed25519.pub      # 公開鍵（パーミッション：644）
├── id_rsa              # RSA秘密鍵（ある場合）
├── id_rsa.pub          # RSA公開鍵
├── authorized_keys     # このマシンへのログインを許可する公開鍵
├── known_hosts         # 接続したサーバーのフィンガープリント
└── config              # クライアント設定ファイル
```

`~/.ssh/` ディレクトリ自体は **700** パーミッション（オーナーのみ読み書き実行可能）が必要です。秘密鍵ファイルは **600** パーミッション（オーナーのみ読み書き可能）が必要です。パーミッションが緩すぎる鍵はSSHが使用を拒否します。

```bash
# 必要に応じてパーミッションを修正
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519
chmod 644 ~/.ssh/id_ed25519.pub
```

### Windows

OpenSSHがインストールされたWindows 10とWindows 11では、SSH鍵は以下に保存されます：

```
C:\Users\YourUsername\.ssh\
├── id_ed25519
├── id_ed25519.pub
├── authorized_keys
├── known_hosts
└── config
```

Windowsでは、エクスプローラー（「隠しファイルを表示」を有効にする必要がある場合があります）またはPowerShellで `~\.ssh\` を使用して `.ssh` ディレクトリにアクセスできます。

### 複数の鍵ファイル

異なる目的（個人のGitHub、職場のGitLab、本番サーバー）に鍵がある場合、`~/.ssh/config` ファイルを使用して各ホストをその鍵にマッピングします：

```
Host github.com
    IdentityFile ~/.ssh/id_ed25519_github

Host gitlab.work.com
    IdentityFile ~/.ssh/id_ed25519_work

Host prod-server
    HostName 203.0.113.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_production
```

## SSHパスフレーズ

SSH鍵ペアを生成する際、`ssh-keygen` は**パスフレーズ**を設定するかどうか尋ねます。これはディスク上の秘密鍵ファイルを暗号化するオプションのパスワードです。

### なぜパスフレーズを使うのか？

強いパスフレーズで秘密鍵ファイルが保護されていれば、ファイルを盗まれただけではなりすましはできません――攻撃者はパスフレーズも解読する必要があります。これは特に物理的に盗難される可能性のあるノートPCにとって重要な第二のセキュリティ層となります。

パスフレーズがなければ、秘密鍵はディスク上に平文で保存されます。マルウェア、物理的アクセス、適切に保護されていないバックアップ経由でファイルを読み取れる人は誰でも即座にそれを使用できます。

### パスフレーズとssh-agentの連携

パスフレーズのデメリットは鍵を使用するたびに入力が必要なことです。**ssh-agent** はセッション中復号した鍵をメモリに保持することでこれを解決します。

```bash
# ssh-agentを起動して鍵を追加
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
# 一度パスフレーズを入力――セッション中キャッシュされる
```

macOSでは、システムキーチェーンがssh-agentと統合されています。再起動後も保持するには `-K` を追加します：

```bash
ssh-add --apple-use-keychain ~/.ssh/id_ed25519
```

### パスフレーズの変更

鍵ペアを再生成することなく既存の鍵のパスフレーズを変更できます：

```bash
ssh-keygen -p -f ~/.ssh/id_ed25519
```

古いパスフレーズの入力を求められ、次に新しいものを設定できます。公開鍵は変わらないため、どのサーバーでも更新する必要はありません。

## SSH鍵のフィンガープリント

すべてのSSH鍵ペアには**フィンガープリント**があります――公開鍵の短いハッシュで一意に識別します。フィンガープリントはいくつかの目的で使用されます：

- **ホストのID確認** ― サーバーに初めて接続するとき、SSHはホスト鍵のフィンガープリントを表示します。それが期待するもの（ホスティングプロバイダが提供したものなど）と一致することを確認すれば、正しいサーバーに接続していることがわかります。
- **鍵の識別** ― 多くの鍵を管理するとき、フィンガープリントで `authorized_keys` やGitHubなどのサービス上のエントリに対応する物理的な鍵を特定します。
- **監査と失効** ― 監査ログはしばしば完全な鍵ではなくフィンガープリントを記録します。鍵のフィンガープリントを知ることで、必要に応じてそれを見つけて削除できます。

### 鍵のフィンガープリントを表示する

```bash
# 公開鍵のフィンガープリントを表示
ssh-keygen -lf ~/.ssh/id_ed25519.pub
# 出力：256 SHA256:abcdef1234567890... your@email.com (ED25519)

# サーバーのホスト鍵のフィンガープリントを表示
ssh-keyscan server.example.com | ssh-keygen -lf -
```

GitHubは **Settings → SSH and GPG keys** で追加されたすべてのSSH鍵のフィンガープリントを表示し、そこに表示されている鍵がマシン上のものと一致するか確認できます。

### `known_hosts` ファイル

新しいSSHサーバーに初めて接続するとき、次のようなメッセージが表示されます：

```
The authenticity of host 'example.com (203.0.113.10)' can't be established.
ED25519 key fingerprint is SHA256:abcdef1234567890...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

`yes` と入力すると、サーバーのフィンガープリントが `~/.ssh/known_hosts` に追加されます。以降の接続では、SSHはサーバーの鍵が保存されたフィンガープリントと一致するかどうかを自動的に検証します。一致しない場合――これはサーバーの設定ミスや中間者攻撃を示している可能性があります――SSHは大きな警告を表示して接続を拒否します。

## よくある質問

### SSH鍵は何に使いますか？

SSH鍵はパスワードなしでリモートサーバーに認証します。最も一般的な用途は：管理のためのLinux/Unixサーバーへのログイン、プッシュ/プルアクセスのためのGitホスティングサービス（GitHub、GitLab、Bitbucket）との認証、パスワードなしのサーバーアクセスが必要な自動デプロイスクリプト、本番環境にコードをデプロイするCI/CDパイプライン。SSH鍵はブルートフォースに耐性があり、侵害にわたって再利用不可で、人間の介在なしに使用できるため、パスワードよりも好まれます。

### SSH鍵はいくつ持つべきですか？

唯一の正解はありませんが、一般的なベストプラクティスは**デバイスごとに1つの鍵**（サービスごとではなく）です。ノートPCに1つの鍵ペア、デスクトップにもう1つ、CIサーバーにまた別の1つ。こうすることで、デバイスが盗まれたり侵害されたりしても、そのデバイスの鍵だけを失効させればいいです。コンテキスト（個人と仕事）でさらに分ける人もいて、デバイスあたり2つの鍵になります。複数のマシン間で単一の鍵を共有することは避けてください――その鍵が侵害されると、すべてのアクセスが同時にリスクにさらされます。

### SSH公開鍵は共有できますか？

はい。公開鍵は共有するために設計されています。メールで送ったり、GitHubのドキュメントに掲載したり、サーバーアクセスを求めるチケットに含めたりしても安全です。公開鍵だけであなたとして認証することはできません――それには対応する秘密鍵が必要で、それを持っているのはあなただけです。これが非対称暗号の基本的な性質です。**秘密**鍵は絶対に誰とも共有しないでください。

### SSH秘密鍵を失くしたらどうなりますか？

秘密鍵を失くしてバックアップもない場合、それで認証する能力を失います。サーバーやサービスにまだインストールされている公開鍵は孤立します。古い公開鍵を `authorized_keys` から削除して新しいものを追加するために、別の方法でサーバーにアクセスする必要があります（パスワード認証が有効であればそれを使用、コンソールアクセス、またはホスティングプロバイダの緊急アクセス機能）。これが秘密鍵のバックアップ（パスワードマネージャーや暗号化バックアップに）が重要な理由です。秘密鍵は公開鍵から再構築できません。

### Ed25519とRSA、どちらが良いですか？

**すべての新しい鍵生成にはEd25519が優れています。** 速く、小さな鍵を生成し、RSAに影響する特定の実装上の脆弱性に影響されず、256ビット鍵でRSA-4096と同等またはそれ以上のセキュリティを提供します。今日RSAを使用する唯一の理由は、非常に古いSSHサーバーソフトウェア（2014年リリースのOpenSSH 6.5以前）やレガシーなハードウェアセキュリティモジュールとの互換性です。モダンなサーバーやGitHubなどのサービスにアクセスするなら、Ed25519を使用してください。
