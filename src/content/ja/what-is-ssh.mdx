# SSHとは何か？セキュアシェル完全ガイド

SSHとは **Secure Shell（セキュアシェル）** の略称で、安全でないネットワーク越しにリモートコンピュータへ安全にアクセス・管理するための暗号化ネットワークプロトコルです。Linuxサーバーへのログイン、GitHubへのコードプッシュ、マシン間のファイル転送など、あらゆる場面でSSHが裏側で動いています。本ガイドでは、SSHとは何か・どう動くか・なぜセキュアなリモートアクセスの標準となったのかを徹底的に解説します。

## SSHとはどういう意味か？

**SSHはSecure Shellの略**です。名前そのものがプロトコルの役割を表しています――リモートマシンへの安全な暗号化シェルセッションを提供するのです。SSH登場以前、管理者はTelnetやrsh（リモートシェル）などのツールでリモートサーバーを管理していました。これらのプロトコルはユーザー名・パスワードを含むすべての情報を平文で送信するため、共有ネットワーク上では容易に盗聴されていました。

SSHは1995年にフィンランドのコンピュータ科学者 **Tatu Ylönen** が作成しました。自分が在籍する大学ネットワークで発生したパスワードスニッフィング攻撃を目撃したことが直接のきっかけです。彼はSSH-1をフリーソフトウェアとして公開し、たちまちインターネット全体に普及しました。その後、IETFによってSSH-2（RFC 4251〜4254）として標準化され、より強力な暗号方式が採用されて元の設計にあったいくつかのセキュリティ上の問題が修正されました。

今日「SSH」といえばほぼSSH-2を指します。最も広く使われる実装は **OpenSSH** です。これはOpenBSDチームが保守するオープンソースプロジェクトで、事実上すべてのLinuxディストリビューションとmacOSに同梱されています。

**SSHの実際の意味**はログインツールにとどまりません。以下を支える完全なプロトコルスイートです：
- セキュアなリモート端末アクセス
- 暗号化ファイル転送（SFTPおよびSCP）
- Git認証（GitHub、GitLab、Bitbucket）
- セキュアなポートフォワーディングとトンネリング
- 自動デプロイパイプライン

## SSHの仕組み

**セキュアシェルプロトコル**は、暗号化・認証・チャネル多重化を個別に処理する階層型アーキテクチャで動作します。この階層を理解することで、なぜSSHが機密性の高い操作に信頼されているのかがわかります。

### 3つのプロトコル層

**1. トランスポート層（RFC 4253）**

SSH接続を開始すると、クライアントとサーバーはまずハンドシェイクを行い、暗号化アルゴリズムを交渉して共有秘密を確立します。モダンなSSHはデフォルトでCurve25519ベースの鍵交換を使用します。ハンドシェイク後のすべての通信は暗号化されます。トランスポート層はMAC（メッセージ認証コード）によるデータ完全性も提供し、転送中のデータ改ざんを防ぎます。

**2. ユーザー認証層（RFC 4252）**

暗号化チャネルが確立されると、サーバーはクライアントに認証を求めます。SSHは複数の認証方式をサポートしています：

- **公開鍵認証** ― 秘密鍵をネットワーク越しに送ることなく、その所持を証明します。最もセキュアで最も一般的な方式です。
- **パスワード認証** ― ユーザーのパスワードを暗号化チャネル経由で送信します。利便性が高い一方、ブルートフォース攻撃に弱い面があります。
- **キーボードインタラクティブ** ― チャレンジ/レスポンス方式で、多要素認証によく使われます。
- **GSSAPI/Kerberos** ― 集中型ID管理を持つエンタープライズ環境で使用されます。

**3. コネクション層（RFC 4254）**

認証後、コネクション層は単一の暗号化接続を複数の論理チャネルに多重化します。これにより、単一のTCP接続上でインタラクティブシェル・リモートコマンド実行・ポートフォワーディング・ファイル転送を同時に行えます。

### 鍵交換の詳細

`ssh user@example.com` と入力したとき、最初の数ミリ秒で以下が起きます：

1. クライアントがサーバーのTCPポート22に接続します。
2. 双方がプロトコルバージョン文字列を交換します。
3. 双方がサポートするアルゴリズムリスト（鍵交換・暗号化・MAC・圧縮）を送ります。
4. Diffie-HellmanまたはECDH鍵交換で共有セッション鍵を導出します。これにより、暗号化トラフィックを録画されても後から復号できません。
5. サーバーが**ホスト鍵**（公開鍵）を送り、クライアントは `~/.ssh/known_hosts` と照合して検証します。
6. 上述の手順で認証を行います。
7. セッションが開始されます。

## SSH対Telnet：なぜSSHが勝ったか

SSH以前、**Telnet** はリモート端末アクセスの標準プロトコルでした。Telnetはシンプルで今でも動作しますが、致命的な欠陥があります：すべてが平文で送信されるのです。ユーザー名、パスワード、入力したすべてのコマンド、すべての出力―― ネットワークトラフィックを傍受できる人なら誰でも見ることができます。

1990年代、インターネットの基盤が信頼された学術・研究ネットワークだった頃は、これは深刻な問題とは見なされていませんでした。インターネットが商業化・対立的な環境になるにつれ、DsniffやEttercapのようなツールでLAN上のTelnet認証情報を簡単に取得できるようになりました。

SSHはすべてを暗号化することでこれを解決しました。以下は直接比較です：

| 機能 | Telnet | SSH |
|---|---|---|
| 暗号化 | なし | 強力（AES、ChaCha20） |
| 認証 | 平文パスワード | 公開鍵＋暗号化パスワード |
| データ整合性 | なし | HMAC検証 |
| ポート | 23 | 22 |
| 現在も推奨 | いいえ | はい |

rsh（リモートシェル）とrloginも同様の問題を抱えていました。どちらもIPベースの信頼に依存しており、これは簡単に偽装できます。SSHはこれらすべてのプロトコルに取って代わり、25年以上にわたって揺るぎない標準であり続けています。

## SSH鍵とは何か？

SSHは2つの主要な認証方式をサポートしています：パスワードと **SSH鍵** です。SSH鍵は非対称暗号に基づく暗号鍵ペア――**公開鍵**と**秘密鍵**――です。

**秘密鍵**はローカルマシンに保管し、必要に応じてパスフレーズで保護します。**公開鍵**はアクセスしたいサーバーの `~/.ssh/authorized_keys` というファイルに配置します。接続時、サーバーは対応する秘密鍵の所持者のみが答えられる暗号チャレンジを発行します。ネットワーク越しにパスワードは送信されません。

このアプローチには複数の利点があります：
- **ブルートフォースに耐性** ― 推測できるものが存在しません。256ビットのEd25519鍵は2²⁵⁶通りの値を持ちます。
- **盗まれるパスワードがない** ― ハンドシェイクを傍受されても有用な情報は何も得られません。
- **自動化可能** ― スクリプトやCI/CDパイプラインが人間の介在なしに認証できます。
- **失効可能** ― `authorized_keys` から公開鍵を削除するだけで即座にアクセスを失効できます。

最もモダンで推奨される鍵の種類は **Ed25519** で、Edwards曲線上の楕円曲線暗号を使用します。RSA鍵も広くサポートされていますが、少なくとも4096ビットを使用すべきです。

鍵ペアの詳細については、<a href="/ja/what-is-an-ssh-key">SSH鍵とは何か</a>のガイドをご覧ください。

## SSHの主なユースケース

SSHは非常に幅広いシナリオで使用されています。最も重要なものを紹介します。

### リモートサーバー管理

最も基本的なユースケースです。システム管理者はSSHターミナルセッションを通じてLinux・Unixサーバーにログインし、ソフトウェアのインストール、設定ファイルの編集、サービスの管理、問題の診断を行います。

```bash
ssh user@192.168.1.100
ssh admin@myserver.example.com
ssh -p 2222 deploy@staging.example.com
```

### SFTPとSCPによるファイル転送

SSHは同じ暗号化チャネル上に構築された2つのファイル転送プロトコルを提供します：

**SCP（セキュアコピープロトコル）** はファイルコピーのためのシンプルなコマンドラインツールです：

```bash
# ローカルファイルをリモートサーバーにコピー
scp report.pdf user@server:/home/user/documents/

# リモートファイルをローカルマシンにコピー
scp user@server:/var/log/app.log ./local-copy.log

# ディレクトリ全体を再帰的にコピー
scp -r ./project/ user@server:/var/www/
```

**SFTP（SSHファイル転送プロトコル）** はFTPに似たインタラクティブインターフェースを提供し、SCPより豊富な機能（転送の再開やディレクトリ一覧表示など）を備えています。

### Git認証

GitHub、GitLab、BitbucketにSSH鍵を追加すると、SSH経由のすべての `git clone`、`git push`、`git pull` が秘密鍵で認証されます。これにより、Git操作のたびにユーザー名とパスワードを入力する必要がなくなり、開発者に推奨される認証方式です。

```bash
# SSHを使用してリポジトリをクローン
git clone git@github.com:username/repository.git

# GitHub SSH接続を確認
ssh -T git@github.com
```

### 自動デプロイとCI/CD

GitHub Actions、GitLab CI、JenkinsなどのデプロイスクリプトやCI/CDシステムはSSH鍵を使用して、人間の介在なしに本番サーバーへコードをデプロイします。秘密鍵はCIシステムのシークレットとして保存され、対応する公開鍵はサーバーの `authorized_keys` ファイルに追加されます。

### データベースとアプリケーションのポートフォワーディング

SSHトンネリングを使用すると、リモートサービスをローカルマシンに安全に公開できます。例えば、インターネットに公開されていないリモートMySQLデータベースにアクセスするには：

```bash
ssh -L 3307:localhost:3306 user@dbserver.example.com -N
```

これによりローカルトンネルが作成されます：マシンのポート3307への接続は透過的にリモートサーバーのポート3306に転送され、SSHトンネル内で暗号化されます。

## SSHポートと設定オプション

### デフォルトポート22

SSHはデフォルトで **TCPポート22** をリッスンします。これはプロトコル作成時からSSHのために予約されているIANA指定のウェルノウンポートです。ポートを指定せずに `ssh user@host` を実行すると、SSHクライアントはポート22に接続します。

一部の管理者はSSHポートを変更（例：2222や8022）することで、ログへの自動スキャンノイズを減らす軽微なセキュリティ対策を取ります。これは「セキュリティ・バイ・オブスキュリティ」と呼ばれ、適切な鍵ベース認証の代替にはなりませんが、ログのノイズを減らす効果はあります。

```bash
# 非標準SSHポートに接続
ssh -p 2222 user@example.com
```

### SSHクライアント設定ファイル

長いコマンドラインオプションを毎回入力する代わりに、`~/.ssh/config` でSSH接続を設定できます。このファイルでエイリアスを定義し、ホストごとのオプションを設定して、ワークフローを大幅に簡略化できます。

```
Host myserver
    HostName 203.0.113.10
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_prod

Host bastion
    HostName jump.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host internal
    HostName 10.0.1.50
    User ubuntu
    ProxyJump bastion
```

この設定により、`ssh myserver` は指定された鍵を使用してカスタムポートの完全なホスト名に接続します――フラグ不要です。

SSHコマンドのオプションとフラグの完全なリファレンスは、<a href="/ja/ssh-command">SSHコマンドガイド</a>をご覧ください。

### サーバー側の設定（sshd_config）

SSHサーバーデーモン（sshd）は `/etc/ssh/sshd_config` で設定されます。主要な設定項目：

```
# パスワード認証を無効化――鍵のみ
 PasswordAuthentication no

# rootログインを無効化
 PermitRootLogin no

# リスニングポートを変更
Port 2222

# 特定ユーザーのみ許可
AllowUsers deploy admin

# 公開鍵認証を有効化
PubkeyAuthentication yes
```

編集後、デーモンをリロード：`sudo systemctl reload sshd`

## SSHトンネリング

SSHトンネリング（SSHポートフォワーディングとも呼ばれる）を使用すると、任意のTCPトラフィックを暗号化されたSSH接続経由でルーティングできます。SSHの最も強力でありながら使われていない機能の一つです。

### ローカルポートフォワーディング

ローカルフォワーディングにより、リモートサービスをローカルマシンでアクセス可能にします。構文は `-L local_port:remote_host:remote_port` です。

```bash
# リモートのPostgreSQLデータベースにローカルからアクセス
ssh -L 5433:localhost:5432 user@dbserver -N

# ジャンプサーバー経由で社内Webアプリにアクセス
ssh -L 8080:internal-app.company.com:80 user@jump.company.com -N
```

`-N` フラグはSSHにリモートコマンドを実行しないよう指示します――トンネルだけが必要な場合に便利です。

### リモートポートフォワーディング

リモートフォワーディングはリモートサーバーにローカルサービスを公開します。構文は `-R remote_port:local_host:local_port` です。

```bash
# ローカルWebサーバーをリモートマシンに公開
ssh -R 9090:localhost:3000 user@server -N
```

リモートホストにアクセスできる同僚と一時的にローカル開発サーバーを共有したり、クライアントに進行中の作業への一時的なアクセスを提供したりする際に便利です。

### 動的ポートフォワーディング（SOCKSプロキシ）

`-D` フラグはローカルマシンにSOCKSプロキシを作成し、SOCKS対応のすべてのトラフィックをSSH接続経由でルーティングします。実質的にリモートサーバーをVPN出口ノードとして使用します。

```bash
# ローカルポート1080にSOCKS5プロキシを作成
ssh -D 1080 user@remote-server -N -f
```

ブラウザやシステムを `SOCKS5プロキシ：127.0.0.1:1080` に設定すると、すべてのウェブトラフィックがリモートサーバーを経由します。地域制限のあるコンテンツへのアクセスや、信頼できないネットワークからの安全なブラウジングによく使われます。

## よくある質問

### SSHとはどういう意味ですか？

SSHは **Secure Shell** の略です。安全でないネットワーク上でネットワークサービスを安全に操作するための暗号化ネットワークプロトコルです。名前はその主な用途を表しています：リモートコンピュータ上で安全なインタラクティブシェルセッションを提供することです。

### SSHとSSLは同じですか？

いいえ。SSHとSSL/TLSはどちらも暗号化プロトコルですが、異なる目的で使われます。**SSL/TLS**（Secure Sockets Layer/Transport Layer Security）はウェブトラフィック（HTTPS）、メール、その他のアプリケーションプロトコルのセキュリティに使用されます。**SSH**はセキュアなリモートシェルアクセス、ファイル転送、トンネリングに特化して設計されています。どちらも同様の暗号プリミティブ（公開鍵暗号、対称暗号化）を使用しますが、ハンドシェイクの仕組みとユースケースが異なります。SSHは内部でSSL/TLSを使用しません。

### SSHはどのポートを使用しますか？

SSHはデフォルトで **TCPポート22** を使用します。これはIANAが割り当てたウェルノウンポートです。`/etc/ssh/sshd_config` の `Port` ディレクティブでSSHサーバーが異なるポートをリッスンするよう設定でき、`ssh -p PORT user@host` で非標準ポートに接続できます。

### SSHはハッキングされますか？

鍵ベース認証が適切に設定されたSSHサーバーを侵害することは極めて困難です。暗号化技術――Ed25519鍵、AES-256暗号化、SHA-2 MAC――は鍵認証に対して量子耐性があると考えられており、ブルートフォースは計算上不可能です。実際のSSH侵害は通常、秘密鍵ファイルの盗難、弱いパスフレーズ、弱いパスワードによるパスワード認証を許可するサーバーの設定ミス、未パッチのCVEを持つ脆弱なソフトウェアバージョン、または暗号攻撃ではなくソーシャルエンジニアリングによるものです。サーバーを常に更新し、パスワード認証を無効にし、秘密鍵ファイルを保護してください。

### SSHとHTTPSの違いは何ですか？

SSHとHTTPSはどちらもネットワークトラフィックを暗号化しますが、異なる目的のために設計されています。**HTTPS** はウェブブラウジングのセキュリティを確保します――TLSを使用してブラウザとウェブサーバー間のHTTPリクエストとレスポンスを暗号化します。**SSH** はリモート管理アクセス、ファイル転送、トンネリングのセキュリティを確保します。GitHubにコードをプッシュする際、SSH（ポート22、鍵ベース認証）またはHTTPS（ポート443、トークンベース認証）のどちらかを使用できます。SSHは認証情報を再入力する必要がないため、一般的に開発者に好まれます。HTTPSはポート22をブロックするファイアウォールも通過できるため、あらゆる環境でサポートされています。
