# Cos'è SSH? La Guida Completa al Secure Shell

SSH, acronimo di **Secure Shell**, è un protocollo di rete crittografico che consente di accedere e gestire computer remoti in modo sicuro attraverso reti non protette. Che si tratti di accedere a un server Linux, di inviare codice a GitHub o di trasferire file tra macchine, SSH è quasi certamente il protocollo che opera in background. Questa guida spiega tutto ciò che occorre sapere su SSH: cos'è, come funziona e perché è diventato lo standard universale per l'accesso remoto sicuro.

## Cosa Significa SSH?

**SSH sta per Secure Shell**. Il nome descrive esattamente ciò che fa il protocollo: fornisce una sessione shell sicura e cifrata verso una macchina remota. Prima dell'avvento di SSH, gli amministratori utilizzavano strumenti come Telnet e rsh (remote shell) per gestire i server remoti. Quei protocolli trasmettevano tutto in chiaro, inclusi nome utente e password, rendendoli facilmente intercettabili su qualsiasi rete condivisa.

SSH è stato creato dall'informatico finlandese **Tatu Ylönen** nel 1995, direttamente motivato da un attacco di sniffing delle password che aveva osservato sulla rete universitaria. Rilasciò SSH-1 come software libero, che fu rapidamente adottato su internet. Il protocollo fu in seguito standardizzato dall'IETF come SSH-2 (RFC 4251–4254), che introdusse una crittografia più robusta e corresse diverse debolezze di sicurezza del progetto originale.

Oggi, quando si parla di "SSH", si intende quasi sempre SSH-2. L'implementazione più diffusa è **OpenSSH**, un progetto open source mantenuto dal team di OpenBSD e incluso praticamente in tutte le distribuzioni Linux e nelle installazioni di macOS.

Il **significato pratico di SSH** va ben oltre il semplice strumento di accesso. È una suite di protocolli completa che supporta:
- Accesso remoto sicuro al terminale
- Trasferimenti di file cifrati (SFTP e SCP)
- Autenticazione Git (GitHub, GitLab, Bitbucket)
- Port forwarding e tunneling sicuri
- Pipeline di deployment automatizzate

## Come Funziona SSH

Il **protocollo Secure Shell** opera su un'architettura a livelli che gestisce separatamente cifratura, autenticazione e multiplexing dei canali. Comprendere questi livelli aiuta a chiarire perché SSH è considerato affidabile per le operazioni sensibili.

### I Tre Livelli del Protocollo

**1. Livello di Trasporto (RFC 4253)**

Quando si avvia una connessione SSH, client e server eseguono prima un handshake per negoziare gli algoritmi di cifratura e stabilire un segreto condiviso. Questo utilizza un algoritmo di scambio di chiavi — SSH moderno usa per impostazione predefinita lo scambio di chiavi basato su Curve25519. Dopo l'handshake, tutta la comunicazione successiva è cifrata. Il livello di trasporto fornisce anche l'integrità dei dati tramite codici di autenticazione dei messaggi (MAC), garantendo che i dati non possano essere alterati in transito.

**2. Livello di Autenticazione Utente (RFC 4252)**

Una volta stabilito un canale cifrato, il server chiede al client di autenticarsi. SSH supporta diversi metodi di autenticazione:

- **Autenticazione a chiave pubblica** — Il client dimostra il possesso di una chiave privata senza mai inviarla sulla rete. È il metodo più sicuro e più comune.
- **Autenticazione con password** — La password dell'utente viene inviata sul canale cifrato. Più comoda ma vulnerabile agli attacchi a forza bruta.
- **Keyboard-interactive** — Un meccanismo challenge/response, spesso usato per l'autenticazione a più fattori.
- **GSSAPI/Kerberos** — Utilizzato in ambienti aziendali con gestione centralizzata delle identità.

**3. Livello di Connessione (RFC 4254)**

Dopo l'autenticazione, il livello di connessione effettua il multiplexing della singola connessione cifrata in più canali logici. Questo è ciò che consente di avere una shell interattiva, eseguire comandi remoti, fare port forwarding e trasferire file attraverso un'unica connessione TCP.

### Lo Scambio di Chiavi nel Dettaglio

Quando si digita `ssh user@example.com`, ecco cosa accade nei primi millisecondi:

1. Il client si connette alla porta TCP 22 del server.
2. Entrambe le parti scambiano le stringhe della versione del protocollo.
3. Entrambe le parti inviano elenchi di algoritmi supportati (scambio di chiavi, cifrari, MAC, compressione).
4. Eseguono uno scambio di chiavi Diffie-Hellman o ECDH per derivare una chiave di sessione condivisa — ecco perché, anche se qualcuno registra il traffico cifrato, non potrà decifrarlo in seguito.
5. Il server invia la sua **chiave host** (chiave pubblica), che il client verifica nel file `~/.ssh/known_hosts`.
6. L'autenticazione procede come descritto sopra.
7. La sessione ha inizio.

## SSH vs Telnet: Perché SSH ha Vinto

Prima di SSH, **Telnet** era il protocollo standard per l'accesso remoto al terminale. Telnet è semplice e funziona ancora, ma ha un difetto fatale: tutto viene trasmesso in chiaro. Nome utente, password, ogni comando digitato e ogni output — tutto visibile a chiunque riesca a intercettare il traffico di rete.

Negli anni '90, quando il backbone di internet era composto da reti accademiche e di ricerca affidabili, questo non era considerato un problema serio. Con la commercializzazione di internet e l'aumento delle minacce, strumenti come Dsniff e Ettercap hanno reso banale la cattura delle credenziali Telnet sulle reti locali.

SSH ha risolto questo problema cifrando tutto. Ecco un confronto diretto:

| Funzionalità | Telnet | SSH |
|---|---|---|
| Cifratura | Nessuna | Robusta (AES, ChaCha20) |
| Autenticazione | Password in chiaro | Chiave pubblica + password cifrata |
| Integrità dei dati | Nessuna | Verifica HMAC |
| Porta | 23 | 22 |
| Ancora consigliato | No | Sì |

rsh (remote shell) e rlogin avevano problemi analoghi. Si basavano anche sulla fiducia basata sull'IP, che è banalmente falsificabile. SSH ha sostituito tutti questi protocolli ed è rimasto lo standard incontrastato per oltre 25 anni.

## Cos'è una Chiave SSH?

SSH supporta due metodi di autenticazione principali: password e **chiavi SSH**. Le chiavi SSH sono coppie di chiavi crittografiche — una **chiave pubblica** e una **chiave privata** — basate sulla crittografia asimmetrica.

La **chiave privata** rimane sulla macchina locale, protetta da una passphrase opzionale. La **chiave pubblica** viene inserita su qualsiasi server a cui si vuole accedere, in un file chiamato `~/.ssh/authorized_keys`. Quando ci si connette, il server emette una sfida crittografica a cui solo il possessore della chiave privata corrispondente può rispondere. Nessuna password viene inviata sulla rete.

Questo approccio offre diversi vantaggi:
- **Immune alla forza bruta** — non c'è nulla da indovinare. Una chiave Ed25519 a 256 bit ha 2²⁵⁶ valori possibili.
- **Nessuna password da rubare** — anche se qualcuno intercetta l'handshake, non ottiene nulla di utile.
- **Automatizzabile** — script e pipeline CI/CD possono autenticarsi senza interazione umana.
- **Revocabile** — rimuovere una chiave pubblica da `authorized_keys` revoca immediatamente l'accesso.

Il tipo di chiave più moderno e consigliato è **Ed25519**, che utilizza la crittografia a curva ellittica sulla curva di Edwards. Anche le chiavi RSA sono ampiamente supportate, sebbene si debbano usare almeno 4096 bit.

Per un approfondimento sulle coppie di chiavi, vedere la nostra guida su <a href="/it/what-is-an-ssh-key/">cos'è una chiave SSH</a>.

## Casi d'Uso Comuni di SSH

SSH viene utilizzato in una gamma straordinariamente ampia di scenari. Ecco i più importanti.

### Amministrazione Remota del Server

Il caso d'uso più fondamentale. Gli amministratori di sistema accedono a server Linux e Unix per installare software, modificare file di configurazione, gestire servizi e diagnosticare problemi — tutto attraverso una sessione terminale SSH.

```bash
ssh user@192.168.1.100
ssh admin@myserver.example.com
ssh -p 2222 deploy@staging.example.com
```

### Trasferimento di File con SFTP e SCP

SSH fornisce due protocolli di trasferimento file costruiti sullo stesso canale cifrato:

**SCP (Secure Copy Protocol)** è un semplice strumento da riga di comando per la copia di file:

```bash
# Copia file locale sul server remoto
scp report.pdf user@server:/home/user/documents/

# Copia file remoto sulla macchina locale
scp user@server:/var/log/app.log ./local-copy.log

# Copia intera directory in modo ricorsivo
scp -r ./project/ user@server:/var/www/
```

**SFTP (SSH File Transfer Protocol)** fornisce un'interfaccia interattiva simile a FTP con un set di funzionalità più ricco rispetto a SCP, inclusi trasferimenti ripristinabili e listing delle directory.

### Autenticazione Git

Quando si aggiunge una chiave SSH a GitHub, GitLab o Bitbucket, ogni `git clone`, `git push` e `git pull` via SSH viene autenticato usando la chiave privata. Questo elimina la necessità di inserire nome utente e password ad ogni operazione Git ed è il metodo di autenticazione consigliato per gli sviluppatori.

```bash
# Clona un repository usando SSH
git clone git@github.com:username/repository.git

# Verifica la connessione SSH a GitHub
ssh -T git@github.com
```

### Deployment Automatizzato e CI/CD

Script di deployment e sistemi CI/CD come GitHub Actions, GitLab CI e Jenkins usano chiavi SSH per distribuire codice sui server di produzione senza intervento umano. La chiave privata è memorizzata come segreto nel sistema CI, e la chiave pubblica corrispondente viene aggiunta al file `authorized_keys` del server.

### Port Forwarding per Database e Applicazioni

Il tunneling SSH può esporre in modo sicuro servizi remoti alla macchina locale. Ad esempio, per accedere a un database MySQL remoto non esposto a internet:

```bash
ssh -L 3307:localhost:3306 user@dbserver.example.com -N
```

Questo crea un tunnel locale: la connessione alla porta 3307 sulla propria macchina reindirizza in modo trasparente il traffico alla porta 3306 sul server remoto, cifrato nel tunnel SSH.

## Porta SSH e Opzioni di Configurazione

### Porta Predefinita 22

SSH è in ascolto sulla **porta TCP 22** per impostazione predefinita. Questa è una porta well-known assegnata da IANA, riservata a SSH dalla creazione del protocollo. Quando si esegue `ssh user@host` senza specificare una porta, il client SSH si connette alla porta 22.

Alcuni amministratori cambiano la porta SSH (ad esempio a 2222 o 8022) come misura di sicurezza minore per ridurre il rumore delle scansioni automatizzate nei log. Questa tecnica è chiamata "sicurezza per oscurità" e non sostituisce un'autenticazione corretta basata su chiavi, ma può ridurre il disordine nei log.

```bash
# Connessione a una porta SSH non standard
ssh -p 2222 user@example.com
```

### Il File di Configurazione del Client SSH

Invece di digitare ogni volta lunghe opzioni da riga di comando, è possibile configurare le connessioni SSH in `~/.ssh/config`. Questo file consente di definire alias, impostare opzioni per host specifici e semplificare notevolmente il flusso di lavoro.

```
Host myserver
    HostName 203.0.113.10
    User deploy
    Port 2222
    IdentityFile ~/.ssh/id_ed25519_prod

Host bastion
    HostName jump.example.com
    User ec2-user
    IdentityFile ~/.ssh/id_ed25519_aws

Host internal
    HostName 10.0.1.50
    User ubuntu
    ProxyJump bastion
```

Con questa configurazione, `ssh myserver` si connette al nome host completo sulla porta personalizzata usando la chiave specificata — senza flag aggiuntivi.

Per un riferimento completo sulle opzioni e i flag del comando SSH, vedere la nostra <a href="/it/ssh-command/">guida al comando SSH</a>.

### Configurazione Lato Server (sshd_config)

Il demone del server SSH (sshd) è configurato tramite `/etc/ssh/sshd_config`. Le impostazioni principali includono:

```
# Disabilita l'autenticazione con password — solo chiavi
 PasswordAuthentication no

# Disabilita il login come root
 PermitRootLogin no

# Cambia la porta di ascolto
Port 2222

# Limita a utenti specifici
AllowUsers deploy admin

# Abilita l'autenticazione con chiave pubblica
PubkeyAuthentication yes
```

Dopo la modifica, ricarica il demone: `sudo systemctl reload sshd`

## Tunneling SSH

Il tunneling SSH (chiamato anche SSH port forwarding) consente di instradare traffico TCP arbitrario attraverso una connessione SSH cifrata. È una delle funzionalità più potenti e sottoutilizzate di SSH.

### Port Forwarding Locale

Il forwarding locale rende accessibile un servizio remoto sulla macchina locale. La sintassi è `-L porta_locale:host_remoto:porta_remota`.

```bash
# Accedi localmente a un database PostgreSQL remoto
ssh -L 5433:localhost:5432 user@dbserver -N

# Accedi a un'app web interna attraverso un jump server
ssh -L 8080:internal-app.company.com:80 user@jump.company.com -N
```

Il flag `-N` indica a SSH di non eseguire un comando remoto — utile quando si vuole solo il tunnel.

### Port Forwarding Remoto

Il forwarding remoto espone un servizio locale sul server remoto. La sintassi è `-R porta_remota:host_locale:porta_locale`.

```bash
# Esponi il tuo web server locale alla macchina remota
ssh -R 9090:localhost:3000 user@server -N
```

Questo è utile per condividere temporaneamente un server di sviluppo locale con un collega che ha accesso all'host remoto, o per dare a un cliente accesso temporaneo al lavoro in corso.

### Port Forwarding Dinamico (Proxy SOCKS)

Il flag `-D` crea un proxy SOCKS sulla macchina locale, instradando tutto il traffico compatibile con SOCKS attraverso la connessione SSH. Questo utilizza di fatto il server remoto come nodo di uscita VPN.

```bash
# Crea un proxy SOCKS5 sulla porta locale 1080
ssh -D 1080 user@remote-server -N -f
```

Configura il browser o il sistema per usare `proxy SOCKS5: 127.0.0.1:1080` e tutto il traffico web verrà instradato attraverso il server remoto. Viene spesso usato per accedere a contenuti con restrizioni geografiche o per navigare in modo sicuro da una rete non affidabile.

## Domande Frequenti

### Cosa significa SSH?

SSH sta per **Secure Shell**. È un protocollo di rete crittografico per gestire in modo sicuro i servizi di rete su una rete non protetta. Il nome si riferisce al suo caso d'uso principale: fornire una sessione shell interattiva sicura su un computer remoto.

### SSH e SSL sono la stessa cosa?

No. SSH e SSL/TLS sono entrambi protocolli crittografici, ma servono scopi diversi. **SSL/TLS** (Secure Sockets Layer / Transport Layer Security) viene utilizzato per proteggere il traffico web (HTTPS), la posta elettronica e altri protocolli applicativi. **SSH** è specificamente progettato per l'accesso shell remoto sicuro, il trasferimento di file e il tunneling. Utilizzano primitive crittografiche simili (crittografia a chiave pubblica, cifratura simmetrica) ma hanno meccanismi di handshake e casi d'uso diversi. SSH non utilizza SSL/TLS internamente.

### Quale porta usa SSH?

SSH usa la **porta TCP 22** per impostazione predefinita. È una porta well-known assegnata da IANA. È possibile configurare il server SSH per ascoltare su una porta diversa usando la direttiva `Port` in `/etc/ssh/sshd_config`, e connettersi a porte non standard con `ssh -p PORTA user@host`.

### SSH può essere violato?

Un server SSH correttamente configurato con autenticazione basata su chiavi è estremamente difficile da compromettere. La crittografia — chiavi Ed25519, cifratura AES-256, MAC SHA-2 — è considerata resistente ai computer quantistici per l'autenticazione con chiave e computazionalmente impossibile da violare con la forza bruta. Le compromissioni SSH nel mondo reale tipicamente coinvolgono: file di chiave privata rubati, passphrase deboli, server mal configurati che consentono l'autenticazione con password deboli, versioni di software vulnerabili con CVE non patchati, o ingegneria sociale piuttosto che attacchi crittografici. Tieni il server aggiornato, disabilita l'autenticazione con password e proteggi il file della chiave privata.

### Qual è la differenza tra SSH e HTTPS?

Sia SSH che HTTPS cifrano il traffico di rete, ma sono progettati per scopi diversi. **HTTPS** protegge la navigazione web — usa TLS per cifrare richieste e risposte HTTP tra browser e server web. **SSH** protegge l'accesso amministrativo remoto, i trasferimenti di file e il tunneling. Quando si invia codice a GitHub, è possibile usare SSH (porta 22, autenticazione con chiave) o HTTPS (porta 443, autenticazione con token). SSH è generalmente preferito dagli sviluppatori perché non richiede di reinserire le credenziali. HTTPS è universalmente supportato anche dai firewall che potrebbero bloccare la porta 22.
